# Task 0083.1 — Implementer Report

## Summary

- Wired the per-environment `CONSOLE_CUSTOM_DOMAIN` value already
  declared in `intent.yaml` through to the `cloudflare-domain`
  Terraform apply step as `TF_VAR_CONSOLE_CUSTOM_DOMAIN`, closing
  out the silent-empty failure that broke Task 0083's post-merge
  soak (`0 added, 0 changed, 0 destroyed`,
  `cloudflare_workers_domain.console` count=0).
- Chose **option (c)** from the Architect Brief: promote
  `CONSOLE_CUSTOM_DOMAIN` to a per-environment
  `parameterDefaults.terraform` value so it flows through the
  existing `.parameters` → `TF_VAR_<key>` loop in the
  `cloudflare-domain-validate` job template. **Rationale**: this
  reuses the exact wiring shape already proven by the
  `cloudflare-hyperdrive` and every other terraform component on
  the repo; no composition / job-template edits; no new env→TF_VAR
  bridge to maintain; the rendered TF_VAR is visible to anyone
  reading `intent.yaml`. Options (a) and (b) would have introduced
  a bespoke wiring path local to `cloudflare-domain` that wouldn't
  generalize to the next variable that needs the same treatment.
- One subtlety caught during local validation: the existing
  `component.yaml` previously carried a defaulted
  `CONSOLE_CUSTOM_DOMAIN: ""` declaration under
  `spec.parameters` (uncommitted from a prior local edit). That
  default was **shadowing** the per-env `parameterDefault` and
  causing the rendered `TF_VAR_CONSOLE_CUSTOM_DOMAIN` to resolve
  to `''` for stage and prod too — the same class of silent-empty
  bug we were trying to fix. The committed version of
  `component.yaml` therefore intentionally does **not** declare a
  default for this parameter; the Terraform variable still
  defaults to `""` in `terraform/variables.tf`, so dev (where the
  parameterDefault is `""`) remains safely gated to `count=0`.
- Post-fix render: `cloudflare-domain.stage.terraform` emits
  `TF_VAR_CONSOLE_CUSTOM_DOMAIN=stage.sourceplane.ai`,
  `cloudflare-domain.prod.terraform` emits
  `TF_VAR_CONSOLE_CUSTOM_DOMAIN=prod.sourceplane.ai`. Plan still
  shows the expected 1 component × 2 jobs (stage + prod, dev not
  selected by `--changed` since the `intent.yaml` edit doesn't
  trigger any dev-targeted component).
- Post-merge soak (acceptance of this PR) belongs to the verifier
  per the post-merge-deploy-profile-gap pattern: live `curl` on
  both apex hostnames must return 200.

## Assumptions

- The Cloudflare provider's `cloudflare_workers_domain` resource
  hasn't been deprecated since the Task 0083 v4 pin; if it has,
  this PR will surface the failure on first apply rather than
  later silently. (Same risk surface as Task 0083 — not made
  worse by this hotfix.)
- The zone for `sourceplane.ai` already exists in Cloudflare under
  the configured account and the IAM/API token has zone-edit
  capability. This was the assumption baked into Task 0083 and
  is not re-litigated here.
- Existing pre-merge `plan-only` profile in `cloudflare-domain`
  will surface the create plan diff at PR time even though apply
  only runs on `triggerRef: github-push-main`, so the verifier
  can sanity-check the planned create count before merging.
