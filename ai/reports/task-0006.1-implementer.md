## Summary

The merge pipeline for PR #30 (`supabase-infra`) failed at `supabase.stage.terraform` apply due to two independent issues:

1. Supabase API rejected `instance_size` for a free-plan organization (HTTP 402).
2. The IAM role lacks `secretsmanager:CreateSecret` permission.

This fix removes `instance_size` from the Terraform resource. The IAM fix requires an `aws-admin` policy update documented below.

## Root Cause

### Error 1: Supabase free-plan incompatibility

The `supabase_project` resource specified `instance_size = "micro"`. The Supabase Management API returns HTTP 402 with `"Instance size cannot be specified for free plan organizations."` The provider README and API docs show `instance_size` is optional â€” omitting it defaults to the free-plan instance.

### Error 2: Missing IAM permission

The role `stage-github-sourceplane-multi-tenant-saas-plan` does not have `secretsmanager:CreateSecret` in its identity-based policy. The `access-and-infra.md` spec requires Secrets Manager read/write for `<org>/<repo>/<component>/<env>`, but the `aws-admin` role was not updated to include these actions.

## AWS Secrets Manager Permission Fix

**Blocking dependency on `aws-admin`.**

The IAM role `arn:aws:iam::306024784101:role/stage-github-sourceplane-multi-tenant-saas-plan` (and the equivalent `prod-` role) needs the following policy statement added:

```json
{
  "Sid": "SecretsManagerMultiTenantSaas",
  "Effect": "Allow",
  "Action": [
    "secretsmanager:CreateSecret",
    "secretsmanager:PutSecretValue",
    "secretsmanager:DescribeSecret",
    "secretsmanager:GetSecretValue",
    "secretsmanager:TagResource",
    "secretsmanager:UpdateSecret"
  ],
  "Resource": [
    "arn:aws:secretsmanager:us-east-1:306024784101:secret:sourceplane/multi-tenant-saas/supabase/stage*",
    "arn:aws:secretsmanager:us-east-1:306024784101:secret:sourceplane/multi-tenant-saas/supabase/prod*"
  ]
}
```

The trailing `*` on the resource ARN accounts for the random suffix AWS appends to secret ARNs.

Until this policy is applied in `aws-admin`, the `terraform apply` step will continue to fail on the `aws_secretsmanager_secret` resource.
