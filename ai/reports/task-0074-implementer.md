# Task 0074 — Implementer Report

## Summary

Added the first production-shaped metering rollup materialization seam.
`@saas/db/metering` now exposes `materializeUsageRollups({ bucketType, start,
end })`, which aggregates `metering.usage_records` into `metering.usage_rollups`
for a bounded window via a single `INSERT … SELECT … ON CONFLICT … DO UPDATE`
statement. `apps/metering-worker` got a Cloudflare Worker `scheduled` handler
that drives the materializer over a conservative recent window (prior + current
hour for `hour` rollups, prior + current day for `day` rollups) on an hourly
cron trigger. No billing, Analytics Engine, Queue, KV, Durable Object,
Terraform, or UI surface was introduced; no public route to trigger rollups
was added.

## Rollup Materialization Design

**Aggregation shape.** A single SQL statement does the full pass:

```
WITH agg AS (
  SELECT org_id, project_id, environment_id, metric,
         date_trunc($1, recorded_at) AS bucket_start,
         SUM(quantity)::BIGINT  AS quantity,
         COUNT(*)::BIGINT       AS record_count
  FROM metering.usage_records
  WHERE recorded_at >= $2 AND recorded_at < $3
  GROUP BY org_id, project_id, environment_id, metric, date_trunc($1, recorded_at)
)
INSERT INTO metering.usage_rollups (...)
SELECT md5(org_id||'|'||COALESCE(project_id,'')||'|'||...),
       org_id, project_id, environment_id, metric,
       $1::text, bucket_start, quantity, record_count, now(), now()
FROM agg
ON CONFLICT (
  org_id,
  (COALESCE(project_id, '')),
  (COALESCE(environment_id, '')),
  metric,
  bucket_type,
  bucket_start
) DO UPDATE SET
  quantity     = EXCLUDED.quantity,
  record_count = EXCLUDED.record_count,
  updated_at   = now();
```

**Why this shape:**

- **Idempotent.** The `ON CONFLICT` target mirrors the existing
  `uq_rollup_dimensions` unique index exactly. Re-running over the same window
  upserts the same rows with the same id; row counts and quantities are
  overwritten with the freshly-computed aggregate, not added on top, so
  re-running is a no-op semantically (the same source rows produce the same
  aggregate).
- **Deterministic id.** The id is an `md5` of the full aggregation key (org +
  project + env + metric + bucket_type + bucket_start). Same key → same id
  across all re-runs, which lets downstream consumers reference rollup rows
  stably without depending on autogenerated UUIDs.
- **Org-scope invariant.** `org_id` is in `GROUP BY` and in the conflict key;
  no aggregation row can ever span organizations. This is asserted by a unit
  test.
- **Parameter-bound.** `bucketType`, `start`, and `end` are passed via `$1`,
  `$2`, `$3`; no string interpolation. Asserted by a unit test.
- **Bounded.** The materializer never scans unbounded history — callers pass
  a window and the scheduled invoker only ever passes a 2-hour or 2-day
  recent window.
- **Validation guards.** `bucketType ∉ {hour, day}` and `end ≤ start` are
  rejected before any SQL is issued.
- **Safe error mapping.** Any executor exception is caught and returned as
  `{ kind: "internal", message: "Failed to materialize usage rollups" }` —
  the raw error (which can include host/port info) never leaves the function.

## Invocation Seam

**Scheduled handler.** `apps/metering-worker/src/index.ts` now exports a
`scheduled` handler alongside `fetch`. The handler delegates to
`runScheduledMaterialization(env)` in `rollups.ts`, which:

1. Returns immediately if `SOURCEPLANE_DB` is missing, logging a single
   `[scheduled] SOURCEPLANE_DB binding missing` error line. No throw — the
   scheduled run completes cleanly so Cloudflare does not retry forever.
2. Opens a Hyperdrive-backed SQL executor and creates the metering
   repository.
3. Runs `materializeRecentRollups(repo)` which fires two independent passes:
   - hour: `[startOfPriorHour, startOfNextHour)` — 2 hours
   - day: `[startOfPriorDay, startOfNextDay)` — 2 days
4. Logs one `[scheduled] rollup <bucket> <start>..<end> ok=<bool>
   rows=<n>` line per pass. Counts only — no per-org rows, no metadata, no
   secrets.

**Why a scheduled handler (not an internal HTTP route):** Cloudflare Workers
natively supports `triggers.crons`, and the existing `webhooks-worker`
already uses the same pattern (`apps/webhooks-worker/wrangler.jsonc`,
`scheduled` handler in `src/index.ts`). No Orun composition change was
required — `cloudflare-worker-turbo` deploys the wrangler config as-is, so
adding a cron is a one-line trigger addition. This avoids exposing rollup
execution as a public surface and keeps the seam internal.

**Cron schedule.** `"5 * * * *"` (hourly, 5 minutes past the hour). The
2-hour recent window means each invocation safely re-aggregates the just-
finished hour plus the in-progress hour, absorbing any clock skew or
late-arriving usage events. Daily rollups are re-aggregated 24 times per
day, which is cheap and self-healing.

## Assumptions

- The unique index `uq_rollup_dimensions` on
  `(org_id, COALESCE(project_id,''), COALESCE(environment_id,''), metric,
  bucket_type, bucket_start)` from migration `100_metering_foundation` is
  the authoritative dedup key for rollups. The `ON CONFLICT` clause mirrors
  it exactly, including the `COALESCE` expressions in the same order.
- `cloudflare-worker-turbo` composition deploys the worker's `wrangler.jsonc`
  unmodified, so a top-level `triggers.crons` registers a Cloudflare cron
  on every deploy environment that defines a deploy profile. The existing
  `webhooks-worker` follows the same pattern and works in production today.
- Hourly cron scope is adequate for both hourly and daily rollups because
  the day pass is a cheap re-aggregation that self-heals any drift.
- A 2-hour and 2-day recent window is conservative enough to absorb late-
  arriving usage events without unbounded scans. Backfill of older windows
  is intentionally deferred and out of scope.
- `recorded_at` is the correct time field for bucketing (it reflects when
  the event happened, not when the row was persisted).

## Remaining Gaps

- **Backfill seam.** This task only materializes a recent window. A bounded
  internal maintenance route (or one-off CLI) for backfilling arbitrary
  historical windows is deferred.
- **Rollup-of-rollups.** Daily rollups are computed directly from
  `usage_records`, not by summing hourly rollups. That is fine at current
  data scale but may need revisiting if the raw-records table grows large.
- **Per-org observability.** The scheduled log line reports only aggregate
  counts. Per-org row counts and bucket-coverage metrics will be useful
  once billing comes online but were intentionally omitted to keep secret
  surface minimal.
- **`getUsageSummary` already reads from `usage_rollups`** and will become
  useful as soon as this PR merges and the first cron fires — no API change
  needed.

## Next Task Dependencies

- A future billing task can now consume `metering.usage_rollups` directly
