## Summary

Task 0083 left both apex hostnames `stage.sourceplane.ai` and
`prod.sourceplane.ai` as NXDOMAIN because `CONSOLE_CUSTOM_DOMAIN` from
`intent.yaml` was set as a runtime `env` value but never threaded into
the `cloudflare-domain` Terraform apply step as
`TF_VAR_CONSOLE_CUSTOM_DOMAIN`. The empty TF input gated
`cloudflare_workers_domain.console` to `count = 0`, and apply produced
`0 added, 0 changed, 0 destroyed`.

Task 0083.1 promoted `CONSOLE_CUSTOM_DOMAIN` for each environment from
`environments.{env}.env` into `environments.{env}.parameterDefaults.terraform`
(option (c) — matching the proven `cloudflare-hyperdrive` pattern) and
removed the shadowing component-level default
`CONSOLE_CUSTOM_DOMAIN: ""` from
`infra/terraform/cloudflare-domain/component.yaml`. The job template's
existing `range $key, $value := .parameters` → `TF_VAR_<key>` loop in
`stack-tectonic/compositions/cloudflare-domain/jobs/cloudflare-domain-validate.yaml`
then exports the variable automatically — zero job-template churn.
