# Task 0082.2 Implementer Report — Workers + Static Assets composition

**Branch:** `impl/task-0082.2-workers-assets-composition`
**Approach:** **A** (preferred) — new `cloudflare-workers-assets-turbo` composition; rewire only `apps/web-console-next/component.yaml`.

## Summary

The white-page regression on `sourceplane-web-console-next-stage.pages.dev` is caused by deploying only `.open-next/assets/**` (static chunks + `BUILD_ID`) via `wrangler pages deploy`. The OpenNext Worker entrypoint at `.open-next/worker.js` — which actually serves every SSR route in the Next.js app — was never uploaded. Pages has no concept of `main` here, so the project served HTTP 404 on `/` and `/orgs`.

Approach A introduces a new Orun composition, `cloudflare-workers-assets-turbo`, modelled on `cloudflare-worker-turbo` but tailored for the Workers + Static Assets pattern (single `wrangler deploy` uploads both the worker bundle and the `ASSETS`-bound directory). Only `apps/web-console-next/component.yaml` is repointed at the new composition. `apps/web-console/` and its `cloudflare-pages-turbo` deploy path are untouched.

## Assumptions

1. **Workers.dev subdomain `rahulvarghesepullely`** — derived from the pre-existing smoke command in `apps/web-console-next/component.yaml` (`api-edge-${ORUN_ENVIRONMENT}.rahulvarghesepullely.workers.dev/health`). If this is wrong the smoke step will 404 in CI; the fix is one parameter edit (`workersDevSubdomain`) and no composition change.
2. **Per-env worker naming** keeps the `sourceplane-web-console-next-{dev,stage,prod}` convention from the prior Pages-project layout, so Task 0083's custom-domain repoint is mechanical.
3. **Hostname will switch from `pages.dev` to `workers.dev`** — this is intrinsic to Workers + Static Assets. The pre-existing Pages projects can be deleted as part of Task 0083 cleanup. The verifier should curl `https://sourceplane-web-console-next-stage.rahulvarghesepullely.workers.dev/` (not the old `pages.dev`) for the live HTML check.
4. **No `wrangler env` flag** — `wrangler.jsonc` declares a single env; per-env isolation is via `--name` override at deploy time. This mirrors how `cloudflare-pages-turbo` parameterises `projectName`.
5. **`installCommand` runs at repo root via `workspaceDir: ../..` orun convention** — same as `cloudflare-pages-turbo` / `cloudflare-worker-turbo`.

## Rollback Plan

If the new deploy regresses after merge:

```bash
# Revert the merge commit (squash) — restores both the composition and
# component.yaml to the pre-0082.2 state.
git revert -m 1 <merge-commit-sha>
git push origin main
```

Or, surgical revert of just the component rewire (keeps the new composition):

```bash
git checkout <pre-0082.2-sha> -- apps/web-console-next/component.yaml
git commit -m "revert(web-console-next): repoint to cloudflare-pages-turbo"
git push origin main
```

Either path restores the prior (white-page) Pages deploy. The vanilla `apps/web-console` `cloudflare-pages-turbo` deploy is unaffected.
