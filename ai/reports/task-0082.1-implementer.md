## Assumptions

- **`@opennextjs/cloudflare@1.0.4`** is the highest 1.x release whose
  bundled `@opennextjs/aws@^3.6.2` peer range still accepts Next 15.0.3.
  All `@opennextjs/cloudflare` versions ≥ 1.5 transitively require
  `@opennextjs/aws` ≥ 3.7, which dropped the older API surface this 1.x
  adapter expects (`patchEnvVars` is missing in 3.7+; build fails with
  `SyntaxError: ... does not provide an export named 'patchEnvVars'`). The
  newer 1.15+ line additionally tightens `next` peerDeps to
  `^14.2.35 || ~15.0.7 || …` which would force a Next minor upgrade —
  explicitly out of scope for this task. Pinning to `1.0.4` keeps Next at
  the existing `15.0.3` and React at `19.0.0-rc-66855b96-20241106`.
- **`@opennextjs/aws@3.6.2` pinned as a direct dependency** of the app
  (not via `pnpm.overrides` in the root `package.json`, which would have
  pushed the diff outside the allowed file set). pnpm hoists the older
  3.6.2 from the app's own `dependencies` entry, so the adapter resolves
  the matching API surface; without this pin pnpm picks the latest
  satisfying `^3.6.2` (currently `3.10.4`) and the build breaks on the
  missing-export error above.
- **Per-package `turbo.json` (option (i))** chosen over widening the root
  `turbo.json`'s `tasks.build.outputs`. The root key currently advertises
  `["dist/**", ".wrangler/**"]` — useful for every other Worker/Vite
  component in the repo. Pulling `.open-next/**` and `.next/**` into the
  root would cause every package's build cache to track those globs even
  though only this app emits them. The per-package override uses
  `extends ["//"]` so the `dependsOn` and other root keys still apply.
- **`output: "standalone"` configured in `next.config.mjs`** rather than
  via the `NEXT_PRIVATE_STANDALONE` env var. The adapter normally sets
  the env var itself, but that path runs Next *inside* the adapter's
  spawned `pnpm build`, which recurses back into this same script. Using
  the public `output` config makes the standalone tree available on the
  outer `next build` invocation, so the adapter can run with `--skipBuild`
  and read `.next/standalone/**` directly.
- **`--skipWranglerConfigCheck`** is passed so the adapter doesn't try to
  rewrite `wrangler.jsonc` on every build; the committed file already
  declares the contract the adapter would scaffold (`main` →
  `.open-next/worker.js`, `assets.directory` → `.open-next/assets`).

## Remaining Gaps

- pnpm install emits a single peer-dependency warning:
  `@opennextjs/cloudflare 1.0.4 → @opennextjs/aws (unmet peer) found 3.6.2`.
  This is the inverse of the issue above — `cloudflare@1.0.4`'s
  *declared* peer range (`@opennextjs/aws` >= 3.7) is forward-looking,
  but the runtime API it actually imports (`patchEnvVars`) only exists in
  3.6.x. The mismatch is a known upstream packaging glitch; pinning
  `@opennextjs/aws@3.6.2` here is the documented workaround. The warning
  is non-fatal and `pnpm install --frozen-lockfile` still exits 0.
- Pages projects `sourceplane-web-console-next-{dev,stage,prod}` are
  auto-created on first push by the `ensure-pages-project` step in the
  `cloudflare-pages-turbo-deploy` profile (PATCH-or-POST against the
  Cloudflare API). The Verify profile this PR exercises stops at
  `verify-build-output`, so the projects don't have to exist yet. They
  will be created on the first push to `main` after merge.
