## Required outcomes

### 1. Nine new resource client files under `packages/sdk/src/`

Use one file per resource, mirroring the `projects.ts` shape (class with `constructor(private readonly transport: Transport) {}`, one method per public route, JSDoc per method, encodeURIComponent on every path segment, idempotency-key-ready POSTs):

| File | Class | Contract source | Routes (consult contracts for exact request/response types) |
|---|---|---|---|
| `memberships.ts` | `MembershipsClient` | `@saas/contracts/membership` | `members.{list, updateRole, remove}` + `invitations.{list, create, revoke, accept}` (group under one client with nested sub-namespaces if cleaner, otherwise flat methods like `listMembers`, `updateMemberRole`, `removeMember`, `listInvitations`, `createInvitation`, `revokeInvitation`, `acceptInvitation`) |
| `apiKeys.ts` | `ApiKeysClient` | `@saas/contracts/api-keys` | `list`, `get`, `create` (returns one-time-reveal secret), `revoke` |
| `webhooks.ts` | `WebhooksClient` | `@saas/contracts/webhooks` | endpoints: `listEndpoints`, `getEndpoint`, `createEndpoint`, `updateEndpoint`, `disableEndpoint`, `deleteEndpoint`, `rotateSecret`; subscriptions: `listSubscriptions`, `getSubscription`, `createSubscription`, `updateSubscription`, `deleteSubscription`; deliveries: `listDeliveryAttempts`, `getDeliveryAttempt` |
| `metering.ts` | `MeteringClient` | `@saas/contracts/metering` | `recordUsage`, `ingestUsageBatch`, `getUsageSummary`, `checkQuota`, `listQuotaViolations` |
| `billing.ts` | `BillingClient` | `@saas/contracts/billing` | `listPlans`, `getCustomer`, `listInvoices`, `getEntitlements`, `checkEntitlement`, `getSummary` |
| `events.ts` | `EventsClient` | `@saas/contracts/events` | `listAuditEntries` (supports both `byOrg` and `byTarget` query shapes — accept a discriminated query object) |
| `securityEvents.ts` | `SecurityEventsClient` | `@saas/contracts/security-events` | `list` |
| `config.ts` | `ConfigClient` | `@saas/contracts/config` | settings: `listSettings`, `createSetting`, `updateSetting`; feature flags: `listFeatureFlags`, `createFeatureFlag`, `updateFeatureFlag`; secrets: `listSecretMetadata`, `createSecretMetadata`, `rotateSecret`, `revokeSecret` |
| `notifications.ts` | `NotificationsClient` | `@saas/contracts/notifications` | `enqueue`, `get`, `getPreferences`, `updatePreferences`, `suppressRecipient` |

**Route URL paths** — derive from existing patterns:
- All resources are org-scoped: `/v1/organizations/${encodeURIComponent(orgId)}/<resource>...`
- Sub-resources follow: `/v1/organizations/${orgId}/projects/${projectId}/<resource>...` where applicable (webhooks have both org-scoped and project-scoped surfaces per `apps/api-edge/src/webhooks-facade.ts`)
- Confirm path shapes against the api-edge facade files (`apps/api-edge/src/{org,project,webhooks,billing,config,metering,audit}-facade.ts`) — **read them, do not assume**
- If a facade file does not expose a clear regex for a route you need, log a clarifying note in your report and pick the shape most consistent with neighbours. Do **not** invent new URL conventions.

**Idempotency-Key**: POST methods that create persistent resources accept `RequestOptions` so callers can pass `idempotencyKey`. Same Stripe-parity rule as projects.ts — SDK does NOT auto-generate.
