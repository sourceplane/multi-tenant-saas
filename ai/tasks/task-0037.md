# Objective

Fix the cross-component audit ID mismatch so organization audit reads include
membership-owned audit events consistently.

Normalize membership-worker event/audit persistence to store raw internal UUIDs
in canonical event fields, keep public IDs at the public API/payload boundary,
and make org audit queries backward-compatible with previously written
membership audit rows that used public `org_` IDs.

# Required Outcomes

1. Canonicalize membership event/audit IDs.
   - Membership-worker must store raw UUIDs in canonical event fields:
     - `event.orgId` / resulting `events.*.org_id`;
     - `event.subjectId` for known invitation/member subject UUIDs;
     - any audit scope IDs derived from those event fields.
   - Do not store `org_`, `inv_`, or `mem_` public IDs in those canonical
     persistence fields for newly written membership events.
   - Public IDs may still appear in safe payload fields, descriptions, and
     public API responses when useful.
   - Keep token hashes, raw invitation tokens, bearer tokens, API keys, SQL,
     connection details, and stack traces out of events, audits, responses, and
     logs.

2. Preserve public audit response behavior.
   - `GET /v1/organizations/{orgId}/audit` must continue returning public IDs
     for known scope and subject kinds:
     - `org_` for organizations;
     - `prj_` for projects;
     - `env_` for environments;
     - `inv_` for invitations;
     - `mem_` for members.
   - No raw known internal UUIDs should leak through the public audit response
     for known current subject kinds or scope columns.
   - Payload redaction from Task 0036 must keep working.

3. Add backward compatibility for legacy membership audit rows.
   - An org audit query for a raw org UUID must include audit rows whose
     `org_id` equals either:
     - the raw org UUID, or
     - the legacy public ID form `org_<uuid-hex>`.
   - Preserve cursor pagination order by `(occurred_at DESC, id DESC)`.
   - Preserve category filtering.
   - Keep SQL parameterized.
   - If the implementation chooses to put this compatibility in
     `@saas/db/events`, keep the behavior narrow and documented by tests. If it
     chooses another seam, keep events-worker and repository boundaries clean.
   - Do not add a live backfill or migration in this task.

4. Update and add tests.
   - Update membership-worker tests that currently expect public IDs in
     canonical event fields. They should instead assert raw canonical IDs plus
     safe public IDs in payload/description where appropriate.
   - Add db tests proving org audit queries include both raw UUID and legacy
     public `org_` rows, including category and cursor combinations.
   - Add or update events-worker tests proving public audit responses map raw
     membership subject/scope UUIDs to public IDs and still redact payloads.
   - Keep existing invitation/member mutation behavior and authorization tests
     intact.
