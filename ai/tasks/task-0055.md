## Objective

Add the first database and repository foundation for organization/project/environment-scoped non-secret settings and feature flags, plus secret metadata placeholders, without exposing runtime routes yet.

This satisfies the Product Overview requirement for config, settings, feature flags, and security-safe secret metadata while preserving bounded-context ownership and keeping public API work for a later PR.

## PR Boundary

1. Add a new migration under `packages/db/src/migrations/070_config_settings_flags/up.sql` that creates a config-owned schema for:
   - non-secret settings at organization, project, and environment scope;
   - feature flag definitions/evaluation metadata at organization/project/environment scope;
   - secret metadata records that never store plaintext secret values.
2. Update `packages/db/src/manifest.ts` with the new migration entry and checksum.
3. Add a new `@saas/db/config` repository surface under `packages/db/src/config/` with typed methods for scoped settings, feature flags, and secret metadata primitives.
4. Add focused `tests/db` coverage for migration manifest integrity, tenant/scope invariants, repository SQL behavior, secret-safety, and cursor/list behavior where applicable.
5. Update package exports and test module mapping for `@saas/db/config`.

## Required Outcomes

- [ ] Migration `070_config_settings_flags` creates a config-owned schema and tables for settings, feature flags, and secret metadata.
- [ ] Scope columns are explicit and tenant-safe:
  - organization-scoped rows carry `org_id`;
  - project-scoped rows carry `org_id + project_id`;
  - environment-scoped rows carry `org_id + project_id + environment_id`.
- [ ] Database constraints prevent ambiguous scope rows and prevent project/environment rows without their parent scope IDs.
- [ ] The schema does not add cross-context foreign keys to `membership.*` or `projects.*`; IDs are opaque bounded-context references.
- [ ] Secret metadata tables never contain plaintext secret values. If an encrypted payload placeholder is introduced, it must be named and documented as ciphertext/envelope data, not plaintext, and repository/API types must not expose reveal behavior.
- [ ] `packages/db/src/manifest.ts` includes the new migration with a correct checksum and description.
- [ ] `@saas/db/config` exports typed repository primitives for create/update/read/list settings, create/update/list feature flags, and create/list/rotate/revoke secret metadata as foundation methods.
- [ ] Repository methods require the full enclosing scope for project/environment operations and cannot query by `projectId` or `environmentId` alone.
- [ ] Focused tests cover migration manifest integrity, repository SQL, scope validation, and secret-safety invariants.
- [ ] Orun validation/changed-plan/dry-run evidence is recorded in the implementer report.

## Non-Goals

- No config-worker application, `apps/config-worker`, Wrangler config, service bindings, or deployment component.
- No policy-engine actions or public authorization flows for settings/flags/secrets.
- No organization/project/environment settings UI.
- No encrypted secret provider integration, KMS/Secrets Store integration, plaintext reveal flow, or secret value mutation API.

- No live provider/resource changes outside normal db migration apply on merge.

## Constraints

1. Preserve bounded-context isolation: config-owned tables may store opaque `org_id`, `project_id`, and `environment_id`, but must not FK-reference membership or projects schemas.
2. Preserve tenant isolation: any project/environment-scoped repository method must require `orgId` and enclosing parent IDs. Do not allow lookup by `projectId`, `environmentId`, `flagKey`, or `secretKey` alone.
3. Secrets must be safe by design: no plaintext secret values in migrations, tests, repository types, logs, reports, audit payload examples, or returned list metadata.
4. Migration SQL must be idempotent and safe for the current Supabase API autocommit runner. Avoid migration patterns that require transaction rollback semantics.
5. Use schema/table names and repository method names that can support later public APIs without changing persisted contracts.
6. Keep the PR DB/repository-focused. If public API or UI needs arise, record them as Next Task Dependencies instead of implementing them.
7. Existing generated/ignored files (`dist`, `node_modules`, `.orun`, `plan.json`, TypeScript build info) must not be staged.

## Integration Notes

- Prefer a new schema named `config` unless code inspection reveals an existing in-repo convention that requires a different name.
- A practical first table set is acceptable if it is explicit and testable, for example:
  - `config.settings` for scoped non-secret JSON settings values;
  - `config.feature_flags` for scoped flag definitions and default enabled/value state;
  - `config.secret_metadata` for key/name, scope, status, version/rotation metadata, and safe timestamps.
- Use check constraints or narrow enums for scope kind/status where useful, but avoid over-designing a full targeting DSL.
- JSONB values are acceptable for flexible non-secret settings/flag payloads if tests prove deterministic behavior and no plaintext secret examples are introduced.
- Follow existing repository result patterns from `@saas/db/projects` and `@saas/db/identity`.
