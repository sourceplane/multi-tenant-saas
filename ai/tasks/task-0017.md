# Objective

Add the first V1 policy authorization seam: a deterministic policy-as-code
engine plus a deployable internal `policy-worker` that exposes the shared
authorization contract.

After this PR, future domain Workers should have a stable contract they can
call with membership role facts to answer:

- is this subject allowed to perform this action on this organization/project
  resource?
- which permissions are effective for this subject at a scope?
- is a proposed role assignment compatible with the tenancy/RBAC contract?

Do not wire `api-edge`, `membership-worker`, or any other caller to the policy
Worker in this PR.

# Required Outcomes

## Policy Contracts

- Add a narrow policy contract module, for example
  `packages/contracts/src/policy.ts`, and export it from `@saas/contracts`.
- The contract must model the request and response shape from
  `specs/contracts/tenancy-and-rbac.md`:
  - subject type and ID;
  - action string;
  - resource kind, ID, `orgId`, optional `projectId`, and optional
    `environmentId`;
  - context membership facts;
  - allow/deny result, stable reason, `policyVersion`, and derived scope.
- Reuse existing tenancy role/fact types where possible instead of inventing a
  parallel role model.
- Unknown future fact objects may be present in input contexts; the contract
  should allow them without widening access.
- Keep public IDs opaque. The policy contract should not parse `org_`, `usr_`,
  `prj_`, or database UUID formats.

## Policy Engine

- Add a pure, deterministic policy engine behind a package export such as
  `@saas/policy-engine`.
- Implement at least:
  - `authorize(input)`;
  - `listEffectivePermissions(input)`;
  - `validateRoleAssignment(input)`.
- The engine must be deny-by-default.
- The engine must ignore unknown future fact shapes.
- The engine must reject cross-organization access. A role fact for one
  organization must never authorize a resource in another organization.
- The engine must treat project-scoped authorization as nested under an
  explicit organization. A project role must not authorize a project with a
  different `orgId` or a missing/mismatched `projectId`.
- Implement a small V1 role/action matrix sufficient for near-term starter
  runtime work:
  - organization read;
  - organization settings update;
  - organization invitation create/list/revoke;
  - organization member list/remove/update-role;
  - project create/read/update/delete;
  - environment read/update;
  - billing read/manage.
- Suggested V1 semantics:
  - `owner` can perform all organization, member, invitation, project,
    environment, and billing actions in its organization.
  - `admin` can manage organization settings, members, invitations, projects,
    environments, and operational resources, but does not automatically get
    billing manage permission unless you intentionally decide otherwise and
    test it.
  - `builder` can create/update operational project and environment resources
    but cannot manage members, invitations, organization settings, or billing.
  - `viewer` can perform read-only organization/project/environment actions.
  - `billing_admin` can read/manage billing and read the organization, but
    cannot manage operational resources or members by default.
  - project roles apply only to the matching project scope:
    `project_admin`, `project_builder`, `project_viewer`.
- Return stable reason codes such as `org_owner`, `org_admin`,
  `org_builder`, `org_viewer`, `billing_admin`, `project_admin`,
  `project_builder`, `project_viewer`, `no_matching_role`, `unknown_action`,
  or `invalid_scope`. Exact names may differ, but they must be stable and
  covered by tests.
- Return `policyVersion: 1`.
- `validateRoleAssignment` must enforce the tenancy/RBAC shape:
  - organization roles are valid at organization scope;
  - project roles require project scope and a project ID;
  - project roles cannot be assigned without an organization ID;
  - unknown roles or scope kinds fail safely.
- The engine must not import Worker handlers, `api-edge`, membership Worker
  code, database repositories, Hyperdrive, Terraform, Wrangler, or Node-only
  migration runner code.

## Policy Worker

- Add `@saas/policy-worker` as a Cloudflare Worker app following the existing
  Worker app patterns.
- Add `apps/policy-worker/component.yaml` as a
  `cloudflare-worker-turbo` component with Node 22 and pnpm 10.12.1. It should
  verify in `dev`, `stage`, and `prod`, and deploy `stage`/`prod` on
  `github-push-main`.
- Add a dedicated `wrangler.jsonc` with:
  - base/local `ENVIRONMENT=local`;
  - `dev` with `ENVIRONMENT=dev`;
  - `stage` with `ENVIRONMENT=stage` and `workers_dev: false`;
  - `prod` with `ENVIRONMENT=prod` and `workers_dev: false`.
- Do not bind Hyperdrive or add provider credentials. V1 policy is
  policy-as-code with facts supplied by callers.
- Add `GET /health` returning service name, environment, timestamp, and policy
  version. Do not include secrets, hostnames, raw errors, tokens, or provider
  internals.
- Expose internal JSON routes for:
  - `POST /v1/internal/policy/authorize`;
  - `POST /v1/internal/policy/effective-permissions`;
  - `POST /v1/internal/policy/role-assignments/validate`.
- Use standard success/error envelopes compatible with
  `specs/contracts/api-guidelines.md`.
- Generate or preserve `x-request-id` in responses.
- Validate JSON bodies conservatively and return `validation_failed` for shape
  errors.
- Map engine denials to successful authorization responses with
  `allow: false`; reserve HTTP errors for malformed requests or Worker/runtime
  failures.
- Keep responses safe: no stack traces, raw exception messages, connection
  strings, bearer tokens, role-fact internals beyond the response contract, or
  provider details.
- Because stage/prod `workers_dev` is disabled and no public route is being
  added, no external auth mechanism is required in this PR. Future callers
  should reach the Worker through Cloudflare service bindings.

## Tests

- Add focused tests that do not require live Cloudflare, Supabase, Hyperdrive,
  AWS, or GitHub access.
- Policy engine tests must cover at minimum:
  - deny-by-default for unknown actions and missing facts;
  - owner/admin/builder/viewer/billing-admin organization semantics;
  - project role semantics with matching and mismatched `orgId`/`projectId`;
  - cross-organization denial;
  - unknown future facts being ignored;
  - stable reason and `policyVersion` values;
  - `listEffectivePermissions` output for common role combinations;
  - `validateRoleAssignment` success and failure cases.
- Policy Worker tests must cover at minimum:
  - health response;
  - valid authorize request returns a success envelope;
  - denied authorization returns HTTP 200 with `allow: false`;
  - malformed JSON and invalid bodies return safe error envelopes;
  - request ID generation/preservation;
  - no raw stack traces or unsafe internals in error responses;
  - `wrangler.jsonc` has `workers_dev: false` for stage/prod.
- Contract tests should cover any newly exported policy contract helpers or
  role/action constants if added.
