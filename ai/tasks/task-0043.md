Identity currently persists users, auth identities, login challenges, and
sessions in `identity.*` tables. It does not yet have identity-owned security
event source records, even though the specs require pre-organization identity
security history to live in the identity bounded context rather than in an
org-less shared audit/event envelope.

# Objective

Add the first identity-owned security-event persistence foundation.

This task creates the database migration, repository types, repository methods,
and focused tests needed to record and query identity security-event source
facts by user. It should not expose a public API route or wire login/logout
runtime recording yet; this PR is the durable persistence seam that unblocks the
next identity-worker task.

# PR Boundary

One PR adds identity security-event source persistence only:

- new identity migration and manifest entry;
- `@saas/db/identity` types and repository methods;
- db tests for migration, SQL shape, mapping, pagination, and secret-safety.

Do not combine this with identity-worker runtime event emission, api-edge route
forwarding, web-console UI, notifications, API keys, or shared events-worker
fanout.

# Required Outcomes

1. Add a new forward-only migration after `040_projects_core`, likely
   `050_identity_security_events`.
   - Create an `identity.security_events` table.
   - Keep the migration idempotent with `IF NOT EXISTS`.
   - Keep ownership inside the identity schema.
   - Use only same-context foreign keys, if foreign keys are used at all.
   - Do not alter the shared `events.event_log` or `events.audit_entries`
     org-scoped contract.

2. Model identity security-event source facts without storing secrets.
   - Store raw UUIDs internally, not public IDs.
   - Include enough fields for future runtime recording and query routes:
     event type/name, outcome, user id when known, optional session id,
     optional challenge id, request id, optional correlation id, optional IP,
     optional user agent, occurred time, created time, JSON metadata/payload,
     and redaction paths or equivalent safe metadata.
   - Support pre-organization identity activity without requiring `org_id`.
   - Do not store raw login codes, bearer tokens, token hashes, full secrets, or
     plaintext credential material in the security-event payload.

3. Extend `@saas/db/identity`.
   - Add exported types for stored security events, create input, query params,
     cursor position, and paged result.
   - Add repository methods such as `recordSecurityEvent` and
     `querySecurityEventsByUser`.
   - Use cursor pagination with `(occurred_at DESC, id DESC)`, matching the
     established list pattern: `limit + 1` fetch, opaque-ready cursor shape, and
     deterministic tie-break by id.
   - Keep SQL parameterized and return safe repository errors.

4. Add focused tests.
   - Migration tests must prove the table, indexes, bounded-context ownership,
     idempotency, no cross-context FKs, JSON metadata storage, and no raw secret
     columns.
   - Repository tests must cover insert mapping, JSON serialization/parsing,
     query by user with pagination, next cursor behavior, cursor filtering,
     parameterized SQL, conflict/internal errors, and no secret leakage in
     stored fields or error messages.
   - Update manifest/checksum tests by adding the new migration to
     `packages/db/src/manifest.ts`.

5. Update compact context only if needed to record durable assumptions found
   while implementing. If a behavior change to the identity or event specs is
   required, write a proposal under `ai/proposals/` instead of silently changing
   scope.
