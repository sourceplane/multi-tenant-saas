# Objective

Add the first policy-gated organization member administration mutation surface:

- `PATCH /v1/organizations/{orgId}/members/{memberId}` to update an active
  member's organization role.
- `DELETE /v1/organizations/{orgId}/members/{memberId}` to remove an active
  member from an organization.

Both routes must be accessible through `api-edge`, authorized through the
private `policy-worker` binding, and audited atomically with the membership
mutation using the transaction pattern established in Tasks 0024 and 0026.

# Required Outcomes

1. Add public member mutation contracts and routing.
   - Add shared contract types for role update and member removal responses in
     `@saas/contracts/membership`.
   - Add `parseMemberPublicId(...)` in membership-worker ID helpers if no
     equivalent already exists.
   - Add membership-worker routing for:
     - `PATCH /v1/organizations/{orgId}/members/{memberId}`
     - `DELETE /v1/organizations/{orgId}/members/{memberId}`
   - Add api-edge forwarding for the same routes after identity-backed bearer
     token resolution.
   - Ensure api-edge forwards `PATCH` request bodies, preserves current
     `x-request-id`, `traceparent`, and `idempotency-key` handling, forwards
     actor headers, and still never forwards the original bearer token to
     membership-worker.
   - Preserve existing organization, member list, invitation, and accept route
     behavior.

2. Implement role update semantics.
   - Request body: `{ "role": "<organization role>" }`.
   - Accepted roles are the V1 organization roles only:
     `owner`, `admin`, `builder`, `viewer`, `billing_admin`.
   - Reject project roles and unknown roles with `validation_failed`.
   - The target must be an active member of the route organization.
   - The mutation should replace the target subject's active organization-scoped
     role assignment set with exactly the requested organization role.
   - Leave project-scoped role assignments alone; project role administration is
     out of scope.
   - If the target already has exactly the requested active organization role
     and no other active organization-scoped roles, return a compatible success
     response without writing an event/audit entry.
   - If the target currently has multiple active organization-scoped roles,
     normalize to the requested single role and emit `membership.updated`.
   - Return target member data using public member IDs and safe role summaries.

3. Implement member removal semantics.
   - The target must be an active member of the route organization.
   - Mark the member `removed` instead of deleting the row.
   - Revoke all active role assignments for the target subject in that
     organization, including organization-scoped and project-scoped assignments,
     so a removed member cannot keep authorizing through stale role facts.
   - Return target member data with public member ID, removed status, and no
     active roles.
   - Do not add member reactivation or reinvitation behavior in this task.

4. Enforce authorization and membership invariants.
   - Role update must authorize with action
     `organization.member.update_role`.
   - Removal must authorize with action `organization.member.remove`.
   - Authorization must use the existing membership-worker ->
     policy-worker service-binding seam and fail closed on missing binding,
     fetch errors, malformed policy responses, or denial.
   - Policy denial should return 404 to avoid organization/member enumeration,
     matching existing protected membership routes.
   - Invalid public organization or member IDs should return 404.
   - Missing DB remains a safe 503/internal error path consistent with existing
     handlers.
   - Do not allow an organization to lose its last active owner:
     - removing the only active owner must fail with `precondition_failed`;
     - changing the only active owner's role to a non-owner role must fail with
       `precondition_failed`;
     - no membership mutation or event/audit write should occur on these
       precondition failures.
   - If the actor modifies their own membership, the same last-owner invariant
     applies; do not invent a separate self-action policy unless the active
     specs already require it.

5. Keep persistence and event/audit atomic.
   - Use `TransactionalSqlExecutor` in production membership-worker paths.
   - The membership mutation and `appendEventWithAudit(...)` call must execute
     inside the same transaction callback.
   - If the membership mutation fails, do not append an event.
   - If event/audit append fails or conflicts, the membership mutation must roll
     back.
   - Repository methods may use multi-statement logic when called from the
     transaction-bound executor, but keep the repository API typed and focused.
   - Prefer no database migration. If you prove a schema change is necessary for
     safe behavior, stop and write a spec proposal instead of silently expanding
     this task.

6. Define safe event/audit contents.
   - Role update event:
     - type `membership.updated`
     - version `1`
     - source `membership-worker`
     - subject kind `member`
     - subject ID as public `mem_...`
     - org ID as public `org_...`
     - payload may include `previousRoles` and `role`
   - Removal event:
     - type `membership.removed`
     - version `1`
     - source `membership-worker`
     - subject kind `member`
     - subject ID as public `mem_...`
     - org ID as public `org_...`
     - payload may include `previousRoles` and `revokedRoleCount`
   - Actor must come from the authenticated actor context forwarded by api-edge.
   - Include `requestId` in event trace fields.
   - Audit category should be `membership`.
   - Audit descriptions should be human-readable and secret-free.
   - Do not include bearer tokens, invitation tokens, token hashes, raw database
     UUIDs, SQL text, stack traces, provider details, or unrelated identity
     profile data in event/audit payloads or descriptions.
   - Avoid target email entirely; members currently expose subject IDs, not
     email addresses.

7. Keep tests broad enough for this larger PR.
   - DB tests should cover:
     - role update uses parameterized SQL;
     - role update revokes old active organization-scoped roles and creates or
       preserves the requested role;
     - same-role no-op behavior;
     - last-owner role update precondition failure;
     - removal marks the member removed and revokes active role assignments;
     - last-owner removal precondition failure;
     - not-found/removed member behavior.
   - Membership-worker tests should cover:
     - successful role update authorizes through policy and appends
       `membership.updated`;
     - successful removal authorizes through policy and appends
       `membership.removed`;
     - event/audit append failure returns a safe error and prevents commit in
       the production transaction path or an equivalent transaction fake;
     - policy denial appends no event;
     - invalid org/member IDs append no event;
     - invalid role body appends no event;
     - target not found/removed appends no event;
     - last-owner failures append no event;
     - missing policy-worker and missing DB remain safe;
     - responses use public IDs and safe role summaries;
     - event/audit values do not expose raw UUIDs, bearer tokens, SQL, stack
       traces, or provider details.
   - Api-edge tests should cover:
     - `PATCH` and `DELETE` member routes are recognized and forwarded;
     - `PATCH` body reaches membership-worker;
     - unsupported methods on member item routes return 405/unsupported;
     - actor headers are forwarded after auth resolution;
     - bearer tokens are not forwarded to membership-worker.
   - Policy-engine tests should only change if code reality shows the existing
     action coverage is incomplete.

# Integration Notes

- `listRoleAssignments(orgId, actor.subjectId)` is the current source of actor
  policy facts for membership-worker protected routes.
- Existing policy actions already include `organization.member.remove` and
  `organization.member.update_role`.
- Existing list-members behavior currently performs per-member role lookups; do
  not broaden this task into list performance work.
- Existing `removeMember(...)` marks a member removed but does not revoke role
  assignments. It is not safe enough by itself for the public removal route.
- Role assignment rows are append/revoke style. Prefer revoking old active role
  assignments and creating a new assignment over mutating old role rows in
  place.
- `appendEventWithAudit(...)` writes the canonical event and audit projection in
  one SQL statement, but this task needs transaction atomicity between that
  append and the membership mutation.
- If you discover the last-owner invariant cannot be enforced reliably without
  a schema change or stronger locking than this task can safely add, write a
  proposal and keep implementation conservative.

# Acceptance Criteria

- `PATCH /v1/organizations/{orgId}/members/{memberId}` is available through
  api-edge, updates the member's organization role when authorized, and appends
  one `membership.updated` event plus one audit entry when a real mutation
  occurs.
- `DELETE /v1/organizations/{orgId}/members/{memberId}` is available through
  api-edge, removes the member when authorized, revokes stale role facts, and
  appends one `membership.removed` event plus one audit entry.
- Both mutation routes fail closed on auth, policy, validation, precondition,
  repository, missing binding, and event append failures.
- Removing or demoting the last active owner is prevented with
  `precondition_failed`.
- Removed members cannot retain active role facts that would authorize future
  policy decisions.
- Event/audit append failure rolls back the corresponding membership mutation.
- Existing organization, member list, invitation, and auth facade tests remain
  compatible.
- No infrastructure, migration, UI, SDK, CLI, or `specs-v2/**` changes are
  included.
- A GitHub PR is opened for exactly this task.
