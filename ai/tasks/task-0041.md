# Current Repo Context

Task 0040 is verified PASS and merged via PR #81 at `9740ca1`. The current web
console is deployed as one Cloudflare Pages project:

`https://sourceplane-web-console.pages.dev/`

That single console currently contains a stage/prod target selector. The product
direction is now different: the platform should expose two separate web-console
deployments, one for stage and one for prod, with different Pages URLs. Each
deployed console should be environment-bound, not a cross-environment selector.

Use these canonical Pages projects and URLs unless code reality proves they are
invalid:

- stage console project: `sourceplane-web-console-stage`
- stage console URL: `https://sourceplane-web-console-stage.pages.dev/`
- prod console project: `sourceplane-web-console-prod`
- prod console URL: `https://sourceplane-web-console-prod.pages.dev/`

The existing `sourceplane-web-console` Pages project may remain as a historical
resource, but it must stop being the canonical deployment target and must not be
used by smoke tests or compact context after this task.

# Objective

Split the web-console deployment into environment-specific stage and prod Pages
consoles, bind each deployed console to the correct API edge, and update CORS
and smoke verification so stage and prod console URLs are independently live and
safe to test.

# Required Outcomes

1. Deploy two separate Cloudflare Pages projects.
   - Stage deploys to `sourceplane-web-console-stage`.
   - Prod deploys to `sourceplane-web-console-prod`.
   - Stage and prod must have distinct `pages.dev` URLs.
   - The old `sourceplane-web-console` project must no longer be the deploy or
     smoke target.

2. Make deployed console builds environment-bound.
   - The stage console must target only
     `https://api-edge-stage.rahulvarghesepullely.workers.dev`.
   - The prod console must target only
     `https://api-edge-prod.rahulvarghesepullely.workers.dev`.
   - Deployed stage/prod consoles must not expose a control that switches to the
     other environment.
   - Local development may keep a multi-target selector if useful, but deployed
     builds must be locked to their environment.
   - The current environment must remain visible in the header or equivalent
     persistent scope display.

3. Make the Pages composition support this safely.
   - Preserve backward compatibility for existing `cloudflare-pages-turbo`
     consumers unless there is a documented reason not to.
   - Prefer a generic composition enhancement, such as an optional environment
     project-name suffix or environment-aware project-name mode, instead of
     hardcoding Sourceplane-specific names in the composition.
   - Ensure the build step exposes the Orun environment to Vite with a `VITE_`
     variable so the web-console can lock its deployed target.
   - Ensure smoke steps can tell which Orun environment they are checking.

4. Update CORS to match the two-console model.
   - Stage API should allow the stage console origin and appropriate stage
     preview origins.
   - Prod API should allow the prod console origin and appropriate prod preview
     origins.
   - Do not allow wildcard origins.
   - Do not allow the stage console origin to call prod API or the prod console
     origin to call stage API.
   - Localhost/127.0.0.1 can remain allowed for local development, but avoid
     granting localhost broad prod access unless the task documents why it is
     needed.
   - Update CORS tests for the full origin/environment matrix.

5. Update smoke verification.
   - Stage deploy job must smoke the stage console URL and stage API health.
   - Prod deploy job must smoke the prod console URL and prod API health.
   - Smoke must not check `https://sourceplane-web-console.pages.dev/` as the
     canonical console.

6. Update specs/context where needed.
   - Clarify that hosted web consoles are environment-specific clients of
     `api-edge`.
   - Clarify that CORS allowlists should map console origins to their matching
     API environment.

# Constraints

- Keep bearer tokens out of logs, URLs, reports, and screenshots.
- Keep CORS explicit and environment-aware.
- Keep the web-console a replaceable public API client; do not call private
  Workers or service-binding-only routes.
- Keep Task 0040 functionality intact for stage: login with debug code, org
  creation/selection, project/environment flows, audit inspection, and logout.
- Prod console must preserve prod-safe behavior: login/start may be exercised,
  but prod must not expose debug codes or require test data mutation.
- Any Cloudflare Pages project creation/update must be done through the existing
  Orun/Cloudflare CI path and verified with non-secret evidence.

# Acceptance Criteria

- `apps/web-console` can build a stage-locked console and a prod-locked console.
- The deployed stage console URL is
  `https://sourceplane-web-console-stage.pages.dev/`.
- The deployed prod console URL is
  `https://sourceplane-web-console-prod.pages.dev/`.
- Stage console calls only stage API; prod console calls only prod API.
- Deployed consoles do not show a cross-environment target selector.
- `api-edge` CORS allows only matching console origins for each environment,
  plus explicitly justified dev origins.
- CORS tests cover:
  - stage console → stage API allowed;
  - stage console → prod API denied;
  - prod console → prod API allowed;
  - prod console → stage API denied;
  - legacy `sourceplane-web-console.pages.dev` no longer canonical.
- Orun changed plan includes the required web-console and api-edge jobs.
- PR is pushed and opened on GitHub.
