## Root Cause (orchestrator analysis)

The orun composition `cloudflare-pages-turbo` (see
`stack-tectonic/compositions/cloudflare-pages-turbo/jobs/cloudflare-pages-turbo-verify-deploy.yaml`,
step `deploy-pages-artifact`) deploys only the directory at
`parameters.outputDir`. For `web-console-next` that is `.open-next/assets`,
which contains **only** the static Next.js chunks (`_next/**`) and a
`BUILD_ID` file. There is no `index.html`, no `_worker.js`, and no
`_routes.json`, because every route in the app is SSR/dynamic (root page is
`redirect("/orgs")`, all `/orgs/**` pages are server components).

The Worker bundle that OpenNext emits at `.open-next/worker.js` (with the
matching `apps/web-console-next/wrangler.jsonc` declaring `main`,
`compatibility_flags: [nodejs_compat, global_fetch_strictly_public]`, and the
`ASSETS` binding) is never deployed. CI logs even surface the smoking gun:

> `WARNING: We detected a configuration file at apps/web-console-next/wrangler.jsonc
> but it is missing the "pages_build_output_dir" field, required by Pages.
> ... Ignoring configuration file for now, and proceeding with project deploy.`

`@opennextjs/cloudflare` upstream targets **Workers + Static Assets** (Pages
support is deprecated). The current `cloudflare-pages-turbo` composition does
`wrangler pages deploy <static-dir>`, which is fine for the vanilla Vite
`apps/web-console`, but is structurally wrong for the OpenNext output.

## Objective

Make `sourceplane-web-console-next-stage.pages.dev` (and the dev/prod
siblings, on their next deploy) return HTML containing `Sourceplane Console`
when the post-merge main CI deploy runs, without regressing the existing
`apps/web-console` (vanilla Vite) `cloudflare-pages-turbo` deploy path.

### Approach A (preferred â€” Workers + Static Assets)

1. Add a new orun composition `cloudflare-workers-assets-turbo` (or
   `cloudflare-pages-opennext`) under `stack-tectonic/compositions/`
   modelled on `cloudflare-worker-turbo` but with:
   - Build step that runs the OpenNext build (already happens through
     `pnpm exec turbo run build --filter=@saas/web-console-next`).
   - `verify-output` checks `.open-next/worker.js` and `.open-next/assets/**`.
   - Deploy step is `pnpm exec wrangler deploy --config wrangler.jsonc` (no
     `--env` flag â€” `web-console-next` uses per-environment Pages-project
     **naming**, not wrangler envs, today; preserve that by parameterising
     the project / worker name with `{{ .orun.environment.name }}` exactly
     as `cloudflare-pages-turbo` does for `environmentAwareProjectName`).
   - No `migrate` / `pre-deploy` / `typecheck` steps (those are
     `cloudflare-worker-turbo`-specific and would break for this app).
   - `smoke` step identical in shape to today (curl the deployed hostname).
2. Define `cloudflare-workers-assets-turbo-{verify,deploy,pull-request}`
   profiles mirroring the existing trio.
3. Re-wire **only** `apps/web-console-next/component.yaml` to
   `spec.type: cloudflare-workers-assets-turbo`, adjusting `parameters`
   accordingly (drop `outputDir: .open-next/assets`, keep
   `projectName: sourceplane-web-console-next`, keep
   `environmentAwareProjectName: true`, keep `environmentBuildVar:
   NEXT_PUBLIC_DEPLOY_ENV`).
4. `apps/web-console/component.yaml` is **byte-identical** before and after.
5. Update `apps/web-console-next/wrangler.jsonc` only if needed to satisfy
   the new deploy command (it already has `main`, `assets`, and the right
   compatibility flags â€” likely no change).

### Approach B (fallback â€” stay on Pages)

1. Modify the `cloudflare-pages-turbo` composition's `deploy-pages-artifact`
   step (or add a new optional `pre-deploy-stage` step) so that, when a
   parameter such as `parameters.openNextBundle: true` is set, the step:
   - Creates a temp dir.
   - Copies `.open-next/assets/**` into `<tempdir>/`.
   - Copies `.open-next/worker.js` to `<tempdir>/_worker.js`.
   - Runs `wrangler pages deploy <tempdir>` with
     `--compatibility-flags nodejs_compat`.
2. Set `openNextBundle: true` (or equivalent) only in
   `apps/web-console-next/component.yaml`.
3. `apps/web-console/component.yaml` and behaviour unchanged.

Approach A is preferred because Pages support in `@opennextjs/cloudflare`
is upstream-deprecated; Approach B is acceptable as a fast unblock but
requires a documented follow-up to migrate.
