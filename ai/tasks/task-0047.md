This PR should introduce the migration, manifest entry, repository types, and repository methods needed to store service principals and API-key records safely, without exposing new public routes, changing auth resolution, or adding UI. The task satisfies Product Overview required capability `API keys and service principals` while keeping bounded-context ownership clear: identity owns credentials, membership/policy own role facts and authorization.

## Required Outcomes

- [ ] Add an identity-owned migration for service principals and API keys.
  - Service principals must be organization-bound and may optionally carry explicit project scope.
  - API keys must belong to a service principal and preserve explicit organization scope.
  - The schema must not store raw API-key secrets or bearer material; only safe derivations such as hash and public prefix may be persisted.
  - The migration must be idempotent and safe for the existing autocommit db-migrate runner.

- [ ] Preserve bounded-context ownership and tenancy invariants.
  - Identity owns user/session/API-key facts.
  - Membership and policy remain the source of role assignments and authorization decisions.
  - Do not add cross-context foreign keys into `membership.*`, `projects.*`, `events.*`, or future billing tables.
  - If project scope is represented, it must carry explicit `org_id + project_id` semantics rather than `project_id` alone.

- [ ] Extend the identity repository surface for the new persistence seam.
  - Add the types and methods needed for later API-key admin and auth-runtime tasks.
  - At minimum, support create/list/revoke-style persistence primitives for API keys and create/get-style primitives for service principals.
  - Add any narrow hash/prefix lookup needed by the storage design only if it remains purely persistence-layer work and is clearly documented for later runtime tasks.

- [ ] Add focused db verification.
  - Migration tests must verify ordering, bounded-context ownership, and secret-safe schema design.
  - Repository tests must verify the intended persistence behavior for service-principal and API-key records.
  - Tests should explicitly confirm that raw secret material is not stored or returned from the persistence seam.

- [ ] Keep the PR foundation-only.
  - Do not add any public API contract, UI, or auth-resolution behavior in this PR.
  - If you discover a follow-on requirement for runtime/admin surfaces, record it in the report as a next-task dependency instead of widening scope here.

## Constraints

1. Follow Constitution rule 8 (`secure by default`): raw API-key values, token material, and hashes must never appear in reports, logs, or response shapes outside the strictly necessary persistence fields.
2. Follow Constitution rule 11 (`bounded contexts are real`): do not couple identity persistence directly to membership or projects tables with cross-context foreign keys.
3. Follow `specs/contracts/tenancy-and-rbac.md`: API keys and service principals must be bound to an organization; project-scoped automation must also carry explicit project scope under that organization.
4. Keep this as a persistence-only PR. No `api-edge`, no `identity-worker` route surface, no web-console.
5. Preserve extraction seams: later runtime and UI tasks must consume this through repository and contract boundaries, not ad hoc SQL spread across apps.
6. Do not stage unrelated local `ai/` carryover files from older tasks.
