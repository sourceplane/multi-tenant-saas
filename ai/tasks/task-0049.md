# Task 0049

Agent: Implementer

## Current Repo Context

- Task 0047 is verified PASS and merged via PR #90 at `08c0e7b`. Identity now owns persisted `service_principals` and `api_keys` records through migration 060 plus repository create/get/list/revoke primitives.
- Task 0048 is verified PASS and merged via PR #91 at `2a78a5d`. Identity resolves both session tokens and active API keys, and `api-edge` now forwards `x-actor-subject-id` / `x-actor-subject-type` for `service_principal` actors without forwarding raw bearer tokens downstream.
- Current repo reality still has no service-principal role-binding administration seam. `membership-worker` and `policy-worker` already accept `subject.type = service_principal`, and membership persistence already supports `subject_type = service_principal`, but there is no scoped route/contract for creating, reading, or revoking those bindings.
- Current inspection found no established `api_key.*` or `organization.service_principal.*` policy action surface in apps/packages/tests, and `apps/identity-worker/src/env.ts` still has no membership/policy/events bindings. That makes public API-key administration too wide for the next PR.
- The highest-leverage next slice is to make authenticated service-principal actors authorizable through the existing membership → policy pipeline without widening into public API-key admin routes, UI, or identity-owned orchestration.

## Objective

Add the membership/policy foundation for service-principal role bindings so `service_principal` actors authenticated by Task 0048 can participate in the existing authorization pipeline.

This PR should introduce a narrow membership-owned internal seam for reading and mutating service-principal role bindings, plus the minimal policy action surface needed to protect that seam. The slice satisfies `specs/contracts/tenancy-and-rbac.md` requirements that API keys and service principals are bound to an organization and role set, while preserving bounded-context ownership: identity validates credentials; membership owns role assignments; policy owns authorization decisions.

## PR Boundary

This PR may include only:

1. `packages/contracts/src/**` only for narrowly scoped internal contract types if the membership/policy seam needs them.
2. `packages/policy-engine/src/**` and `apps/policy-worker/src/**` for the minimal service-principal role-binding policy action surface.
3. `apps/membership-worker/src/**` plus `packages/db/src/membership/**` only as needed for an internal service-principal role-binding read/write seam built on the existing role-assignment repository.
4. Focused tests in `tests/membership-worker/**`, `tests/policy-worker/**`, `tests/policy-engine/**`, and minimal task-scoped `ai/` report files required by workflow.

Explicit non-goals for this PR boundary:
- no public `GET /v1/auth/api-keys`, `POST /v1/auth/api-keys`, or `DELETE /v1/auth/api-keys/{apiKeyId}` routes
- no `apps/identity-worker` membership/policy/events bindings
- no `apps/api-edge` public route expansion beyond any strictly necessary internal test fixture updates
- no service-principal web-console, CLI, or SDK UI
- no billing, metering, notifications, or webhook work
- no `specs-v2/**` work

## Required Outcomes

- [ ] Add a membership-owned internal seam for service-principal role bindings.
  - The seam must support reading active bindings for a `service_principal` subject within an explicit organization scope.
  - The seam must support creating and revoking the bindings needed for later API-key admin/runtime work, using existing membership role-assignment persistence where possible.
  - Keep the seam internal-only for now. Do not expose it through `api-edge`.

- [ ] Add the minimal policy surface needed to protect that seam.
  - Introduce only the narrow action set required for service-principal role-binding read/write administration.
  - Owner/admin callers should be able to manage service-principal bindings; authorization must remain deny-by-default for everyone else.
  - Keep action naming and permission mapping consistent with existing `organization.*` / `project.*` patterns.

- [ ] Preserve canonical subject-ID compatibility with Task 0048.
  - The service-principal subject identifier used for role bindings must match the opaque actor identifier that downstream workers receive via `x-actor-subject-id` for authenticated `service_principal` actors.
  - Do not invent tenant guessing or implicit `projectId`-only scoping.
  - If a helper is needed to avoid ID-shape drift, keep it small and shared rather than duplicating string-shaping logic.

- [ ] Add focused tests that prove service-principal authorization can be wired safely.
  - Membership tests must cover valid `service_principal` binding creation/read/revoke behavior plus invalid subject/scope input.
  - Policy tests must cover allow/deny behavior for the new action surface.
  - If you add internal route handlers, tests must prove fail-closed behavior and no cross-org leakage.

- [ ] Keep the PR foundation-only.
  - Do not add public API-key administration routes here.
  - Do not add identity-worker orchestration across membership/policy/events.
  - Record the public API-key admin follow-on in the implementer report instead of widening scope.
