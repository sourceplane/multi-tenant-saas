## Architect Brief

- **Product bar:** Stripe-quality webhook-endpoint lifecycle UX ŌĆö disable
  and re-enable are visibly symmetric, audit-legible, no surprises.
- **User moment:** an org admin notices a webhook was paused (by ops, by
  failure quarantine, or by a teammate) and re-activates it from the
  Console without a delete/recreate detour.
- **Free to make** (record one-line rationale in implementer report):
  console copy, button placement on the detail page (replacing the inline
  notice card), confirm-modal shape (mirror disable, or omit since
  re-enable is additive ŌĆö your call), and the audit-event row schema for
  `webhook_endpoint.enabled` as long as it mirrors `webhook_endpoint.disabled`.
- **Must not make without a spec proposal:** no schema migration on
  `webhooks.webhook_endpoints` (the existing `status` / `disabled_at` /
  `disabled_reason` columns already cover enable ŌĆö clearing them is the
  whole job); no change to the `/disable` route shape; no change to
  `UpdateWebhookEndpointRequest` (a flip via PATCH was explicitly rejected
  by the spec proposal ŌĆö symmetric route is required); no new public route
  outside the existing `/v1/organizations/:orgId/webhooks/endpoints/:id/*`
  family.
- **Failure modes that invalidate the PR even if tests pass:** raw
  `precondition_failed` strings reaching the user; missing audit row on
  enable while disable still emits one; UI primitive rendered outside the
  design system; a re-enabled endpoint that does not actually deliver the
  next event; bypassing api-edge.

## Objective

Add a webhook-endpoint **re-enable** path symmetric to disable: contract,
SDK method, api-edge facade hop, webhooks-worker route, db repo function,
audit + event emission, and console wiring (replacing the inline notice
card on the detail page). When a `disabled` endpoint is re-enabled it
returns to `status: "active"`, `disabled_at` and `disabled_reason` clear,
a `webhook_endpoint.enabled` event + audit row are appended, and the next
matching event resumes delivery.

## Constraints

1. Symmetric route shape: `POST .../endpoints/:endpointId/enable` only.
   Do not flip status via `PATCH .../endpoints/:endpointId`.
2. Atomicity: event + audit emit must roll back if mutation fails, and
   mutation must roll back if event/audit emit throws. Mirror Task 0024
   `revoke-invitation.ts` pattern; verify with the worker test seam.
3. No secrets logged. Endpoint signing secret must not appear in any new
   log line, response body, or test fixture.
4. Idempotency-key forwarding must mirror disable; do not invent a new
   header path.
5. The disabled-state inline notice card on the detail page must be
   removed only when the working button replaces it. Do not orphan the
   user with a button that 500s.
6. `pnpm -w typecheck` 0 errors, `pnpm -w lint` 0 new errors,
   `pnpm -w test` strictly increases the passing-test floor by at least
   +6 net.
7. `kiox -- orun validate --intent intent.yaml`, `kiox -- orun plan
   --changed --intent intent.yaml --output plan.json`, and `kiox -- orun
   run --plan plan.json --dry-run --runner github-actions` all green
