# Required Outcomes

1. Clarify the list pagination contract.
   - Add a concise clarification to `specs/contracts/api-guidelines.md` for V1
     list endpoints:
     - query params: `limit` and `cursor`;
     - default limit: `50`;
     - maximum limit: `100`;
     - cursors are opaque and endpoint-owned;
     - invalid `limit` or `cursor` returns `validation_failed`;
     - the next cursor is returned in response `meta.cursor`;
     - no total count is required.
   - Keep this as a clarification of the existing cursor-pagination rule, not a
     broad API rewrite.

2. Add membership pagination primitives.
   - Add a small membership-worker pagination helper for parsing `limit` and
     `cursor` from query params and encoding/decoding opaque cursors.
   - Cursor format must be versioned and opaque to clients.
   - Cursor decode failures must produce a safe `validation_failed` response.
   - Do not leak raw SQL, stack traces, or provider details in cursor errors.
   - Prefer Web/Worker-safe APIs over Node-only globals in Worker code.

3. Add paginated repository methods while preserving existing ones.
   - Keep existing unpaginated repository methods available so unrelated code is
     not broken.
   - Add paginated methods for:
     - organizations for subject;
     - active members for organization.
   - Use deterministic ordering and `limit + 1` reads to determine whether a
     next cursor exists.
   - Cursor comparisons must include both a timestamp and stable ID tie-breaker.
   - No database migration is needed.
   - Add DB repository tests proving SQL uses parameterized queries, stable
     ordering, limit-plus-one behavior, cursor filtering, and safe empty results.

4. Wire pagination into membership-worker routes.
   - `GET /v1/organizations` must parse `limit` and `cursor` and return only the
     requested page.
   - `GET /v1/organizations/{orgId}/members` must parse `limit` and `cursor`
     and return only the requested page.
   - Response envelopes must set `meta.cursor` to the next opaque cursor or
     `null`.
   - Existing default behavior without query params should return the first page
     with default limit.
   - `GET /v1/organizations/{orgId}/members` must continue to authorize through
     policy-worker using `organization.member.list` before listing members.
   - Policy denial, actor role-list failure, member-list failure, member
     role-list failure, malformed policy responses, and missing bindings must
     still fail closed.
   - Existing organization create/read behavior must remain unchanged.

5. Preserve api-edge facade behavior.
   - Confirm `api-edge` preserves query strings when forwarding both list routes.
   - Add tests for forwarding `limit` and `cursor` query params to
     `membership-worker`.
   - Continue resolving auth through `IDENTITY_WORKER`.
   - Continue not forwarding raw bearer tokens to `membership-worker`.

6. Add or update tests.
   - Membership-worker tests must cover:
     - default limit behavior;
     - explicit valid limit;
     - invalid limit;
     - valid cursor forwarded to the repository page call;
     - invalid cursor returns `validation_failed`;
     - `meta.cursor` is set when another page exists and `null` otherwise;
     - member list still authorizes before page query;
     - failure modes from Task 0019 remain safe.
   - Api-edge tests must cover query-string passthrough for both list routes.
   - DB tests must cover paginated repository SQL and mapping.
