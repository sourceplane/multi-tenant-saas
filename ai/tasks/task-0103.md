  (`specs/roadmap.md` §U10), states "Depends on: B4." B4 is now closed.
  U10 requires `apps/web-console-next` to drop its bespoke
  `src/lib/api.ts` (297 LOC, 8 consumer call sites) and consume the
  `@saas/sdk` resource clients exclusively.
- Gap blocking U10: the SDK has **no `AuthClient`** today. The console's
  `ApiClient` exposes 5 auth methods used at runtime — `loginStart`,
  `loginComplete`, `getSession`, `logout`, `getProfile` — none of which
  have an SDK equivalent. Without an `AuthClient`, U10 would either
  reach into `client.transport.request` (the same workaround pattern
  Task 0101 shipped and Task 0102 had to clean up) or keep the bespoke
  `ApiClient` for auth-only, defeating U10's "single client surface"
  premise.
- Edge routes already exist and are stable (`apps/api-edge/src/auth-facade.ts`
  declares `/v1/auth/login/start`, `/v1/auth/login/complete`,
  `/v1/auth/session`, `/v1/auth/resolve`, `/v1/auth/logout`,
  `/v1/auth/security-events`; `/v1/auth/profile` accepts GET + PATCH).
  Contract types live in `packages/contracts/src/auth.ts`.

## Objective

Add a 13th resource client `AuthClient` to `@saas/sdk` that wraps the
identity-worker public auth surface, mirroring the
`EnvironmentsClient` / `EventsClient` cadence (encodeURIComponent on
every dynamic segment, caller-owned `idempotencyKey` on POSTs, types
imported directly from `@saas/contracts/auth`). This is the last
SDK-side dependency before Task 0104 (U10 console refactor) becomes a
pure consumer-side swap.

## PR Boundary

1. NEW `packages/sdk/src/auth.ts` exporting `AuthClient` with these
   methods, all mirroring the wire shape `auth-facade.ts` enforces:
   - `loginStart(input: LoginStartRequest, opts?: RequestOptions): Promise<LoginStartResponse>`
   - `loginComplete(input: LoginCompleteRequest, opts?: RequestOptions): Promise<LoginCompleteResponse>`
   - `getSession(opts?: RequestOptions): Promise<SessionResponse>`
   - `logout(opts?: RequestOptions): Promise<LogoutResponse>`
   - `getProfile(opts?: RequestOptions): Promise<ProfileResponse>`
   - `updateProfile(input: UpdateProfileRequest, opts?: RequestOptions): Promise<ProfileResponse>` (PATCH /v1/auth/profile)
2. `packages/sdk/src/index.ts`: import + wire `AuthClient` onto the
   `Sourceplane` class as `readonly auth: AuthClient`. Construct it
   from the same `Transport` instance every other client uses.
3. NEW test file `packages/sdk/src/__tests__/auth.test.ts` with at
   minimum: URL-shape proof per method, idempotencyKey passthrough on
   POSTs, typed error decoding (Unauthenticated / RateLimit / Validation
   / Internal) on the auth surface, requestId surfacing through the
   envelope. Target: ≥10 new `it()` blocks, mirroring prior client tests.
4. NO changes to `packages/sdk/src/transport.ts` (its public API is
   sealed). NO changes to other resource clients. NO changes to
   `packages/contracts/**`. NO changes to `packages/cli/**` or
   `apps/**` — Task 0104 will consume `client.auth.*` separately.

## Constraints

1. `AuthClient` constructor signature MUST match the
   `EnvironmentsClient` pattern: `constructor(private transport: Transport)`,
   no other state. Re-use the same `Transport` instance from `Sourceplane`.
2. Caller-owned `Idempotency-Key` on POSTs (Stripe parity). The SDK
   MUST NOT auto-generate idempotency keys for `loginStart` /
   `loginComplete` / `logout`. Pass through `opts.idempotencyKey`
   verbatim if provided.
3. Every POST/PATCH method MUST go through `transport.request<T>`
   (envelope-aware) — same pattern Task 0102 used for `EnvironmentsClient`.
   Do NOT use `transport.fetchImpl` directly.
4. No `eslint-disable`, `@ts-ignore`, `@ts-expect-error`, `as unknown as`,
   or `as any` under `packages/sdk/**`. Hazard scan must come back
   clean (same ban as Tasks 0096b–f, 0097, 0098, 0099, 0100, 0101, 0102).
5. No `node:*` imports. `@saas/sdk` runs on browser, Node ≥20, Workers,
   and Bun.
6. Types imported directly from `@saas/contracts/auth` subpath. Do
   NOT redefine `LoginStartResponse` / `LoginCompleteResponse` /
   `SessionResponse` / `ProfileResponse` / `LogoutResponse` locally.
7. Tests use the same in-memory `fetchImpl` mock pattern as
   `packages/sdk/src/__tests__/resources.test.ts` and
   `packages/sdk/src/__tests__/events.test.ts`.
8. No public-API deletions on the existing `Sourceplane` surface (the
   property add is additive only).
9. Update `packages/sdk/component.yaml` ONLY if the implementer
   actually changes `spec.path` (they should not). Otherwise leave
   byte-identical vs `main`.

## Integration Notes

- `auth-facade.ts` returns the same JSON envelope all other facades
  return: `{ data, meta: { requestId, cursor? } }` on success,
  `{ error: { code, message, requestId? }, meta }` on failure.
  `transport.request<T>` already handles this — no special-casing.
- `getProfile` is GET, `updateProfile` is PATCH (per
  `auth-facade.ts:15`: `"/v1/auth/profile": new Set(["GET", "PATCH"])`).
- `securityEvents` is intentionally OUT of `AuthClient` scope — it
  already lives in `SecurityEventsClient` (Task 0099). Do not duplicate.
- `/v1/auth/resolve` is internal (used by service-binding resolution);
  do NOT expose it on `AuthClient`. Keep public surface narrow.
