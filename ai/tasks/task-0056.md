## Objective

Create the first deployable `config-worker` runtime and public read-only config API surface through `api-edge` for organization/project/environment-scoped settings, feature flags, and secret metadata.

This satisfies the Constitution rules for bounded contexts, API/UI parity, secure-by-default secret handling, and organization/project/environment scope invariants while deferring mutating config APIs to a follow-up task.

## PR Boundary

1. Add `apps/config-worker` as a private Cloudflare Worker component with the same environment/deployment pattern as `projects-worker` and `events-worker`.
2. Add shared public contract types for read-only config list responses and request/query validation.
3. Expose read-only list routes through `api-edge` and `config-worker` for:
   - organization scope: `/v1/organizations/{orgId}/config/settings`, `/feature-flags`, `/secrets`
   - project scope: `/v1/organizations/{orgId}/projects/{projectId}/config/settings`, `/feature-flags`, `/secrets`
   - environment scope: `/v1/organizations/{orgId}/projects/{projectId}/environments/{environmentId}/config/settings`, `/feature-flags`, `/secrets`
4. Add policy coverage for read-only config access and focused tests for contracts, policy, config-worker, and api-edge forwarding.

Explicit non-goals: no config mutations, no setting/flag create/update/delete routes, no secret value storage or reveal flow, no web-console UI, no KV cache, no config versioning/promotion, no specs-v2 work.

## Constraints

1. `config-worker` owns config runtime behavior. `api-edge` must remain only a public transport facade and must not query `@saas/db/config` directly.
2. Use `@saas/db/config` from Task 0055; do not duplicate SQL or bypass the repository from route handlers.
3. Use the existing membership-worker authorization-context seam before policy-worker; do not query membership storage directly.
4. Add policy actions deliberately. Suggested minimum:
   - `organization.config.read` for org-scoped list routes.
   - `project.config.read` or an equivalent project-scoped config read action for project/environment-scoped list routes.
   - Owner/admin/builder/viewer should have read access consistent with read-only role semantics; billing_admin should not gain operational config access unless the policy spec explicitly justifies it.
5. Preserve deny-by-default behavior. Policy denial should not expose cross-org resource existence.
6. Secret metadata routes are metadata-only. Never select or serialize `ciphertext_envelope`; never add `value`, `plaintext`, `secret`, `token`, or hash fields to public responses.
7. All public responses must use the established success/error envelope shape with `meta.requestId`; list responses include `meta.cursor`.
8. Keep route contracts `v1` backward-compatible and explicit. If the current specs are too ambiguous for exact route names or policy names, write a bounded proposal under `ai/proposals/` and implement the conservative read-only route shape above.
9. Do not stage unrelated untracked carryover `ai/` files. Before committing, inspect `git status --short` and `git diff --cached --name-only`.
