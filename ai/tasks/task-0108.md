## Architect Brief

- **Product bar:** Stripe-quality webhook signing-secret rotation. The
  buyer-shaped guarantee is "rotate returns the new secret exactly once,
  in plaintext, prefixed `whsec_`, never persisted in any log/event/audit
  payload, and deliveries during the configured grace window carry both
  the new and the previous signature so receivers can roll forward without
  downtime."
- **User moment this PR improves:** an org admin can rotate a webhook
  signing secret, see the new value once, copy it into their receiver,
  and ship the rotation without a delivery gap. (The console/CLI surface
  ships in 0109/0110; this PR makes those surfaces possible.)
- **Free to decide:** column names on the new migration (suggest
  `previous_secret_ciphertext`, `previous_secret_version`,
  `previous_secret_expires_at`), grace-window default (suggest **24h** â€”
  matches industry norm; expose as a per-env constant in
  `apps/webhooks-worker/src/config.ts` if one exists, else a
  `WEBHOOK_SECRET_ROTATION_GRACE_SECONDS` constant in the rotate handler
  with default `86400`), exact `whsec_` hex length (suggest 32 hex chars
  â†’ 128-bit), header name for the previous-key signature (suggest
  `X-Webhook-Signature-Previous` â€” symmetric with existing
  `X-Webhook-Signature`), test layout, and helper-function decomposition
  inside the worker.
- **Must not decide without a spec proposal:**
  - changing the canonical signature scheme (`HMAC-SHA256` over
    `${timestamp}.${body}`, hex, `sha256=` prefix);
  - changing the existing `X-Webhook-Signature` header name or value
    shape;
  - editing `packages/webhook-verifier/**` (locked from Task 0105 â€” the
    helper does NOT yet know about a "previous" signature header; that's
    a separate follow-on once this lands and the contract stabilises);
  - touching `packages/cli/src/commands/webhook-{verify,sign}.ts` (locked
    from 0106/0107);
  - introducing a second crypto primitive or alternate KDF.
- **Failure modes that invalidate the PR even if tests pass:**
  - any plaintext secret material reaching an event payload, audit row,
    log line, or `console.*` call;
  - the rotate response shape leaking `secretCiphertext`, the encryption
    envelope, or the previous secret;
  - a delivery during grace that carries the previous-key signature in
    `X-Webhook-Signature` (must be in `X-Webhook-Signature-Previous`);
  - a delivery after grace expiry that still emits
    `X-Webhook-Signature-Previous`;
  - a rotation that loses the previous secret (i.e. no grace window
    actually persisted), or one that overwrites a still-active previous
    secret without first checking expiry.
- **Recommended pattern:** mirror the existing
  `encryptSigningSecret(env)` helper for the new "produce + encrypt
  plaintext, return both" path. Decrypt previous secret in the same
  fashion as the current secret in `delivery.ts:243-253`. Make the
  rotation atomic at the SQL layer: a single `UPDATE` that copies
  `secret_ciphertext â†’ previous_secret_ciphertext`, sets
  `previous_secret_expires_at = now() + interval`, then writes the new
  secret + bumps `secret_version`.

## Objective

Ship the **backend half** of the B5 webhook secret rotation UX:

1. Schema and persistence support for a dual-secret window.
2. Worker-side dual-signature delivery during the window.
3. A reveal-once plaintext secret in the rotate response.
4. Contract + SDK type plumbing so console (0109) and CLI (0110) can
   consume it.
