Agent: Implementer

## Objective

Add a small public-API-backed account profile settings UI to `apps/web-console` so a signed-in user can view and update their own display name through `GET/PATCH /v1/auth/profile`.

This satisfies the Product Overview requirement for account settings and the Identity spec's self-scoped profile route contract, while keeping the web console a replaceable client of public api-edge routes only.

## PR Boundary

1. Add web-console API client methods for:
   - `GET /v1/auth/profile`
   - `PATCH /v1/auth/profile`
2. Add an account/profile settings view in `apps/web-console/src/main.ts` that lets an authenticated user:
   - load their current `email` and `displayName`;
   - update `displayName` only;
   - clear `displayName` by submitting an empty value or explicit clear control;
   - see success, validation, loading, and error states.
3. Keep the existing Account Security and workspace flows intact; profile settings may share the account-level view/navigation but must not remove security-event access.
4. Add only focused styling/state changes needed for this UI.

Explicit non-goals:

- No backend changes in identity-worker, api-edge, DB, policy, membership, or contracts unless a type-only import/export fix is strictly required.
- No email-change, MFA/passkey, password, session-management, or security-settings implementation.
- No organization-scoped profile/admin user management.
- No API-key behavior changes.
- No design-system overhaul or router rewrite.
- No `specs-v2/**` work.

## Constraints

1. The web console must remain a client of public `api-edge` routes only; do not call internal Workers or service-binding-only routes.
2. Profile routes are self-scoped by bearer session; the UI must not send a `userId`, `email`, `status`, role, organization, or other user-management field in the PATCH body.
3. API-key/service-principal actors are rejected by the backend; the UI may surface that backend error but must not attempt to bypass it.
4. Do not persist display name outside existing app state/local storage patterns unless necessary; avoid creating a second source of truth.
5. Do not log or display bearer tokens, API keys, raw secrets, or security-event sensitive metadata.
6. Keep the change PR-sized and frontend-focused. If backend contract drift is discovered, stop and write a proposal or report the blocker rather than expanding scope.
7. Existing generated/ignored files (`dist`, `node_modules`, `.orun`, `plan.json`, TypeScript build info) must not be staged.
