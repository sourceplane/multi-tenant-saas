## Objective

Land a Stripe-quality durable idempotency replay store at `api-edge`,
keyed on `(orgId | "anon", idempotencyKey, routePath)`, backed by
Cloudflare KV with a 24-hour TTL. A duplicate unsafe POST that arrives
within the TTL window with the same `Idempotency-Key` MUST replay the
exact stored response (status + headers + body) without forwarding to
the downstream worker, even across edge instances.

## Architect Brief

- **Product bar: Stripe API.** Idempotency-Key semantics must match
  Stripe's published contract: 24h cache, replay returns the exact
  original response, the second call must NOT re-execute the downstream
  side effect. Surface stable error envelopes â€” never leak KV / Hyperdrive
  internals through the wire.
- **User moment improved:** a customer integration that retries a
  network-flapped invitation POST gets one invitation, not two â€” closing
  the duplicate-pending-invitation open risk that's been parked since
  the membership-worker shipped.
- **Free to make** (record one-line rationale in the implementer report,
  do not stop to ask):
  - Exact KV key encoding (e.g. `idem:v1:{orgId}:{routePath}:{key}` or a
    hashed equivalent) â€” pick what's collision-safe, debuggable, and
    fits the 512-byte KV key cap.
  - Stored response envelope shape inside KV (JSON serialisation
    strategy, header allow-list, body encoding base64 vs raw text â€” your
    call as long as binary safety is preserved).
  - Whether to namespace by `routePath` literal or by a stable
    canonicalised route id â€” pick whichever makes audit / debug easier.
  - Whether org scope is derived from URL path (`/v1/organizations/{orgId}/...`)
    or from `resolveActor`. Either is acceptable; document the trade-off.
  - Pre-write "in-flight" lock semantics if you choose to add them
    (optional in v1 â€” Stripe accepts the small race window).
- **Must NOT make without a spec proposal:**
  - Do not change the `parseIdempotencyKey` contract surface in
    `@saas/contracts/idempotency` (callers depend on it).
  - Do not change error code names (`validation_failed`, etc.) or the
    400 status currently returned by the edge gate.
  - Do not introduce a new public route. Replay is internal seam only.
  - Do not bypass `resolveActor` ordering on the cache-miss path.
  - Do not touch `infra/terraform/cloudflare-domain/**` or the
    `cloudflare ~> 4.52` provider pin (deferred).
- **Failure modes that invalidate the PR even if tests pass:**
  - A second POST with the same valid key fires the downstream worker
    twice (real correctness regression â€” verifier MUST live-prove this).
  - A stored response leaks Hyperdrive errors, KV errors, or stack
    traces back to the caller.
  - The edge gate now rejects valid keys it used to accept (regression
    against Task 0094 acceptance).
  - KV namespace IDs hardcoded in `wrangler.jsonc` without Terraform
    creation in CI (rule: all CF resources programmatic via Orun).
  - A KV failure (binding missing, network error) **fails the request**
    instead of degrading to "execute downstream once, no replay." A
    Stripe-quality system fails open on the cache, never closed.
- **Recommended pattern (deviate only with documented one-line rationale):**
  - On unsafe POST with present-and-valid key: KV `get(key)`. Hit â†’
    return reconstructed `Response` from stored envelope. Miss â†’ forward
    downstream, capture response, KV `put(key, envelope, { expirationTtl: 86400 })`,
    return original response.
  - Make the replay logic a single chokepoint in `apps/api-edge/src/idempotency.ts`
    (extend the existing module â€” do not create a parallel seam). Each
    facade calls `await replayOrExecute(request, requestId, env, () => downstreamForward())`
    instead of (validate then forward). Keep the validation gate semantics
    inside the same helper so facades stay tidy.
  - KV failure path: log via existing observability hook, proceed as if
    cache-miss with no write, return real downstream result.
