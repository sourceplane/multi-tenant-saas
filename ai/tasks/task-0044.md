source repository.

challenge creation, login completion failures, session creation, and session
revocation/logout. Keep the work inside the identity bounded context and do not
expose public security-event query routes yet.

# Required Outcomes

1. Record source security events for successful auth mutations.
   - `login.challenge.created` (or a similarly precise identity-owned event
     type) when `startLogin` creates a login challenge.
   - `session.created` when `completeLogin` creates a session.
   - `session.revoked` when `logout` revokes a session.
   - Include raw UUID references internally (`userId`, `challengeId`,
     `sessionId`) where available; public IDs may appear only in safe metadata
     if needed, not as canonical internal references.

2. Record failed login-complete attempts when the request reaches the auth
   service.
   - Capture invalid challenge ID, bad code, expired challenge, and
     already-consumed challenge outcomes without storing the submitted code.
   - Use `userId` and `challengeId` only when safely known.
   - Keep failure metadata limited to safe values such as failure reason,
     challenge ID validity, and method. Never store raw code, code hash, bearer
     token, token secret, token hash, API key, or provider secret values.

3. Pass safe request context into recorded events.
   - Preserve the existing `x-request-id` behavior.
   - Include `requestId` on security events when available.
   - Include optional IP and user-agent if the handler can collect them from
     standard request headers without trusting them for authorization.
   - Optional correlation ID is fine if it follows the existing request/header
     conventions; do not invent a broad tracing system in this PR.

4. Keep security-event writes production-safe.
   - Do not silently ignore failures when recording required source events for
     successful auth mutations. Return a safe `internal_error` or document a
     narrow, tested recovery behavior.
   - For failed auth attempts, preserve user-facing auth error semantics while
     avoiding secret leakage. If a logging failure would change behavior, cover
     the chosen behavior in tests.
   - If the current non-transactional auth repository makes a required outcome
     unsafe, write a spec proposal under `ai/proposals/` and keep the
     implementation conservative.

5. Improve identity-worker tests around real source code.
   - Prefer tests that import the real `createAuthService` from
     `apps/identity-worker/src/services/auth.ts`.
   - Avoid expanding the existing mirrored auth-service implementation in
     `tests/identity-worker` unless you are explicitly replacing it.
   - Update the fake identity repository to store and expose security events for
     assertions.
   - Cover successful challenge/session/revoke events, failed complete-login
     events, request context propagation, and secret-safety.

# Constraints

- Preserve the shared event envelope rule: `tenant.orgId` remains required for
  shared events/audit, and pre-organization identity history stays in
  `identity.security_events`.
- Do not log or persist raw one-time codes, bearer tokens, session token
  secrets, token hashes, code hashes, API keys, or provider secrets.
- Keep debug delivery behavior unchanged: stage/dev may return raw login codes
  only through the existing `DEBUG_DELIVERY=true` response path, and prod must
  not expose codes.
- Do not change public auth response shapes except for safe error behavior
  needed when required security-event recording fails.
- Do not add unrelated cleanup, broad test rewrites, or README refreshes.

# Integration Notes

- The current `IdentityRepository` already exposes `recordSecurityEvent`.
- `recordSecurityEvent` stores UUIDs and audit-safe metadata in the identity
  schema only; it does not publish shared events.
- Prefer event IDs generated in identity-worker, for example by adding a
  `generateSecurityEventId` helper in `ids.ts`.
- If you add an auth-service context/options type, keep it small and explicit.
- For IP, prefer `cf-connecting-ip`; fall back to `x-forwarded-for` only as
  informational metadata. Never authorize from these headers.

# Acceptance Criteria

- Existing auth flows still work and keep their public response contracts.
- Successful login challenge creation records an identity security event.
- Successful login completion records a session-created security event.
- Failed login completion records an identity security event without storing
  the submitted code or hashes/secrets.
- Successful logout/session revocation records a session-revoked security event.
- Tests prove security-event metadata does not contain raw codes, bearer tokens,
  token secrets, token hashes, code hashes, API keys, or provider secrets.
- No public route, api-edge facade, web-console UI, shared audit-copy behavior,
  or migration changes are included.
- A GitHub PR is opened for the task.
