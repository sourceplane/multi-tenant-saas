# Task 0081

Agent: Implementer

## Current Repo Context

- Tasks 0078-0080 established billing entitlement checks as a private, provider-neutral service-binding seam and proved two callers: `projects-worker` gates project creation on `limit.projects`, and `membership-worker` gates invitation creation on `limit.members`.
- `apps/projects-worker` already has a `BILLING_WORKER` env binding for stage/prod, a `billing-worker` component dependency, and `apps/projects-worker/src/billing-client.ts` with the shared internal entitlement-call pattern.
- `POST /v1/organizations/{orgId}/projects/{projectId}/environments` is currently policy-gated and transactionally writes `environment.created` event/audit rows, but it does not yet enforce an environment quota.

## Objective

Gate environment creation on a billing entitlement named `limit.environments`, preserving projects/billing bounded-context separation and preventing environment rows or `environment.created` event/audit side effects when the organization is over plan limits.

## PR Boundary

1. Extend the projects-worker billing decision logic for `limit.environments` (reuse the existing service-binding client; add a small `decideEnvironmentsLimit` helper or equivalent rather than introducing a new billing protocol).
2. Add a projects-owned repository helper such as `countActiveEnvironments(orgId, projectId)` or clearly documented org-wide equivalent if the entitlement is intentionally organization-wide. The task default is project-scoped counting under the parent project, because environment APIs are explicitly `orgId + projectId` scoped.
3. Gate `handleCreateEnvironment` after request validation, membership authorization-context lookup, and `environment.create` policy allow, but before UUID generation for the environment/event/audit, `createEnvironment`, and `appendEventWithAudit`.
4. Add focused tests for projects-worker environment gating and DB count semantics. Existing Task 0079 project-create behavior must remain unchanged.
5. Write `ai/reports/task-0081-implementer.md`, create/reuse a task branch, commit, push, and open a GitHub PR.

## Required Outcomes

- [ ] Environment creation calls billing-worker's internal entitlement seam with `entitlementKey: "limit.environments"` and `x-internal-caller: projects-worker` through the existing service binding.
- [ ] Billing is called only after body validation, membership authorization-context lookup, and `environment.create` policy allow.
- [ ] Billing/count checks occur before environment UUID/event UUID/audit UUID generation, `createEnvironment`, and `appendEventWithAudit`.
- [ ] `limit.environments` semantics are explicit and tested: `allowed:true`, `valueType:"quantity"`, `limitValue:null` allows unlimited; finite non-negative limit allows only while active environment count is below the limit; disabled/not_configured/malformed_limit/limit_reached deny with safe 412 details; service/count failures return generic 503.
- [ ] Unauthorized or policy-denied callers do not call billing-worker.
- [ ] No cross-domain data coupling: projects-worker uses billing contracts/service binding only; billing-worker does not read projects persistence.
- [ ] Existing project-create `limit.projects` behavior remains covered and unchanged.

## Non-Goals

- No billing mutations, provider adapters, Stripe/payment SDK, webhook ingest, Secrets Store, Queue/KV/Durable Object/Analytics Engine, Terraform, or database migration.
- No membership-worker changes, policy-worker entitlement refactor, api-edge routing change, web-console UI change, or `specs-v2/**` work.
- No project-level billing model or per-project billing customer semantics. This is only a product quota check for environment creation.
- No changes to environment archive/list/get behavior except tests needed to keep the suite compiling.

## Constraints

1. Billing remains the owner of entitlement state. Projects-worker must not import `@saas/db/billing` or query `billing.*`.
2. Projects-worker owns project/environment persistence and counting. Billing-worker must not import or query projects persistence.
3. Policy remains mandatory: billing entitlement checks are an additional invariant, not a replacement for membership/policy authorization.
4. Denied or service-error requests must not generate environment/event/audit IDs, create rows, append events, or leak SQL/provider/internal details.
5. Keep the PR as one coherent environment-create quota gate. Do not add adjacent billing provider or UI behavior.
