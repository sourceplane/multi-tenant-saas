# Objective

Add the first web-console account-security surface by exposing the authenticated
identity security-event history in `apps/web-console`.

This PR should let a signed-in user review their own security events from the
existing public API contract without adding new backend routes, API-key
management, account-settings mutation, org-scoped audit copies, or broad UI
redesign.

# PR Boundary

behavior, persistence, auth runtime semantics, CORS policy, or infrastructure

# Required Outcomes

1. Add a web-console account-security navigation surface.
   - Introduce a clear signed-in account-level entry point such as
     `Account Security` or `Security Events`.
   - Keep organization/project context visible where appropriate, but make it
     clear this view is user-scoped rather than org-scoped.
   - Do not hide the existing workspace flows or turn the app into a marketing
     or settings shell.

2. Add a small API client method for the existing route.
   - Call `GET /v1/auth/security-events` through the existing public `api-edge`
     target selected by the console.
   - Preserve bearer-token handling through the `Authorization` header only.
   - Preserve standard success/error envelope handling and request ID display.
   - Support V1 pagination parameters `limit` and `cursor`.

3. Render the user’s security-event history.
   - Show core fields already present in the public contract, including event
     type, outcome, occurred-at timestamp, and any safe request context returned
     by the API (for example request ID, correlation ID, IP, user agent).
   - Render empty, loading, and error states cleanly.
   - Keep the presentation compact and operational rather than decorative.
   - Do not invent fields absent from the contract.

4. Support pagination in the UI.
   - Use the existing cursor-based contract from Task 0045.
   - Provide a simple, understandable pagination action such as `Load more`.
   - Preserve already-loaded items when fetching the next page.
   - Handle invalid or failed follow-up loads with visible semantic error
     feedback instead of silent failure.

5. Respect secret-safety and contract boundaries.
   - Do not expose raw codes, bearer tokens, token hashes, API keys, or any
     credential material in the UI.
   - Treat returned metadata as already-redacted contract data; do not attempt
     to reconstruct hidden values.
   - Keep the console as a public API client only. Do not call internal Workers
     or service-binding-only routes.

6. Add focused frontend verification.
   - Add or update the minimal test coverage that is justified by the current
     frontend structure, if practical.
   - At minimum, verify manually in browser or local preview that the new view
     works for a signed-in user on stage and behaves safely for prod.
   - If the current web-console has no automated test harness, do not balloon
     this PR into a framework adoption task. Keep the verification proportional.

7. Prevent scope drift and recurring ai/ carryover mistakes.
   - Do not stage unrelated `ai/tasks/task-0042-verifier.md`,
     `ai/tasks/task-0043*.md`, `ai/tasks/task-0044*.md`,
     `ai/tasks/task-roadmap-2026-05-28.md`, or old report carryover files unless
     they are explicitly required by this task and explained in the report.
   - Keep the branch limited to Task 0046 changes plus the required implementer
     report.
