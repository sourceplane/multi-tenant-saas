# Objective

Add the first Supabase Postgres migration harness and migration ownership
conventions for the reusable SaaS starter.

This task establishes how future bounded contexts add and verify SQL migrations.
It must not apply schema changes to live Supabase databases and must not
implement identity, membership, project, API, or Hyperdrive runtime behavior.

# Required Outcomes

- Create a `packages/db` workspace package with:
  - TypeScript build/typecheck/lint scripts matching repo patterns.
  - A `component.yaml` using the existing `turbo-package` composition.
  - A public export that future packages can use to inspect migration metadata
    without importing Node-only runtime code into Workers.
  - A clear internal boundary between Node/CLI-only migration tooling and
    Worker-safe shared types.
- Define a deterministic migration layout for future bounded contexts, for
  example a manifest plus SQL files grouped by owner/schema. The exact layout is
  the implementer's choice, but it must make ownership, ordering, and checksum
  or content validation explicit.
- Add a migration verifier/test suite that runs without live database
  credentials. It should at minimum prove:
  - migration IDs are unique and sorted deterministically;
  - each migration declares an owning bounded context or schema;
  - migration files are included in the manifest;
  - checksums or equivalent content guards catch accidental drift;
  - any tenant/project-scoped migration examples preserve the
    `org_id + project_id` invariant from `specs/domain-model.md`.
- Add a first migration convention artifact that is useful but safe. Prefer a
  baseline metadata/control migration or explicit example fixture over creating
  broad domain tables in this PR. If the implementer believes real identity,
  membership, or projects tables should be introduced now, write a spec proposal
  instead of expanding scope.
- Add a first-class Orun test component for the DB migration verifier if the
  existing composition model supports it. The new `packages/db` component should
  have a dependency edge to the matching test component when that is possible
  under the local Stack Tectonic contract.
- Refresh the root `README.md` only where it is stale because of the current
  infrastructure and new migration package. Keep the README factual and short.
- Write `ai/reports/task-0007-implementer.md`.
