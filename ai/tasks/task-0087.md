# Objective

Wire `identity-worker` magic-link login to send through
`notifications-worker` over an internal service binding, so that
`POST /v1/auth/login/start` produces a real notification lifecycle row
(and a real `local-debug` provider attempt on stage/prod) instead of
silently dropping the generated code. This closes the most visible gap
left by Task 0086 and unblocks roadmap B1 (real auth) end-to-end: when
the real provider is later swapped behind `NOTIFICATIONS_PROVIDER`,
magic-link email begins delivering to inboxes with zero call-site
changes.

After this PR, calling `POST /v1/auth/login/start` on stage/prod:

- still returns the existing `LoginStartResponse` shape (no contract
  change for the console / browser caller)
- additionally enqueues one notification through `NOTIFICATIONS_WORKER`
  with `category=transactional`, `templateKey="auth.magic_link"`,
  `templateData` containing only the `code` and a short bounded set of
  presentation fields, and `recipient.channel="email"` keyed to the
  login email
- continues to expose the `code` in the response **only** when
  `DEBUG_DELIVERY === "true"` (local_debug mode, unchanged behavior)
- treats notifications-worker errors as best-effort for the login path
  (login itself must not 5xx because notifications-worker is down), but
  logs/records the failure so the existing audit seam stays honest

# Constraints

   throw, or missing binding MUST NOT propagate to the login response.
   The login endpoint stays 2xx on a valid challenge regardless of
   notifications health.
2. `templateData` MUST NOT contain the email address or any
   user-provided string beyond what is already in the existing login
   flow. Allowed fields: `code` (string), `emailHint` (string),
   `expiresAt` (ISO string), `requestId` (string).
3. The notifications enqueue call MUST set
   `category="transactional"`, `templateKey="auth.magic_link"`,
   `recipient.channel="email"`, and pass the login email as the
   recipient address.
4. Internal request headers MUST match notifications-worker's
   `InternalActor` expectations (actor type/id + request id) as already
   used by other internal callers. Do not invent a new header scheme.

   the same as `events-client.ts`.
