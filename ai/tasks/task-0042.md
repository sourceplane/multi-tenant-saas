# Task 0042

## Role

Act as Implementer.

## Objective

Make Cloudflare custom domains a first-class, intent-driven platform capability.
The repo must be able to set the active SaaS domain from `intent.yaml`, manage
or adopt the matching Cloudflare zone through an Orun component, and publish the
environment-specific web consoles on custom domains:

- Stage console: `https://stage.sourceplane.ai/`
- Prod console: `https://prod.sourceplane.ai/`

`sourceplane.ai` is already managed in the current Cloudflare account. The
implementation must also leave a clear path for another context to replace that
base domain and let this repo create/manage the zone when it is not already
present in Cloudflare.

## Required Outcomes

1. **Intent-owned domain configuration**
   - Add the canonical Cloudflare domain settings to the main `intent.yaml`.
   - For this repo/context, configure:
     - base/root domain: `sourceplane.ai`
     - stage console hostname: `stage.sourceplane.ai`
     - prod console hostname: `prod.sourceplane.ai`
   - Avoid making `sourceplane.ai`, `stage.sourceplane.ai`, or
     `prod.sourceplane.ai` values exist only as hardcoded app/runtime literals.
     Component manifests or generated config may repeat resolved values only
     when the source of truth remains `intent.yaml` and the behavior is tested.
   - Keep `dev` free of live custom-domain provisioning unless you can prove a
     safe no-op/verify-only behavior.

2. **New Cloudflare domain component type**
   - Add a new Stack Tectonic composition/component type for Cloudflare domain
     management, for example `cloudflare-domain`.
   - The type must be Orun-native: schema, job template(s), profiles, smoke
     fixture/docs if that is the local pattern, and validation through the
     committed composition source.
   - The type should be Terraform-backed unless you find a materially better
     Cloudflare API approach. If you use API calls directly, keep them typed,
     idempotent, and isolated in the composition rather than in app components.
   - It must support both modes:
     - existing-zone/adopt mode for `sourceplane.ai` so this task does not try
       to create a duplicate zone;
     - managed-zone/create mode for another context where the domain is not
       already added to Cloudflare.
   - If Terraform cannot safely express "lookup existing, else create" in one
     graph without brittle provider failures, expose an explicit parameter such
     as `manageZone` or `zoneMode` and document the operational tradeoff.
   - The component must report non-secret outputs/status only: zone name, zone
     ID, hostname names, target project/worker names, and activation/status
     fields. Never log Cloudflare tokens.

3. **Concrete Sourceplane domain component**
   - Add the first consuming component under the infra tree, named clearly
     (for example `infra/terraform/cloudflare-domain/component.yaml` or
     `infra/cloudflare-domain/component.yaml`, following the pattern you choose).
   - It must subscribe to `stage` and `prod`.
   - PR/pull-request runs must plan/validate only. `github-push-main` must be
     the only path that applies Cloudflare domain changes.
   - It must attach/manage the Pages custom domains:
     - `stage.sourceplane.ai` -> `sourceplane-web-console-stage`
     - `prod.sourceplane.ai` -> `sourceplane-web-console-prod`
   - Add the dependency edge(s) needed so the domain attachment does not race
     the Pages project creation path.

4. **Custom-domain support for Cloudflare app components**
   - Extend the Cloudflare component contracts so custom domains are supported
     consistently for Pages and Worker components.
   - Pages support must cover the web-console use case above.
   - Worker support must exist as a typed optional contract for public Workers
     such as `api-edge`, but do not broaden exposure for bounded-context
     Workers.
   - `identity-worker`, `membership-worker`, `policy-worker`,
     `projects-worker`, and `events-worker` must not receive custom domains
     unless a component explicitly opts in. Preserve existing service-binding
     boundaries and existing `workers_dev` behavior unless a separate security
     task changes it.
   - If the cleanest design is that the new `cloudflare-domain` component owns
     all hostname bindings while Pages/Worker components only declare metadata,
     document that contract and make validation catch mismatches.

5. **CORS and console verification path**
   - Update `api-edge` CORS so:
     - stage API allows `https://stage.sourceplane.ai`;
     - prod API allows `https://prod.sourceplane.ai`;
     - cross-environment custom-domain origins are denied;
     - existing stage/prod Pages URLs and preview URL allowances remain during
       this transition unless you intentionally remove them and update the smoke
       story.
   - Update tests for the custom-domain CORS matrix.
   - Update web-console smoke checks so the custom domains are verified by the
     domain component after merge. Keep Pages-project smoke useful for first
     deployment/project creation.

6. **Specs and docs**
   - Update `specs/components/12-web-console.md` to name the custom console
     hostnames and describe the relationship to the environment-specific Pages
     projects.
   - Update `specs/components/01-edge-api.md` to include the custom-domain CORS
     matrix.
   - Update infra/domain docs near the new component explaining:
     - how to change the base domain in another context;
     - how to use existing-zone/adopt mode versus managed-zone/create mode;
     - what must be verified after a new domain is delegated to Cloudflare.

## Constraints

- Do not delete the legacy `sourceplane-web-console.pages.dev` project or the
  new environment-specific `*.pages.dev` URLs.
- Do not move bounded-context Workers onto public custom domains.
- Do not add `dev` Supabase, database, or live domain provisioning.
- Do not change unrelated product behavior or read/apply `specs-v2/**`.
- Do not hardcode Cloudflare account IDs or secrets in source.
- Do not rely on the Cloudflare dashboard as the source of truth. All live
  Cloudflare mutations in scope must be done by Orun-controlled CI jobs.
- Verify current Cloudflare API/Terraform provider resource names before
  implementing. The docs links above are orientation, not permission to guess.
