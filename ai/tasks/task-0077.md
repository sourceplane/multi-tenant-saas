# Task 0077

Agent: Implementer

## Current Repo Context

- Tasks 0075 and 0076 completed the billing foundation and first read API surface:
  `billing.{plans,billing_customers,subscriptions,invoices,entitlements}` is live
  on stage/prod, `@saas/db/billing` and `@saas/contracts/billing` are available,
  and `apps/billing-worker` exposes five org-scoped read routes through `api-edge`.
- The public read routes are:
  - `GET /v1/organizations/{orgId}/billing/plans`
  - `GET /v1/organizations/{orgId}/billing/customer`
  - `GET /v1/organizations/{orgId}/billing/summary`
  - `GET /v1/organizations/{orgId}/billing/invoices`
  - `GET /v1/organizations/{orgId}/billing/entitlements`
- `apps/web-console` currently has workspace tabs for Members, Invitations,
  Projects, API Keys, Config, and Audit, but no Billing tab. The console is a
  replaceable client of public `api-edge` contracts and must not call internal
  Workers directly.

## Objective

Add a read-only Billing tab to the web console that consumes the five public
org-scoped billing read routes shipped by Task 0076. This satisfies the Product
Overview UX baseline (`review usage and billing`) and Billing spec acceptance
that plan state and entitlements are queryable from the public API.

## PR Boundary

1. Extend `apps/web-console/src/api.ts` with typed billing read client methods
   using `@saas/contracts/billing` public types.
2. Add a `Billing` workspace/sidebar tab in `apps/web-console/src/main.ts` that
   renders provider-neutral read-only billing data for the selected organization.
3. Add scoped CSS in `apps/web-console/src/style.css` for the Billing tab using
   the existing calm/editorial design language from Task 0073.
4. Add focused validation/tests if the repo already has an appropriate web-console
   test seam; otherwise run and report web-console typecheck/build plus Orun plan
   validation.

## Required Outcomes

- [ ] `apps/web-console/src/api.ts` imports the public billing contract types and
      exposes methods for all five Task 0076 read routes.
- [ ] Web-console navigation includes a `Billing` tab for authenticated org users.
- [ ] The Billing tab loads and displays, at minimum:
      - billing summary (active subscription/plan/customer state where present),
      - available plans,
      - customer profile/status if one exists,
      - entitlements,
      - invoices with safe status/amount/date rendering and pagination/load-more
        if the route returns a cursor.
- [ ] Empty states are explicit and user-readable when customer, subscription,
      entitlements, invoices, or plans are absent.
- [ ] All rendered provider fields are treated as opaque display references only;
      no secret-bearing values, raw provider payloads, SQL, stack traces, or
      bearer tokens are exposed.
- [ ] The UI uses text-node-safe rendering patterns consistent with Config and
      Audit tabs, not `innerHTML` for API data.
- [ ] Existing workspace behavior is preserved: auth, org switcher, project
      selection, Account/Profile/Security, API Keys, Config, Audit, and Projects.
- [ ] `apps/web-console` typecheck/build pass and Orun changed-plan/dry-run are
      recorded in the implementer report.

## Non-Goals

- No billing mutations: no checkout, portal session creation, subscription
  change/cancel, customer creation, entitlement editing, invoice payment, or
  provider write path.
- No Stripe/payment-provider SDK, webhook handling, provider credentials, Secrets
  Store, Queue/KV/Durable Object/Analytics Engine/Terraform, or Wrangler binding
  changes.
- No database schema, repository, billing-worker, api-edge, policy-worker, or
  metering-worker changes unless a compile error reveals a narrow type export bug
  directly blocking the UI.
- No `specs-v2/**` product-specific Git catalog work.
- No fake billing data; use empty states if the API returns empty arrays/nulls.
- No framework migration, React/Tailwind/shadcn introduction, design-system rewrite,
  or unrelated visual refresh.

## Constraints

1. The web console must call only public `api-edge` routes through `ApiClient`;
   it must not call `billing-worker` or any internal Worker directly.
2. Billing remains provider-neutral. Provider ids/URLs may be displayed only as
   opaque references already present in `@saas/contracts/billing` public types.
3. Secret safety is mandatory: never render plaintext credentials, ciphertext,
   API keys, webhook secrets, raw bearer tokens, SQL, stack traces, or raw
   provider payloads. Do not log response bodies to the console.
4. The UI must be read-only. All button labels/actions must make clear that no
   billing state can be changed in this task.
5. Preserve Task 0073 CSS/design direction and existing DOM helper style.
6. Keep this PR web-console scoped unless a narrow shared-contract import/export
   issue blocks compilation.
7. Do not modify active orchestration state files except for the implementer
   report required by this task.
