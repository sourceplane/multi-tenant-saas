# Required Outcomes

1. Add atomic organization bootstrap event/audit writes.
   - Newly successful `POST /v1/organizations` requests must append:
     - `organization.created`
     - `membership.added` for the creator's initial owner membership
   - The organization, owner member, owner role assignment, both events, and
     both audit entries must commit or roll back together in the live
     database-backed path.
   - If either event/audit append fails, the organization bootstrap must not be
     reported as successful and must not leave partial membership state.
   - Keep the existing public success response shape unchanged.

2. Use canonical internal IDs in event/audit persistence fields.
   - Store raw organization UUIDs in `event.orgId` and resulting audit `org_id`.
   - Use the raw organization UUID as the `organization.created` subject ID.
   - Use the raw membership UUID as the `membership.added` subject ID.
   - Do not store `org_` or `mem_` public IDs in canonical persistence columns.
   - Public IDs may appear in safe payload fields and descriptions.

3. Keep payloads useful and safe.
   - `organization.created` payload should include safe public organization
     context such as public org ID, name, and slug.
   - `membership.added` payload should include safe public membership context
     such as public org ID, public member ID, subject type/id, and assigned
     role.
   - Do not include bearer tokens, raw session tokens, raw invitation tokens,
     token hashes, API keys, SQL, connection details, stack traces, or secrets
     in events, audit payloads, responses, or logs.

4. Preserve public audit response behavior.
   - `GET /v1/organizations/{orgId}/audit` must expose public IDs for the new
     audit rows:
     - `org_` for organization scope and organization subject IDs;
     - `mem_` for membership subject IDs.
   - No raw known internal organization or member UUID should leak through the
     public audit response fields for these new rows.
   - Existing payload redaction behavior must continue to work.

5. Add focused tests.
   - Update or add membership-worker tests proving organization bootstrap
     appends exactly the expected `organization.created` and
     `membership.added` event/audit entries with raw canonical IDs and safe
     public payloads.
   - Add a regression test proving event/audit failure does not return a
     successful organization creation response. Prefer a test seam that also
     demonstrates rollback/atomic intent for the live transactional path.
   - Add or update events-worker tests proving public audit responses map the
     new organization and member subject IDs to public IDs.
   - Keep existing organization create/list/read behavior, slug validation, and
     invitation/member mutation tests intact.
