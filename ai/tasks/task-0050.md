## Objective

Reconcile and lock the V1 public API-key administration contract in the active `specs/**` pack so the next runtime task can implement against one coherent, bounded API surface.

This PR is a spec-alignment task, not a runtime implementation task. It should resolve the route-shape conflict, document the ownership/orchestration model across identity and membership, and define the minimal V1 request/response semantics for create/list/revoke without adding live routes or modifying application code.

## Accepted Direction For This Task

The Orchestrator is accepting the following direction for V1 public API-key administration:

1. Public API-key administration is tenant-scoped, not account-scoped.
2. The canonical V1 route family is:
   - `GET /v1/organizations/{orgId}/api-keys`
   - `POST /v1/organizations/{orgId}/api-keys`
   - `DELETE /v1/organizations/{orgId}/api-keys/{apiKeyId}`
3. `/v1/auth/api-keys` is stale and should be removed or rewritten from the active spec pack.
4. Project scope remains an explicit narrowing attribute of the backing service principal / role binding model, not a required path segment for the initial public admin surface.
5. V1 does not require a separate public service-principal CRUD surface before API-key admin. The later runtime implementation may orchestrate service-principal creation and initial membership binding behind the create-API-key route.
6. API-key create/revoke remain subject to identity-owned security events and org-scoped audit/event copies when organization context exists.

If implementation inspection reveals a hidden conflict with this accepted direction, document it in `ai/proposals/task-0050-spec-update.md` and keep the spec edits conservative.

## Required Outcomes

- [ ] Remove the public-route contradiction from the active spec pack.
  - `specs/components/02-identity.md` must stop presenting `/v1/auth/api-keys` as the recommended public admin surface.
  - `specs/components/01-edge-api.md` and `specs/contracts/api-guidelines.md` must align with the accepted tenant-scoped V1 route family under `/v1/organizations/{orgId}/api-keys`.
  - The resulting spec text must clearly distinguish org-scoped API-key administration from optional future project-scoped narrowing.

- [ ] Document bounded-context ownership clearly.
  - Identity owns API keys, service principals, and identity security events.
  - Membership owns service-principal role bindings / role assignments.
  - Policy owns authorization decisions for create/list/revoke administration.
  - The future runtime path should read as orchestration across these seams, not identity re-owning role assignments.

- [ ] Define the minimal V1 API-key administration semantics needed for the follow-on runtime task.
  - List what create/list/revoke do at a contract level.
  - Clarify that create returns the raw API key secret exactly once while persistence remains hash-only.
  - Clarify that list never returns raw key material.
  - Clarify the expected metadata at minimum: key ID, label, prefix, createdAt, expiresAt, lastUsedAt, revokedAt, and backing service-principal summary / scope summary as appropriate.
  - Clarify whether create may provision the backing service principal and initial binding as part of the workflow.

- [ ] Clarify V1 scope rules.
  - Org context is always explicit in the path.
  - If project-scoped keys are mentioned, they must remain within an explicit organization/project pair and must not imply a separate required project-scoped public route for V1.
  - Avoid tenant guessing or project-only addressing.

- [ ] Clarify mutation-side security and audit expectations.
  - API-key create/revoke require explicit acting subject and deny-by-default authorization.
  - API-key create/revoke must emit identity-owned security events.
  - When org context exists, create/revoke must also emit normal org-scoped audit/event copies using the shared envelope.
  - POST create should honor the repo’s idempotency guidance from `specs/contracts/api-guidelines.md`.

- [ ] Keep the PR spec-only.
  - Do not add or modify live runtime code.
  - Record any newly discovered runtime dependency or unresolved edge case in the implementer report and, if necessary, in `ai/proposals/task-0050-spec-update.md`.
