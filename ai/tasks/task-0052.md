## Objective

Add the first web-console API-key management surface so an authenticated organization user can create, list, and revoke API keys through the already-live public API.

This task should complete the first user-facing follow-through on Task 0051 without reopening backend runtime work. Keep the PR as a single coherent UI/client slice: one org-scoped API-key management surface in `apps/web-console`, backed only by the public `api-edge` routes.

## Required Outcomes

- [ ] Add the minimal public API-key contract/types required by the UI.
  - If the console can stay cleanly typed without new shared contract files, keep contract changes minimal.
  - If the console needs shared shapes, add the smallest public contract surface necessary for:
    - API-key list items
    - create response shape including one-time `secret`
    - revoke response shape
    - service-principal metadata displayed by the UI
  - Export any new contract types through `packages/contracts/src/index.ts`.

- [ ] Extend the web-console API client for the live public route family.
  - Add list/create/revoke methods using only:
    - `GET /v1/organizations/{orgId}/api-keys`
    - `POST /v1/organizations/{orgId}/api-keys`
    - `DELETE /v1/organizations/{orgId}/api-keys/{apiKeyId}`
  - Preserve existing bearer-token handling through the `Authorization` header only.
  - Preserve standard success/error envelope handling and request ID propagation.
  - Support existing list query semantics where useful: `limit`, `cursor`, and optional `projectId` narrowing.

- [ ] Add an org-scoped API Keys workspace surface.
  - Introduce an `API Keys` tab or equivalent org-scoped entry point inside the selected-organization workspace.
  - Keep organization context visible.
  - Make it clear this surface is organization-scoped API-key administration, not the existing user-scoped `Account Security` view.
  - Do not hide or break existing `Members`, `Invitations`, `Projects`, `Audit`, or `Account Security` flows.

- [ ] Render API-key list data safely.
  - Show at minimum the public fields already returned by the live backend:
    - `id`, `label`, `prefix`, `createdAt`, `expiresAt`, `lastUsedAt`, `revokedAt`
    - `servicePrincipal.id`, `servicePrincipal.displayName`, `servicePrincipal.role`, `servicePrincipal.projectId`
  - Handle loading, empty, success, and error states clearly.
  - Support cursor pagination with a simple operator-friendly control such as `Load More`.
  - Optional `projectId` filtering may be exposed as a narrow filter control, but must remain a filter on the org route rather than inventing a project-path route.

- [ ] Add a safe create flow.
  - Allow the operator to enter the live backend inputs already supported by the public API:
    - `label`
    - `role`
    - optional `projectId`
    - optional `expiresAt`
  - Make the project-scope rule understandable: project-scoped roles require `projectId`.
  - On success, display the one-time raw secret clearly and prominently.
  - The one-time raw secret must appear only in the immediate create-success path.
  - Provide a simple copy affordance if straightforward, but do not add heavy dependencies just for clipboard UX.
  - After creation, refresh or append to the API-key list without exposing the raw secret in list state.

- [ ] Add a safe revoke flow.
  - Allow revoke from the list UI with an explicit confirmation step.
  - After revoke, refresh the list or update the row state so the revoked state is visible.
  - Do not widen this task into service-principal lifecycle management beyond the existing API-key revoke route.

- [ ] Preserve secret-safety and boundary rules.
  - Do not store raw API-key secrets in `localStorage`, `sessionStorage`, URL params, screenshots, or reports.
  - Do not expose raw secrets in list rows, reload paths, or subsequent fetches.
  - Do not call internal Workers or service-binding-only routes from the console.
  - Keep the console a pure public `api-edge` client.

- [ ] Add proportional verification.
  - If the current web-console still has no dedicated automated test harness, do not balloon this PR into framework adoption.
  - Use the existing package verification commands (`typecheck`, `lint`, `build`) and proportional manual/browser checks.
  - Record clearly that the current frontend slice uses manual verification if no automated harness exists for this surface.

- [ ] Prevent recurring carryover mistakes.
  - Do not stage unrelated historical untracked `ai/tasks/task-0042-verifier.md`, `task-0043*.md`, `task-0044*.md`, `task-0046*.md`, `task-0047*.md`, `task-0048*.md`, `task-0049*.md`, `task-0050*.md`, `task-0051*.md`, old reports, or the roadmap file unless this task explicitly updates them and the report explains why.
