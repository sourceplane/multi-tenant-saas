## Objective

Create the first deployable Identity Worker runtime slice in
`apps/identity-worker`. It must expose the initial passwordless auth route
surface on the Identity Worker itself:

- `POST /v1/auth/login/start`
- `POST /v1/auth/login/complete`
- `GET /v1/auth/session`
- `POST /v1/auth/logout`
- `GET /health`

Use the Task 0012 repository adapters for persistence. This PR intentionally
does **not** wire `apps/api-edge` to the Identity Worker yet. The follow-up task
can add the `api-edge` service-binding facade after this PR creates the
service-binding target and proves deployment. Keep the public response contract
compatible with the future edge forwarding path.

- Add an explicit non-secret debug-delivery flag. Stage may enable debug
  delivery so the verifier can exercise a live end-to-end flow. Prod must have
  debug delivery disabled, and tests must prove prod responses never include a
  raw login code.

  `cloudflare-worker-turbo` component with Node 22 and pnpm 10.12.1. It should

- Add a `/health` response that reports service name, environment, timestamp,

### Auth behavior

- Implement passwordless email-code login using the `@saas/db/identity`
  repository and the Worker-safe Hyperdrive SQL executor.
- Normalize email addresses consistently. Store and look up `email_lower`.
- `login/start`:
  - Validate JSON and email input.
  - Find or create the user and `email` auth identity. Treat conflicts by
    re-reading the canonical row when safe.
  - Generate a one-time code with Web Crypto, not `Math.random()`.
  - Hash the code before passing it to the repository. Do not store or return
    the hash.
  - Create a short-lived login challenge, defaulting to about five minutes.
  - Return `challengeId`, `expiresAt`, and a delivery object. Include the raw
    code only when debug delivery is enabled for that environment.
- `login/complete`:
  - Accept public `challengeId` plus code.
  - Resolve the challenge first so repository `not_found`, `expired`, and
    `already_consumed` cases map to useful API errors.
  - Verify by calling `consumeLoginChallenge(id, codeHash, consumedAt)` so code
    checking remains atomic.
  - Create a high-entropy bearer session token with Web Crypto.
  - Store only a token hash through the repository.
  - Return the raw token exactly once in the success response.
- `session`:
  - Resolve `Authorization: Bearer <token>`.
  - Return actor/session/user data for valid, unexpired, unrevoked sessions.
  - Return `unauthenticated` for missing, malformed, expired, revoked, or
    unknown tokens.
- `logout`:
  - Resolve the current bearer token, revoke the session through the repository,
    and ensure the same token no longer resolves.
- Keep session/user IDs opaque in the API. The database stores UUIDs; the
  Worker should own the public prefix mapping such as `usr_`, `ses_`, and
  `chl_`. Do not expose raw repository hash fields.

### HTTP contract

- Use `/v1` route prefixes and the standard success envelope:

```json
{
  "data": {},
  "meta": {
    "requestId": "req_...",
    "cursor": null
  }
}
```

- Use the standard error envelope:

```json
{
  "error": {
    "code": "validation_failed",
    "message": "Human-readable safe message.",
    "details": {},
    "requestId": "req_..."
  }
}
```

- Use only the semantic error codes from `specs/contracts/api-guidelines.md`.
- Accept or generate request IDs. Preserve an incoming request ID header if it
  is present and valid.
- Return JSON with appropriate status codes.
- Do not log or report raw login codes, bearer tokens, token hashes, code
  hashes, database credentials, connection strings, or provider secrets.

### Contracts and tests

- Put reusable auth/session request and response types in `packages/contracts`
  instead of hiding public contract shapes inside the Worker.
- Add tests that do not require a live database. Use fake repositories or
  injected executors for domain/handler behavior.
- Cover at minimum:
  - start-login success with debug delivery enabled
  - start-login with debug delivery disabled, especially prod behavior
  - invalid email/body validation
  - complete-login success and one-time challenge consumption
  - wrong code, consumed challenge, expired challenge, and unknown challenge
  - session resolution for valid, expired, revoked, malformed, and missing
    tokens
  - logout revokes a session and prevents reuse
  - raw codes/tokens are never passed to persistence, except token/code hashes
  - API envelopes and request IDs
- Add an Orun test component for `tests/identity-worker` if you add a new test
  package. Do not add an unsound cross-environment `dependsOn`; if you do add a
  dependency edge, prove it in the rendered Orun DAG.
