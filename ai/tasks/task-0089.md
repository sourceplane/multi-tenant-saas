## Objective

Wire `accept-invitation` to enqueue an `invitation.accepted` notification
through `notifications-worker`, AND extract the duplicated
`notifications-client.ts` into a shared `@saas/notifications-client`
workspace package consumed by all three callers (identity-worker,
membership-worker create-invitation, membership-worker accept-invitation).

This is one coherent PR because the third caller is the contractual
trigger to extract the package â€” extracting earlier would have produced
a 2-consumer abstraction identical to the duplication; extracting now
consolidates three consumers in one move.

## Constraints

1. Enqueue MUST be placed strictly outside `executor.transaction()`
   (post-commit). The transaction must NOT depend on enqueue success.
2. Enqueue MUST be best-effort: any failure path
   (`no_binding | non_2xx | network_error | bad_response`) returns
   the normal `200` acceptance response.
3. No raw invitation token (`rawToken`) may flow to `templateData`.
   Only the existing `DEBUG_DELIVERY === "true"` short-circuit path
   may carry the raw token in the response body, and that path is
   outside notifications scope.
4. Shared package exports are exactly the four identifiers consumed
   today: `enqueueNotification`, `NotificationsEnvBinding`,
   `NotificationsClientContext`, `EnqueueNotificationResult`.
   No public API expansion.
5. Workspace package follows the `@saas/shared` layout
   (`package.json` with `exports`, `tsconfig.build.json`, no runtime
   deps beyond `@saas/contracts`).
6. Do not touch `kiox.lock`, `pnpm-lock.yaml` lockfile entries
   beyond what `pnpm install` produces for the new workspace package
   (i.e. just add the new workspace edge).
7. Branch reuse / creation: create a fresh branch
   `impl/task-0089-shared-notifications-client` from `origin/main`.
