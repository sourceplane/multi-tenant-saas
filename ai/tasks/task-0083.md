# Task 0083 — Custom-domain cutover: `{stage,prod}.sourceplane.ai` → web-console-next Workers; decommission legacy `apps/web-console`

Agent: Implementer

## Current Repo Context

- `main` is at `50fd095` (PR #128) and CI is green
  (last run `26629552285`, push to main).
- `apps/web-console-next` is the live console. It deploys via the new
  `cloudflare-workers-assets-turbo` Orun composition (introduced in PR #126 /
  Task 0082.2) as per-env Workers + Static Assets:
  - `sourceplane-web-console-next-stage.rahulvarghesepullely.workers.dev` → 200,
    SSR `<title>Sourceplane Console</title>`, root redirects `/` → `/orgs`.
  - `sourceplane-web-console-next-prod.rahulvarghesepullely.workers.dev` → 200,
    same shape.
  - Dev: no `deploy` profile rule yet (verify-only on PRs); not in scope here.
- `apps/web-console` (vanilla Vite, `cloudflare-pages-turbo`) remains
  byte-identical to the pre-Task-0082 baseline and is still publicly served at
  `sourceplane-web-console-{stage,prod}.pages.dev`. **Custom domains
  `stage.sourceplane.ai` and `prod.sourceplane.ai` are currently bound to
  those OLD Pages projects** through
  `infra/terraform/cloudflare-domain/terraform/main.tf` →
  `cloudflare_pages_domain.console` (project name
  `${pagesProjectPrefix}-${environment}` =
  `sourceplane-web-console-{stage,prod}`).
- `apps/api-edge` CORS allow-list already includes the new
  `sourceplane-web-console-next-*.rahulvarghesepullely.workers.dev` origins
  (PR #128 / Task 0082.2.2), so once the apex hostnames swing to the new
  Workers, cross-origin calls will continue to work via both the
  `*.workers.dev` shadow hostnames and the new custom-domain hostnames.

## Objective

Swing `stage.sourceplane.ai` and `prod.sourceplane.ai` from the legacy
`sourceplane-web-console-{env}` Pages projects to the new
`sourceplane-web-console-next-{env}` Workers, then delete `apps/web-console`
and the now-orphaned `cloudflare_pages_domain.console` resource. The legacy
Pages projects themselves are **NOT deleted** by this task (Orun does not
manage Pages project lifecycle on teardown; manual cleanup is a deliberate
soak-period follow-up).

## PR Boundary (single PR, single rollback unit)

1. **Terraform: replace Pages domain attachment with Worker custom domain.**
   In `infra/terraform/cloudflare-domain/terraform/main.tf`:
   - Add a new variable `workerNamePrefix` (default `sourceplane-web-console-next`)
     and a local `worker_name = "${var.workerNamePrefix}-${var.environment}"`.
   - **Remove** `resource "cloudflare_pages_domain" "console"` and its
     `pages_domain_status` output.
   - **Add** `resource "cloudflare_workers_custom_domain" "console"` guarded by
     `count = local.has_custom_domain ? 1 : 0`, with:
     - `account_id = var.cloudflare_account_id`
     - `zone_id = local.zone_id`
     - `hostname = local.console_custom_domain`
     - `service = local.worker_name`
     - `environment = "production"` (Workers, not Orun — top-level deploy
       environment of the Worker; we don't use wrangler environments for
       web-console-next, so this is always `"production"` for every Orun env).
   - Replace the `pages_domain_status` output with a `worker_custom_domain_id`
     output reading `try(cloudflare_workers_custom_domain.console[0].id, "not_configured")`.
   - Verify `pagesProjectPrefix` variable is still referenced by
     `pages_project_name` local — keep that local + output for now (read-only,
     used for state-file diffing during the soak); they will be removed in a
     later cleanup task after the legacy Pages projects are manually deleted.
   - Confirm provider pin in `infra/terraform/cloudflare-domain/terraform/versions.tf`
     (or wherever required_providers lives) supports
     `cloudflare_workers_custom_domain`. The cloudflare provider must be
     `~> 4.30` and present in the local provider cache; if the pin needs to
     bump for the resource to be available, bump to `~> 4.52` and call it out
     explicitly in the report.
2. **Terraform plan/apply must show a destroy of the Pages domain attachments
   on stage and prod**, and a create of the Worker custom-domain attachments
   on stage and prod. Cloudflare will atomically swing DNS during apply.
3. **`infra/terraform/cloudflare-domain/component.yaml`:**
   - Change `dependsOn` from `web-console` to `web-console-next`.
   - Leave `parameters.pagesProjectPrefix` in place for one more cycle
     (state-file output only; not consumed by any resource).
4. **Decommission `apps/web-console`:**
   - Delete the directory `apps/web-console/` entirely (component.yaml,
     package.json, src, Vite config, everything).
   - Remove the workspace entry from `pnpm-workspace.yaml` if it lists
     `apps/web-console` explicitly (it may match via `apps/*` — in that
     case no edit needed).
   - Update `turbo.json` if it has a `web-console`-specific pipeline override
     (most likely it doesn't; just confirm).
   - Grep the repo for `sourceplane-web-console` references (NOT
     `sourceplane-web-console-next`) and remove any lingering references in
     docs, CI configs, or component manifests. Specs (`specs/components/12-web-console.md`)
     are updated explicitly in step 5; do not leave stragglers elsewhere.
5. **Spec update — `specs/components/12-web-console.md`:** rewrite the
   "Deployment Model" section (currently lines 127–151) to describe the new
   reality:
   - Per-env Workers + Static Assets via `@opennextjs/cloudflare` and the
     `cloudflare-workers-assets-turbo` Orun composition.
   - Worker names `sourceplane-web-console-next-{env}` on
     `*.rahulvarghesepullely.workers.dev`.
   - Custom domains `{stage,prod}.sourceplane.ai` attached via
     `cloudflare_workers_custom_domain` in `infra/terraform/cloudflare-domain/`.
   - `VITE_DEPLOY_ENV` reference becomes `NEXT_PUBLIC_DEPLOY_ENV`.
   - Remove the line claiming `environmentAwareProjectName` is a
     `cloudflare-pages-turbo` feature; mention it's now provided by
     `cloudflare-workers-assets-turbo` instead (worker-name suffixing).
6. **Spec update — `specs/components/01-edge-api.md`:** if the CORS allow-list
   section still references `sourceplane-web-console-{stage,prod}.pages.dev`,
   replace with the new custom-domain hostnames + the `*.workers.dev`
   fallbacks already present in code (don't invent new allow entries — mirror
   what `apps/api-edge` now permits after PR #128).
7. **Orun golden path doc (`specs/orun-golden-path.md`):** if it enumerates
   compositions, add `cloudflare-workers-assets-turbo` to the list. This was
   flagged as an out-of-scope leftover in the Task 0082.2 implementer report.

## Acceptance Criteria

- `terraform plan` for `infra/terraform/cloudflare-domain` on stage shows
  exactly one destroy (`cloudflare_pages_domain.console`) and one create
  (`cloudflare_workers_custom_domain.console`); same shape for prod; dev
  shows no diff (still `has_custom_domain == false`).
- After `main` merge, post-merge CI apply run shows the cloudflare-domain
  component succeed on stage and prod with the new Worker custom-domain
  resources reaching `active` (or at least `pending` if Cloudflare needs
  TLS-cert provisioning time — `pending` is acceptable for green CI).
- `curl -sfL https://stage.sourceplane.ai/` returns 200 with response body
  containing `Sourceplane Console`.
- `curl -sfL https://prod.sourceplane.ai/` returns 200 with response body
  containing `Sourceplane Console`.
- `apps/web-console` directory no longer exists; `git grep -n
  'sourceplane-web-console[^-]'` (the legacy prefix without the `-next`
  suffix) returns zero hits outside of historical files in
  `ai/reports/` and `ai/tasks/`.
- `apps/web-console-next` component on stage and prod still deploys and
  smoke-passes (i.e. removing the legacy app didn't break workspace install
  / turbo graph).
- `specs/components/12-web-console.md` Deployment Model section reflects
  the new reality with correct project names and resource types.
- Open one PR. Squash-merge. Implementer report committed on the PR branch
  at `ai/reports/task-0083-implementer.md`.
