# Task 0082

Agent: Implementer

## Current Repo Context

- Tenant + billing entitlement spine is complete through Task 0081. The platform side of the SaaS starter is healthy: identity, membership, projects, environments, policy, events/audit, config, metering, and entitlement-gated creation flows for projects, members, and environments are all live in stage/prod.
- The product surface is the weak link. `apps/web-console` is a single 2,414-line vanilla-TypeScript file (`src/main.ts`) using a hand-rolled `h()` DOM helper, `sessionStorage` for "routing", no design system, no URL-driven scope, no Cmd-K, no skeletons, no real empty states. It works end-to-end but is not a credible buyer-facing surface and is rapidly becoming a maintenance ceiling.
- Spec 12 (`specs/components/12-web-console.md`) gives the agent explicit freedom to pick a modern frontend stack that deploys well on Cloudflare and to build a small design system in `packages/ui`. Roadmap input has named Next.js 15 (App Router) on Cloudflare Pages via `@opennextjs/cloudflare` as the target.
- Two Cloudflare Pages projects are already provisioned and Terraform-owned: `sourceplane-web-console-stage` (custom domain `stage.sourceplane.ai`) and `sourceplane-web-console-prod` (custom domain `prod.sourceplane.ai`). Each console build is locked to a single api-edge environment via `VITE_DEPLOY_ENV` today; the new app must honor the same single-environment-lock invariant under whatever env-var name it chooses.

## Objective

Stand up a Next.js 15 (App Router) web console at `apps/web-console-next`, deployed to Cloudflare Pages via `@opennextjs/cloudflare`, that reaches feature parity with the current vanilla console on the core SaaS starter flows and visibly raises the design bar to "Vercel/Linear quality" — real design system, URL-driven scope selector, Cmd-K command palette, generated forms from contracts, designed empty states, designed plan-limit/upgrade UX, and dark-mode-by-default theming. The vanilla console remains live and unchanged until parity is proven; cutover is a follow-up task, not this one.

## Architect Brief

- **Product bar**: Vercel / Linear / Stripe Dashboard. If a screen would not look at home next to those products, iterate before reporting complete.
- **User moment this PR improves**: a developer signing in for the first time should land in a working, polished surface — not a vanilla-TS prototype with raw error codes and `"Loading..."` text.
- **Free to decide without asking back**: design-system library inside the shadcn/Radix/Tailwind v4 family (or comparably modern); palette, type scale, motion, iconography; Cmd-K library; form library; folder layout under `apps/web-console-next/`; component naming; routing conventions inside the documented URL shape; theming token naming; Pages-project naming for the *new* app.
- **Must not decide without a spec proposal**: changing any contract in `packages/contracts`, any worker behavior, any db migration, api-edge routing, the deployment model for the *existing* console, or the URL invariant (`/orgs/:orgSlug/projects/:projectSlug/environments/:envSlug/...`).
- **Failure modes that invalidate this PR even if tests pass**: any raw `precondition_failed` / `policy_denied` / error-code string reaching the user; any UI primitive rendered outside `packages/ui` (or its in-app equivalent); any bypass of `apps/api-edge`; `sessionStorage` used as routing source-of-truth; an unstyled flash between dark/light mode; an empty list view rendered without a designed empty state.
- **Recommended stack** (deviate only with a one-line rationale in the implementer report): Next.js 15 App Router + `@opennextjs/cloudflare`; shadcn/ui + Radix Primitives + Tailwind v4 with CSS-variable tokens; `cmdk` for the palette; `react-hook-form` + `zod` for forms; `lucide-react` for icons. Theming via CSS variables on `:root` and `[data-theme="light"]`.
- **Implementer latitude statement**: the implementer has full latitude on design-system internals, palette, type scale, motion, and component composition inside the Constraints below. Decisions taken under this latitude must be recorded with one-line rationale in the implementer report. Anything outside those dimensions requires a spec proposal before implementation.

## PR Boundary

This is intentionally a **large, coherent PR-sized task** — the user has explicitly asked for a single greenfield Next.js console scaffold that an implementer can take from zero to opinionated v0. Split is at the natural risk boundary: ship the new app alongside the old, do not delete the old, do not change any worker, contract, infra, or Terraform component.

In scope for this PR:

1. New workspace package `apps/web-console-next` (or a name the implementer prefers; document the choice) containing a Next.js 15 App Router project configured for Cloudflare Pages via `@opennextjs/cloudflare`. Must build, typecheck, lint, and run locally via `pnpm dev`.
2. New shared `packages/ui` design-system package (or co-located inside the app if the agent judges a separate package premature — document the decision). Recommended baseline: shadcn/ui + Radix Primitives + Tailwind v4 with CSS-variable design tokens. The agent has full freedom on token names, palette, type scale, motion, and component breadth — but every primitive used in the app must come from this design system, not bespoke per-page styling.
3. App shell:
   - URL-driven routing for org/project/environment scope (`/orgs/:orgSlug/projects/:projectSlug/environments/:envSlug/...`). `sessionStorage` as routing source-of-truth is forbidden in the new app.
   - Persistent top-left scope switcher rendering `[Org] / [Project] / [Env]` with active scope visible on every page (multi-tenant invariant exposed in the UX layer).
   - Cmd-K command palette (`cmdk` library or equivalent) covering at minimum: switch org, switch project, jump to each page, "create invitation", "create API key", logout.
   - Sidebar + topbar layout with collapsible sidebar, dark-mode-by-default theming, light-mode toggle.
4. Auth flows (parity with current console):
   - email-code login start + complete
   - bearer-token import path (kept for prod testing parity; may be hidden behind a small dev-only affordance)
   - session persistence across refresh; logout
5. Org + workspace flows (parity):
   - list orgs, create org, select org, "personal-org auto-create on first login" is OUT OF SCOPE for this task — keep current behavior.
   - members list, invite member, remove member, change role
   - invitations list, revoke invitation
6. Project + environment flows (parity):
   - projects list, create project, archive project
   - environments list under a project, create environment, archive environment
7. API keys, config, audit, billing tabs reach parity with the current console at minimum, with the design quality upgrade applied.
8. **Designed plan-limit / upgrade UX** (this is the highest-leverage UI surface — do not ship raw error codes):
   - When any create flow (project, environment, invitation) returns `412 precondition_failed` from the entitlement seam, the UI must render a designed upgrade prompt that names the limit hit, shows usage vs limit when available in the error envelope, and offers an action (CTA + "talk to sales" fallback). The requestId remains accessible behind a "Details" disclosure for support.
   - Distinguish the four reason codes documented in `ai/context/current.md` (`disabled`, `not_configured`, `malformed_limit`, `limit_reached`) with appropriate copy.
9. Designed empty states for every list (members, invitations, projects, environments, api keys, audit, billing). No `"No X"` + emoji placeholders.
10. Skeleton loading states for every list and detail view. No `"Loading..."` text spinners.
11. **Forms generated or strongly assisted by contracts**: wire `react-hook-form` + Zod (or equivalent) so create-org, create-project, create-environment, create-invitation, create-api-key forms are driven by the shared types in `packages/contracts`. The agent may extract a small form-from-zod helper into `packages/ui`.
12. Cloudflare Pages deploy story:
   - Add the new app to Orun composition so it is discoverable and produces build/typecheck/lint jobs (do NOT wire it to the existing `sourceplane-web-console-stage` / `sourceplane-web-console-prod` Pages projects — that would replace the old console in prod). Either (a) declare two new Pages projects named e.g. `sourceplane-web-console-next-stage` / `-prod` and add them to `infra/terraform/cloudflare-domain` only if the agent is confident, or (b) keep the new app build-only / preview-only in this PR and defer the Pages projects to a follow-up task. Option (b) is the safe default for a single-PR scope; document the choice.
   - The build must be locked to a single api-edge environment at build time (mirror the current `VITE_DEPLOY_ENV` invariant under whatever env-var name the agent picks — e.g. `NEXT_PUBLIC_DEPLOY_ENV`). Cross-env switching may exist only in local dev.
13. `apps/web-console-next/README.md` documents: stack chosen, dev commands, env vars, deploy story, and an explicit "Parity Status" table mapping each page in the old console to its equivalent in the new one with a parity flag (✅ / ⚠️ partial / ❌ missing).
14. `ai/reports/task-0082-implementer.md` with summary, files changed, design decisions taken, screenshots or a screenshot directory listing under `ai/reports/task-0082-shots/` (the agent should capture local screenshots of each major screen for the verifier and human reviewer), and the real PR number.

Out of scope for this PR — do NOT touch:

- The existing `apps/web-console` source. It remains the production console until a follow-up cutover task.
- Any `apps/*-worker`, any contract in `packages/contracts`, any `packages/db` migration, any policy/billing/events behavior.

- Notifications-worker, admin-worker, SDK package, CLI package, SSO/SCIM. Each is a separate roadmap item.

- Removal or deprecation of the bearer-token import path; it remains for prod manual testing.
- "Personal org auto-create on signup" — explicit follow-up task.

- Stripe / billing portal integration — separate roadmap item.

## Constraints

1. The existing `apps/web-console` must remain buildable, deployable, and behaviorally unchanged. Verifier will run its existing typecheck + build to confirm.
2. The new app must not bypass `apps/api-edge`. All API calls go through the public edge facade, exactly as the current console does.
3. The new app must consume types from `packages/contracts`. No duplicated/forked types. If a needed type is missing, document it in the implementer report as a follow-up — do not add it in this PR (contract changes are out of scope).
4. URL is the source of truth for org/project/env scope. `sessionStorage` for navigation state is forbidden in the new app. `localStorage` for auth token persistence is acceptable.
5. The build is locked to a single api-edge environment via a build-time env var (mirror the current `VITE_DEPLOY_ENV` invariant). Runtime cross-env switching exists only in local dev.
6. Every UI primitive must come from `packages/ui` (or the documented in-app design-system directory). No bespoke per-page styling beyond layout composition.
7. Dark mode is the default; light mode must also work. Theming via CSS variables, not hard-coded color literals in components.
8. No new live Cloudflare or Supabase or AWS resources unless the agent explicitly opts in to declaring new Pages projects for the new app — and even then, the existing console's Pages projects must not be repointed.
9. No secret values, full bearer tokens, or full connection strings may appear in the implementer report, screenshots, or committed code. The bearer-token import field in the UI must redact on display.
10. The PR must remain a single coherent "new Next.js console scaffold" change. Do not absorb adjacent worker/contract/infra cleanup, even if tempting.
