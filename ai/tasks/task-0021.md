# Objective

Add the first policy-gated invitation administration API surface:

- `POST /v1/organizations/{orgId}/invitations`
- `GET /v1/organizations/{orgId}/invitations`
- `DELETE /v1/organizations/{orgId}/invitations/{invitationId}`

This PR should let authorized organization owners/admins create, inspect, and
revoke invitation records through the public `api-edge` facade. It must not add
invitation acceptance yet.

# PR Boundary

This PR must not add invitation acceptance, member removal, role update routes,
project routes, audit/event persistence, notification queues, real email
delivery, database migrations, Terraform/resource changes, policy-engine

# Required Outcomes

1. Add public invitation contract types.
   - Extend `@saas/contracts/membership` with task-scoped invitation request and
     response types.
   - Public invitation records must use opaque IDs, for example `inv_...`, not
     raw database UUIDs.
   - Public records may include email, role, status, inviter subject ID,
     `expiresAt`, `createdAt`, `acceptedAt`, and `revokedAt`.
   - Public records must never include `token_hash`, raw database UUIDs, raw
     bearer tokens, or provider/stack traces.
   - Restrict V1 invitation roles to organization roles only:
     `owner`, `admin`, `builder`, `viewer`, `billing_admin`.
     Project roles require project scope and are not valid for this route.

2. Add invitation ID and token helpers.
   - Add `inv_` public ID helpers alongside the existing org/member ID helpers.
   - Generate unguessable invitation acceptance tokens, but store only a
     SHA-256 hash through the existing repository input.
   - Raw invitation tokens must not be logged, persisted, placed in cursors, or
     included in reports.
   - Follow the identity-worker debug-delivery pattern:
     - local/dev/stage may expose the raw invitation token in the create
       response under a clearly named local-debug delivery object;
     - prod must not expose the raw token.
   - Add a membership-worker config flag for this boundary, preferably the same
     `DEBUG_DELIVERY` convention used by identity-worker unless local code
     strongly suggests a better name.
   - Set stage/local/dev debug delivery to true and prod to false in
     `apps/membership-worker/wrangler.jsonc`.

3. Add policy-gated membership-worker routes.
   - `POST /v1/organizations/{orgId}/invitations`
     - Requires actor headers.
     - Parses and validates JSON body.
     - Required body fields: `email`, `role`.
     - Server sets expiration, with a simple fixed default such as 7 days.
     - Validates email format and role allowlist.
     - Authorizes with policy-worker action
       `organization.invitation.create`.
     - Creates the invitation only after authorization succeeds.
     - Returns safe public invitation metadata and debug delivery information
       only when debug delivery is enabled.
   - `GET /v1/organizations/{orgId}/invitations`
     - Requires actor headers.
     - Authorizes with policy-worker action
       `organization.invitation.list`.
     - Uses the Task 0020 `limit`/`cursor` pagination pattern from day one.
     - Returns `{ invitations: [...] }` with `meta.cursor`.
   - `DELETE /v1/organizations/{orgId}/invitations/{invitationId}`
     - Requires actor headers.
     - Parses public `inv_` ID.
     - Authorizes with policy-worker action
       `organization.invitation.revoke`.
     - Revokes the invitation only after authorization succeeds.
     - Returns the revoked public invitation metadata in a success envelope.
   - Invalid public organization or invitation IDs should return `not_found`
     without leaking parse details.
   - Policy denial should fail closed. Match the existing organization/member
     pattern and avoid organization enumeration.
   - Missing policy binding, malformed policy response, policy fetch failure,
     role-list failure, and database failure must return safe envelopes.

4. Add repository support needed for paginated invitation listing.
   - Keep existing repository methods available unless a fully compatible
     signature is simpler.
   - Add a paginated invitation-list method if the current unpaginated
     `listInvitations(orgId)` is insufficient.
   - Use deterministic ordering with `created_at DESC, id DESC`.
   - Use `limit + 1` reads to determine the next cursor.
   - Cursor comparisons must include both timestamp and UUID tie-breaker.
   - No database migration is expected for this task.
   - If an invitation is past `expiresAt` but has not been accepted or revoked,
     public response status may be derived as `expired` without mutating the
     database. Keep this behavior explicit and tested if implemented.

5. Extend api-edge routing.
   - Forward the new invitation routes to `MEMBERSHIP_WORKER` only after
     resolving the bearer session through `IDENTITY_WORKER`.
   - Forward:
     - collection route query strings for list pagination;
     - POST JSON bodies for create;
     - request ID, traceparent, idempotency-key, and content-type as appropriate.
   - Continue not forwarding raw bearer tokens to `membership-worker`.
   - Allow only:
     - `POST` and `GET` on the invitation collection route;
     - `DELETE` on the invitation item route.
   - Return safe `unsupported`/405 envelopes for wrong methods.

6. Add tests.
   - Membership-worker tests must cover:
     - create success;
     - create validation failures for bad JSON, invalid email, missing role, and
       project-scoped or unknown role;
     - debug delivery includes raw token only when enabled;
     - prod delivery response never includes raw token;
     - token hash is passed to the repository and raw token is not;
     - create authorization action and authorization-before-create ordering;
     - list success, pagination params, `meta.cursor`, and action
       `organization.invitation.list`;
     - revoke success and action `organization.invitation.revoke`;
     - invalid org/invitation public IDs;
     - policy denial and binding/fetch/malformed response failures fail closed;
     - database/repository failures return safe envelopes.
   - DB tests must cover any new paginated invitation repository method,
     including parameterized SQL, ordering, limit-plus-one behavior, cursor
     filtering, and empty results.
   - Api-edge tests must cover:
     - route matching for collection and item invitation routes;
     - auth resolution before forwarding;
     - POST body forwarding;
     - DELETE forwarding;
     - list query-string forwarding;
     - method restrictions;
     - bearer-token redaction.
