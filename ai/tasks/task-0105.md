## Objective

Add a new workspace package `packages/webhook-verifier` published as
`@saas/webhook-verifier` that exposes a tiny, dependency-free helper for
**third-party consumers** to verify the HMAC-SHA256 signatures Sourceplane
attaches to outbound webhook deliveries. The signing scheme is already
locked in code; the helper codifies it once so external customers and the
console-side replay tooling don't reinvent it (and don't reach into
`apps/webhooks-worker`). Roadmap leg B5, "Webhooks polish".

## PR Boundary

In scope (one PR):

1. New package `packages/webhook-verifier/` — `package.json` (name
   `@saas/webhook-verifier`, `private: true`, `type: module`, `exports`
   pointing at `./src/index.ts`, `dependencies: {}` — no runtime deps,
   not even `@saas/contracts`), `tsconfig.json` extending
   `@saas/tsconfig`, `tsconfig.build.json`, and a single `src/index.ts`.
2. `src/index.ts` exports:
   - `SIGNATURE_HEADER` (`"X-Webhook-Signature"`),
     `TIMESTAMP_HEADER` (`"X-Webhook-Timestamp"`),
     `WEBHOOK_ID_HEADER` (`"X-Webhook-ID"`) constants.
   - `SIGNATURE_PREFIX` (`"sha256="`).
   - `DEFAULT_TOLERANCE_SECONDS = 300` (5-minute replay window).
   - `verifyWebhookSignature(input)` — async, returns a tagged result
     `{ ok: true } | { ok: false; reason: "missing_signature" |
     "missing_timestamp" | "malformed_timestamp" |
     "timestamp_out_of_tolerance" | "malformed_signature" |
     "signature_mismatch" }`. Inputs: `{ secret, body, headers, now?,
     toleranceSeconds? }`. `headers` accepts both a plain `Record<string,
     string | string[] | undefined>` and `Headers` (case-insensitive
     lookup either way). Comparison is **constant-time** (manual
     `xor`-byte loop after equal-length hex decode — do NOT short-circuit
     on first-mismatch). `now` defaults to `() => new Date()`. Uses
     **WebCrypto only** (`crypto.subtle.importKey` + `sign`); zero
     `node:crypto`, zero `node:*`, zero non-WebCrypto APIs — runs
     verbatim on Workers, Bun, browsers, modern Node.
   - `signWebhookPayload({ secret, body, timestamp })` — also async,
     returns the `sha256=<hex>` string. Useful for fixtures, tests, and
     symmetric debugging. Codifies the same scheme
     `apps/webhooks-worker/src/delivery.ts:45-61` already implements.
3. `src/__tests__/verify.test.ts` — vitest suite ≥18 `it()` blocks
   covering: happy-path verify, header-case-insensitivity, `Headers`
   instance input, `Record<string, string|string[]|undefined>` input,
   each `reason` enum branch (one test per failure mode, exactly), wrong
   secret, body-byte tamper, timestamp-tamper, missing
   `sha256=` prefix, hex-length mismatch, oversized timestamp, custom
   `toleranceSeconds: 0`, custom `now` injection, signature
   produced by `signWebhookPayload` round-trips through
   `verifyWebhookSignature`, byte-identity match against a frozen vector
   captured from `apps/webhooks-worker/src/delivery.ts` (compute live in
   the test using the same `${timestamp}.${body}` message template so
   any future drift in the worker fails this test).
4. `component.yaml` mirroring the canonical `turbo-package` shape from
   `packages/shared/component.yaml` (metadata.name `webhook-verifier`,
   `spec.type: turbo-package`, `spec.domain: starter-shared`, the same
   3-env quick-check subscribe, the same nodeVersion/pnpmVersion/build
   commands, `labels.layer: shared`, `labels.surface: shared`). This is
   non-negotiable — see the workspace-package-component-yaml-audit
   pitfall: any new directory under `packages/` MUST ship a
   `component.yaml` in the same PR or it becomes invisible to Orun
   discovery and CI lane subscription.
5. `README.md` (≤1.5 KB) — single example showing how an external
   integrator verifies a Sourceplane webhook: imports the helper,
   reads the secret from their env, passes the raw request body string
   and headers, branches on `result.ok`. One sentence on tolerance
   tuning. No marketing copy.
6. Root `pnpm-workspace.yaml` already covers `packages/*`; no change
   needed there. `pnpm install` will pick the new package up.
7. Top-level `tsconfig.json` references list and root `package.json`
   `workspaces` glob already cover `packages/*`; only edit those if
   `pnpm -r typecheck` or `pnpm -r build` actually fails to discover
   the new package — verify by running both.
