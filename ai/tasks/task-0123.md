## Objective

Stand up the greenfield **`apps/admin-worker`** internal support/administration
worker per spec 16, delivering V1 **audited, read-only support diagnostics** with
deny-by-default support authorization and a support-action audit trail — without
introducing any privileged shortcut around tenant policy or audit. Satisfies
roadmap **B8** and the Constitution's secure-by-default rule.

## Milestone

**B8 — Admin / support worker (V1: read-only diagnostics + audited support actions).**

Deliver, end to end:

1. **`apps/admin-worker`** — a new `cloudflare-worker-turbo` internal worker mirroring
   the established worker shape (router.ts, handlers/, env.ts, http.ts, package.json,
   tsconfig, eslint.config.js, wrangler.jsonc, **component.yaml**). Internal-only:
   **NOT routed through api-edge**, no public surface. Implement the spec-16 V1
   capability subset:
   - `authorizeSupportAction` — deny-by-default; access granted only to a recognized
     support role or explicit system override. Denial emits `support.access_denied`.
   - `recordSupportAction` + `listSupportActions` — persist and read support-action
     records (support actor, target org, reason, requestId, timestamp). Every recorded
     action emits `support.action_recorded` via the established audit-event seam.
   - `lookupOrganizationForSupport` + `lookupUserForSupport` — tenant-safe **read-only**
     diagnostic projections (narrow is fine per spec-16 Agent Freedom; do NOT expose
     raw domain tables or secrets).
2. **DB migration** — a new checked-in migration under `packages/db/src/migrations`
   (next ordinal after `130_webhook_secret_rotation_grace`, i.e. `140_*`) creating the
   support-action record table(s) this worker owns. Follow the existing migration
   conventions (forward-only, idempotent guards as used by peers).
3. **Events** — register/emit the spec-16 support events that V1 uses
   (`support.action_recorded`, `support.access_denied`; impersonation events are V1
   out-of-scope, see Non-Goals). Reuse the events-audit seam; do NOT fork a new
   event-publish mechanism.
4. **Tests** — handler-level tests for: deny-by-default authorization (denied without
   support role/override, emits `support.access_denied`), record+list round-trip
   (emits `support.action_recorded`), and read-only lookups returning narrow
   projections. Mirror the membership-worker handler test style.

## Constraints

1. **Deny-by-default**: support authorization must fail closed. No support role / no
   system override ⇒ denied, and the denial is audited.
2. **Audit every support action**: every recorded support action creates an audit
   event through the existing `appendEventWithAudit` seam, inside the same transaction
   as the write (mirror membership-worker atomicity — write + event commit or roll back
   together).
3. **Tenant-safe reads only**: support reads use narrow diagnostic projections; never a
   privileged shortcut around policy/audit; never expose secrets or full connection
   strings.
4. **Internal-only**: admin-worker is not reachable via api-edge; no public route.
5. **Orun/golden-path**: the new app ships a `component.yaml` mirroring an adjacent
   peer (`apps/policy-worker/component.yaml` or `apps/membership-worker/component.yaml`);
   `spec.path` singular; deploy lane on `triggerRef: github-push-main` for stage/prod.
6. Terraform backend / secrets conventions unchanged; do not log secrets.
7. DB migration is forward-only and checked in; follows the `NNN_name` ordinal pattern.

## Integration Notes
