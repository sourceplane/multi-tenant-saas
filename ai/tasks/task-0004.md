# Current Repo Context

Task 0003 should have aligned `multi-tenant-saas` with the `aws-admin` Orun
golden path. The next blocker is AWS access: `multi-tenant-saas` needs
repo-scoped GitHub OIDC roles before it can use S3 Terraform state and AWS
Secrets Manager in CI.

This task is intentionally cross-repo. The implementation work happens in the
sibling repo `../aws-admin`; the planning context and downstream dependency are
tracked here in `multi-tenant-saas`.

# Objective

Add an `aws-admin` Terraform component that creates environment-scoped IAM roles
for `sourceplane/multi-tenant-saas`, with permissions to use shared S3
Terraform state and read/write AWS Secrets Manager secrets under:

```text
<org>/<repo>/<component>/<env>
```

For this repo, the namespace starts with:

```text
sourceplane/multi-tenant-saas/
```

# PR Boundary

One PR in `../aws-admin`.

The PR creates or updates only the repo-scoped IAM component, policy documents,
Terraform files, component README, and any minimal shared module changes needed
to support the permission model.

# Required Outcomes

- Create a new `aws-admin` component for `sourceplane/multi-tenant-saas` under
  the existing GitHub repository access domain.
- Follow the existing `aws-admin` component style exactly:
  - `component.yaml` with metadata labels;
  - Terraform root under `terraform/`;
  - `policies/plan.json` and `policies/deploy.json`;
  - colocated README with metadata, purpose, resources, trust model,
    permissions, parameters, outputs, and usage.
- Configure GitHub OIDC trust subjects for:
  - pull request planning;
  - main branch verification;
  - protected deploy/apply workflows, using the same environment-subject
    pattern as existing repo components unless a safer repo convention exists.
- Add read-oriented plan permissions for:
  - S3 state bucket/object reads for `sourceplane-dev`, `sourceplane-stage`,
    and `sourceplane-prod`;
  - IAM/OIDC inspection needed by Terraform;
  - Secrets Manager describe/list/get permissions for
    `sourceplane/multi-tenant-saas/*`;
  - `sts:GetCallerIdentity`.
- Add deploy/apply permissions for:
  - S3 state object writes/deletes for this repo's state keys;
  - Secrets Manager create/update/put/get/tag/restore/delete permissions limited
    to `sourceplane/multi-tenant-saas/*`;
  - any minimal KMS permissions required by the account's Secrets Manager
    encryption configuration.
- Keep permissions least-privilege and environment-aware. Avoid `Resource: "*"`
  for mutating Secrets Manager permissions unless AWS requires it for a list or
  discovery action, and document any exception.
- Run and verify the `aws-admin` Orun/Terraform checks locally and in CI.
- Merge the `aws-admin` PR only after CI is green and the roles/policies are
  verified in AWS.
- Update `multi-tenant-saas` context files only if the task runner is also
  asked to commit a tracking update here after the `aws-admin` merge.

# Non-Goals

- No changes in `multi-tenant-saas` CI or Terraform backend consumption. That
  is Task 0005.
- No Supabase database/project creation. That is Task 0006.
- No broad `aws-admin` module rewrite unless required for least-privilege
  secret path support.
- No manual IAM role creation in the AWS console.

# Constraints

- Use Orun/Terraform as the only provisioning path.
- Do not commit AWS credentials, account secrets, Terraform state, or generated
  `.orun/**` plans.
- Trust existing `aws-admin` conventions over assumptions.
- If role names would exceed AWS length limits, use the existing module's
  truncation/hash behavior and document the resolved names.
- If AWS account ID, KMS key, or Secrets Manager ARN format cannot be resolved,
  stop and record the blocker instead of widening permissions.

# Integration Notes

- The shared state buckets are named `sourceplane-<env>`.
- The multi-tenant SaaS Terraform state key pattern must be compatible with
  `env/<environment>/multi-tenant-saas/<component>/terraform.tfstate`.
- The downstream repo will need non-secret role ARNs or a deterministic way to
  derive them for `dev`, `stage`, and `prod`.
