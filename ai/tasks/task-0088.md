## Objective

Wire `membership-worker`'s invitation creation flow to enqueue an
`invitation.created` notification through `notifications-worker` over a
new `NOTIFICATIONS_WORKER` service binding, using the same best-effort,
non-blocking, contract-respecting pattern Task 0087 established for
`identity-worker`. This makes `membership-worker` the second real caller
on notifications V1 and exercises the `category: "invitation"` path of
the contract end-to-end on stage + prod.

## Architect Brief

- **Product bar:** Stripe-quality webhook delivery UX, applied to an
  internal enqueue: the user-visible invitation flow must never 5xx
  because notifications failed. Logging the create-invitation 201 to the
  caller is the source of truth; the notification is downstream.
- **User moment:** "I invited a teammate, and they got an email." This
  PR makes that real for prod (and keeps stage's existing
  debug-token-in-response contract intact for developers).
- **Implementer latitude (no spec proposal needed):**
  - shape of the membership-worker notifications client (new module vs
    importing/duplicating identity-worker's pattern — copying the file
    in-place is acceptable; do **not** introduce a shared package in
    this PR);
  - exact `templateKey` for the invitation email, but it MUST be
    stable, code-controlled, and named clearly (recommendation:
    `"invitation.created"` to match the existing `invite.created` event
    type the worker already emits in the same transaction);
  - the `templateData` shape (must remain redaction-safe — see
    Constraints);
  - test fixtures and mocking strategy for the new client;
  - whether to land the wire inside the `executor.transaction(...)`
    callback or immediately after it commits successfully. Recommended:
    enqueue **after** the transaction commits (a notification for a
    rolled-back invitation would be a worse failure than a missed
    notification for a committed one — same defensive ordering Task
    0087 used vs. `auth.startLogin`).
- **Implementer must NOT (would need a spec proposal):**
  - change `NotificationCategory` or any `EnqueueNotificationRequest`
    field shape;
  - add `"membership-worker"` to the internal-actor allow-list — it is
    already there;
  - rework `notifications-worker` provider / template handling;
  - introduce a `notifications-worker-dev` deploy target (deferred to a
    separate task);
  - touch `infra/terraform/cloudflare-domain/**` or the cloudflare
    provider pin (Task 0085b deferred);
  - propagate notification-enqueue errors into the membership-worker
    response — a non-2xx from notifications must NOT change the 201/4xx
    the caller sees.
- **Failure modes that invalidate this PR even if tests pass:**
  - the raw invitation token leaving the worker boundary inside
    `templateData` (raw token is a secret; only the hash is
    persisted, and only `DEBUG_DELIVERY === "true"` is allowed to put
    the token on the wire — and only in the response body, not in a
    notification);
  - a notifications-worker 5xx, missing binding, or thrown exception
    propagating into the invitation response;
  - a notification being enqueued for an invitation that did **not**
    commit (i.e., enqueueing inside the transaction such that a
    rollback still produces a user-visible email);
  - the `dev` env block sprouting a `NOTIFICATIONS_WORKER` binding to a
    non-existent service (no `notifications-worker-dev` exists).

> The implementer has full latitude on **client module shape, template
> key naming, templateData composition (within redaction-safety), test
> fixture style, and ordering relative to the transaction commit boundary**
> inside the Constraints below. Decisions taken under this latitude must
> be recorded with one-line rationale in the implementer report.
> Anything outside the listed dimensions requires a spec proposal before
> implementation.

## Constraints

- The raw invitation token (`rawToken` from `generateInvitationToken()`)
  MUST NOT appear in `templateData`, headers, or any logged surface
  outside the existing `DEBUG_DELIVERY === "true"` response body. Only
  the token hash is persisted; the raw token is delivery payload only.
- The enqueue MUST be best-effort: notifications-worker downtime,
  binding absence, or any client error MUST NOT change the invitation
  response status code, body, or headers.
- The enqueue MUST happen **outside the database transaction**, after
  it commits successfully. A rolled-back invitation MUST NOT produce a
  notification.
- The enqueue MUST NOT fire on policy-deny, billing-deny, validation,
  or `not_found` error branches.
- `templateData` keys and values stay redaction-safe per
  `packages/contracts/src/notifications.ts:86-89`: no secrets, no
  provider payloads, bounded primitives only.
- No changes to `notifications-worker`. The internal-actor allow-list
  already accepts `"membership-worker"`.
- Wrangler bindings only land on `stage` and `prod` env blocks. The
  `dev` block stays bindings-less; `notifications-worker-dev` does not
  exist and the implementer must not invent one.
- The notifications-client module's `Fetcher` type usage MUST match the
  same global-type convention identity-worker uses (no need to import
  `@cloudflare/workers-types` explicitly if the worker tsconfig already
  exposes it — mirror the precedent file).
