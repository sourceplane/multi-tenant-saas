# Current Repo Context

The reusable SaaS starter specs under `specs/**` are authoritative for this task. The current repo has specs, agent instructions, a minimal README, and initialized orchestrator state, but no implementation scaffold yet.

There are no open PRs. Prior merged PRs only updated specs and orchestration behavior. The next roadmap dependency is the initial Orun-discovered monorepo skeleton so later foundation and domain tasks have a real component graph.

# Objective

Create the first production-grade monorepo scaffold for the reusable multi-tenant SaaS starter: pnpm workspace, TypeScript/Turbo tooling, Orun intent/runtime files, Orun-only GitHub Actions CI, starter app/package/test/infra directories, and minimal runnable Worker and Pages app scaffolds.

# PR Boundary

One PR adds the initial workspace and Orun component skeleton only.

The PR should make a fresh checkout installable and locally verifiable at the scaffold level, but it must not implement domain business logic or provision live Cloudflare/Supabase resources.

# Required Outcomes

- Add root workspace files for a TypeScript pnpm monorepo, including package scripts for `dev`, `build`, `test`, `lint`, `typecheck`, and `deploy`.
- Add Turbo or an equivalent task graph configuration that supports the required root scripts without hardcoding CI behavior outside Orun.
- Add shared TypeScript/tooling configuration that apps and packages can extend.
- Add `intent.yaml` with discovery roots for `apps/`, `packages/`, `tests/`, and `infra/`, pinned to `oci://ghcr.io/sourceplane/stack-tectonic:0.12.0` unless inspection proves a newer approved version is required.
- Add `kiox.yaml` pinning the Orun runtime image required by the specs.
- Generate and commit `.orun/compositions.lock.yaml` if the local Orun lock command succeeds. If it cannot be generated because the external composition source or runtime is unavailable, record the exact command and failure in the report and keep the rest of the scaffold valid.
- Add `.github/workflows/ci.yml` that uses Orun plan/run only. Do not add direct `pnpm`, `turbo`, Wrangler, Terraform, or deploy commands in GitHub Actions.
- Add minimal `component.yaml` descriptors for the initial component graph:
  - `apps/api-edge`
  - `apps/web-console`
  - `packages/contracts`
  - `packages/shared`
  - `packages/testing`
  - `tests/contracts`
  - `infra/terraform/state`
  - `infra/terraform/core`
- Add a minimal Cloudflare Worker scaffold under `apps/api-edge` with typed environment bindings and a health-style response suitable for local development.
- Add a minimal Pages-compatible web console scaffold under `apps/web-console` that builds successfully but does not implement product UI flows yet.
- Add `packages/contracts` with starter contract exports and at least one schema/type that can be consumed by the contract test package.
- Add `tests/contracts` with a targeted contract test that runs through the workspace tooling.
- Add README content that documents the scaffold commands and makes clear that live Cloudflare/Supabase provisioning is not part of this PR.
- Write the implementer report at `ai/reports/task-0001-implementer.md`.

# Constraints

- Keep the PR scoped to one initial foundation scaffold. If Terraform resource implementation, database migrations, or real deployment behavior becomes necessary, stop and propose a follow-up task instead of expanding this PR.
- Preserve extraction-safe boundaries: shared packages may contain contracts, generic helpers, and tests only; no domain logic belongs in `packages/shared`.
- Component descriptors must follow the type names and dependency model described in `specs/repo.md` unless local Orun validation proves the spec is stale. If the spec is stale in a behavior-changing way, create an `ai/proposals/task-0001-spec-update.md` proposal and keep implementation conservative.
- The workflow must keep Orun as the only CI entry point.
- Use deterministic package manager metadata. Do not leave generated cache/build output committed.
- Keep all task commits on one branch and open one PR.

# Integration Notes

- Start from the actual empty implementation state; do not assume hidden packages or prior scaffolds exist.
- The API Worker and web console can be intentionally minimal, but they must be real buildable projects rather than placeholders.
- The initial infra component directories may include component descriptors and explanatory README files, but should not provision resources in this PR.
- The contract test should prove that `packages/contracts` is consumable through workspace imports.
- Prefer local reusable config packages or `tooling/*` only where it reduces duplication for this first scaffold.

# Acceptance Criteria

- Fresh install succeeds with the chosen package manager.
- `pnpm lint`, `pnpm typecheck`, `pnpm test`, and `pnpm build` succeed locally, or the report identifies a concrete external-tooling blocker with exact command output.
- The API Worker and web console each have a colocated `component.yaml`.
- `tests/contracts` has a colocated `component.yaml` and runs at least one meaningful test against `packages/contracts`.
- `.github/workflows/ci.yml` contains Orun plan/run workflow steps and no direct package test/build/deploy commands.
- `intent.yaml` discovers `apps/`, `packages/`, `tests/`, and `infra/`.
- If Orun is available, these commands are run and results are reported:

```bash
/Users/irinelinson/.local/bin/kiox -- orun compositions lock --intent intent.yaml
/Users/irinelinson/.local/bin/kiox -- orun validate --intent intent.yaml
/Users/irinelinson/.local/bin/kiox -- orun plan --changed --intent intent.yaml --output plan.json
/Users/irinelinson/.local/bin/kiox -- orun run --plan plan.json --dry-run --runner github-actions
```

- The PR does not create or modify live Cloudflare/Supabase resources.
- `ai/reports/task-0001-implementer.md` exists and includes the PR number.
