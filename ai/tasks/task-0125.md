## Why this task

This is a **stabilize-the-baseline / drift-proofing** task that permanently kills a recurring
failure class. The migration-manifest verifier hard-codes its own list of valid bounded
contexts:

- `tests/db/src/migrations.test.ts:52` — `const VALID_CONTEXTS: BoundedContext[] = [ ... ]`
  (11 string literals, manually maintained).
- `packages/db/src/types.ts:1` — `export type BoundedContext = "control" | ... | "support";`
  (the *intended* source of truth, but a **type** — erased at runtime, so the test array is a
  hand-copied duplicate).

Because TypeScript types vanish at runtime, the test array cannot be checked against the union
by the compiler in any load-bearing way — the two drift silently every time a new context is
added. This has already bitten the repo:

- **Task 0117 / PR #172** — a migration declared `context: "notifications"` but the test array
  was missing it → red `each migration declares a valid bounded context`; fixed by hand-adding
  one literal.
- Re-flagged as residual drift risk in the **0117, 0118, 0120, and 0124** verifier reports
  ("the test array and the `BoundedContext` union remain two hand-maintained lists").

Every future context (Task 0124 added `"support"`; future billing/SSO legs will add more) re-arms
the same trap. Deriving both from one runtime constant removes the duplication entirely.

## Goal

Make the set of valid bounded contexts a **single runtime source of truth** in `@saas/db`, and
derive **both** the `BoundedContext` type **and** the test's `VALID_CONTEXTS` from it. After this
change it must be structurally impossible for the test list to drift from the type union.

## Approach (the standard `as const` → `typeof[number]` pattern)

1. In `packages/db/src/types.ts`, introduce a runtime constant as the source of truth and derive
   the type from it. Keep the exact same 11 members, in the same order, so the public
   `BoundedContext` union is byte-for-byte equivalent:

   ```ts
   export const BOUNDED_CONTEXTS = [
     "control",
     "identity",
     "membership",
     "projects",
     "billing",
     "events",
     "config",
     "webhooks",
     "metering",
     "notifications",
     "support",
   ] as const;

   export type BoundedContext = (typeof BOUNDED_CONTEXTS)[number];
   ```

2. Export `BOUNDED_CONTEXTS` (value export) from the barrel `packages/db/src/index.ts` alongside
   the existing `BoundedContext` type export. Note `index.ts` currently uses
   `export type { ... }` for the type — you'll need a separate `export { BOUNDED_CONTEXTS } from
   "./types.js"` value-export line (do not move the type into a value export).

3. In `tests/db/src/migrations.test.ts`, **delete** the hand-maintained `VALID_CONTEXTS` literal
   (lines 52–64) and import the constant instead:

   ```ts
   import { manifest, BOUNDED_CONTEXTS } from "@saas/db";
   import type { BoundedContext, MigrationEntry } from "@saas/db";
   // ...
   const VALID_CONTEXTS = BOUNDED_CONTEXTS;
   ```

   (Either alias it or use `BOUNDED_CONTEXTS` directly in the `toContain` assertion — your call;
   keep the assertion semantics identical.) Leave the two `PROJECT_SCOPED_CONTEXTS:
   BoundedContext[] = ["projects"]` locals (lines 95, 112) untouched — they are a deliberate
   narrow subset, not the full-set duplicate, and are not the drift source.
