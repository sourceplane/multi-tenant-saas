# Objective

Add the first identity persistence foundation: an identity-owned database
migration plus Worker-safe typed repository adapters that future identity
Worker code can use without knowing about Hyperdrive, Postgres.js, migration
runner internals, or raw SQL.

This task should establish the reusable repository-adapter pattern for bounded
contexts while staying below the public API/domain-service line. Do not build
the identity Worker or login flow yet.

# Required Outcomes

## Worker-Safe SQL Executor

- Extend the Worker-safe DB surface with a small parameterized SQL executor
  abstraction suitable for repository adapters.
- The executor must support typed row results and parameterized values without
  string interpolation of untrusted input.
- `@saas/db/hyperdrive` may expose a factory for this executor or the existing
  adapter may grow the executor methods, as long as the existing health/ping
  behavior remains compatible.
- Keep `@saas/db` default exports Worker-safe and free of Postgres.js, `pg`,
  AWS SDK, `node:*`, filesystem, and migration-runner imports.
- Node-only migration runner code must remain behind `@saas/db/runner`.

## Identity Schema Migration

- Add an identity-owned migration with context `identity`, for example
  `010_identity_core`.
- The migration must be idempotent enough for the current Management API runner
  behavior documented in Task 0008.
- Prefer plain Postgres primitives. Do not require extensions such as `citext`
  unless you justify them in the report and tests.
- Create a dedicated identity schema or namespace. Suggested minimum tables:
  - `identity.users`
  - `identity.auth_identities`
  - `identity.login_challenges`
  - `identity.sessions`
- Store only secret hashes for login codes and session tokens. Do not store raw
  login codes, raw bearer tokens, raw API keys, or signing/encryption secrets.
- Use normalized email columns/indexes for lookup. Do not rely on
  case-sensitive raw email uniqueness.
- Keep foreign keys inside the identity context only. Do not add cross-context
  foreign keys to membership, projects, billing, events, or future tables.
- Add useful indexes and table/column comments where they clarify ownership or
  secret-handling boundaries.
- Update the manifest and checksum tests so migration ordering and drift
  detection still work.

## Identity Repository Adapter

- Add a Worker-safe identity repository export behind an explicit subpath.
- Repository methods should be typed and persistence-oriented, not public API
  handlers or domain-service orchestration.
- Include the minimum methods needed by a future first-party passwordless login
  task, such as:
  - create/get user by ID;
  - get user by normalized email;
  - attach/get auth identity by provider and subject;
  - create/get/consume login challenge using hashed codes;
  - create/get/revoke session using hashed tokens.
- Repository methods must accept caller-generated IDs, timestamps, hashes, and
  normalized inputs where appropriate. Do not add token generation, code
  generation, email delivery, passwordless-domain policy, or crypto secret
  ownership to `packages/db`.
- Map expected persistence outcomes to stable typed results or safe repository
  errors. Do not leak raw SQL errors, connection strings, token hashes, code
  hashes, usernames, hostnames, or credentials.
- Use parameterized SQL for all user-controlled values.
- Keep repository code extractable: no direct imports from Worker route
  handlers, `api-edge`, future identity-worker code, or unrelated bounded
  contexts.

## Tests

- Add focused tests for the SQL executor/repository layer using fake or
  in-memory executors; PR CI must not require a live database.
- Tests must cover:
  - parameterized query usage for user/email/session/challenge lookups;
  - successful row mapping;
  - not-found behavior;
  - conflict/duplicate behavior where the repository exposes it;
  - safe error mapping/redaction;
  - no raw code/token values in repository outputs or thrown errors;
  - no import path from Worker-safe identity/hyperdrive exports to runner-only
    modules.
- Extend migration tests to prove:
  - the identity migration is present, ordered, checksummed, and has context
    `identity`;
  - identity SQL creates the expected schema/tables/indexes;
  - project-scoped invariant tests still apply only to `projects` context
    migrations.

## Orun Migration Path

- Because this task adds a real migration, verify that the rendered changed
  Orun plan includes the right database jobs.
- PR CI must run migration plan/read-only behavior only; it must not apply
  migrations.
- Post-merge main behavior must apply the pending identity migration to
  `stage` and `prod` through `infra/db-migrate`, not through ad hoc CI or local
  scripts.
- If adding a migration under `packages/db/src/migrations/**` does not select
  `db-migrate` jobs in the changed plan, fix the Orun/component relationship in
  the smallest bounded way. If the current Orun model cannot support this
  safely, write a spec proposal, mark the report blocked, and do not pretend the
  live apply path is covered.
