# Task 0065

Agent: Implementer

## Current Repo Context

- Tasks 0055-0064 completed the config bounded context foundation through metadata-only secret create/rotate/revoke runtime.
- `config.secret_metadata` has a nullable `ciphertext_envelope` column, but `@saas/db/config` intentionally excludes that column from its safe read model and current public routes reject any secret material.
- `apps/config-worker` currently supports:
  - settings and feature flags list/create/update at organization, project, and environment scope;
  - secret metadata list/create/rotate/revoke at organization, project, and environment scope.
- `apps/web-console` currently exposes Config UI for settings/feature flags mutations and secret metadata listing only. It does not create, rotate, revoke, or store secret values.
- `specs/components/07-config-secrets-flags.md` requires encrypted tenant secrets, `putSecret`, `rotateSecret`, metadata listing, and no plaintext in logs/audit/event payloads.
- Current code reality: no encryption adapter, write-only secret payload API, encrypted payload persistence method, or Cloudflare Secrets Store/Worker secret binding exists for config-worker.
- Main baseline is green. Recent main CI runs `26585584501` and `26585656635` completed successfully.

## Objective

Add the first production-safe encrypted secret payload storage path for config secrets. Implement write-only create/rotate APIs that accept secret values, encrypt them in config-worker before persistence, store only ciphertext envelopes in `config.secret_metadata.ciphertext_envelope`, and continue returning/listing only safe public metadata.

This should turn the Task 0064 metadata-only secret surface into a real write-only secret storage primitive while preserving the no-reveal contract.

## PR Boundary

One coherent PR should cover the tightly coupled secret payload storage slice:

1. Add an encryption adapter seam for config-worker secret payloads.
   - Use Worker-compatible Web Crypto primitives.
   - Use an environment binding/secret value for envelope encryption key material rather than hardcoded keys.
   - Keep the adapter testable without Cloudflare deployment.
2. Extend the config repository with write-only persistence methods that set `ciphertext_envelope` while returning safe `SecretMetadata` only.
   - No repository method may expose ciphertext or plaintext through existing read/list/get surfaces.
   - If new internal methods are required, keep ciphertext write-only or unavailable outside config-worker internals.
3. Update config-worker create/rotate secret handlers so they can accept secret material only for the new write-only storage flow.
   - Create should require or clearly validate a secret payload when using the encrypted-storage endpoint semantics.
   - Rotate should accept replacement secret payload, encrypt it, update ciphertext envelope, increment version, and update `lastRotatedAt`.
   - Responses remain `PublicSecretMetadata` only.
4. Preserve exact route-scope enforcement, policy authorization, and transaction-coupled event/audit writes from Task 0064.
5. Add contract request types for write-only secret storage that make the sensitive field explicit but never include it in responses.
6. Add tests for encryption, persistence, route behavior, secret-safety, transaction behavior, and api-edge forwarding of POST bodies without downstream bearer forwarding.
7. Update config-worker deployment/config documentation or Wrangler binding shape only as needed for the encryption key binding, without provisioning live secrets in this PR.

## Required Outcomes

- [ ] Config-worker has a testable encryption adapter for secret payloads using Web Crypto and a configured key binding/secret, with no hardcoded production key material.
- [ ] Secret create accepts a write-only secret value, encrypts it before database persistence, and returns only public metadata.
- [ ] Secret rotate accepts a replacement secret value, encrypts it, updates the stored ciphertext envelope, increments version, sets `lastRotatedAt`, and returns only public metadata.
- [ ] Existing secret metadata list/get/create/rotate/revoke public response shapes remain free of plaintext, ciphertext, hashes, tokens, passwords, credentials, and raw bearer tokens.
- [ ] `ciphertext_envelope` is written only after encryption and is never selected by safe read/list/get repository methods.
- [ ] Authorization remains unchanged: org scope uses `organization.config.write`; project/environment scope uses `project.config.write` with explicit `projectId`.
- [ ] Exact route-scope enforcement for rotate/revoke remains in place before mutation.
- [ ] Event/audit payloads remain safe metadata only and never include plaintext or ciphertext envelopes.
- [ ] Mutation plus event/audit append remains transaction-coupled in production paths; encryption failure must abort before DB mutation.
- [ ] api-edge forwards the public create/rotate request body to config-worker, but still never forwards raw Authorization bearer tokens downstream.
- [ ] Tests cover success, validation failures, missing/invalid encryption key binding, encryption failure, authorization denial, exact-scope mismatch, event failure rollback/throw behavior, and secret material absence from responses/events/audit/loggable errors.
- [ ] Implementer report is written to `ai/reports/task-0065-implementer.md` and committed on the PR branch.

## Non-Goals

- No plaintext reveal/read/decrypt API.
- No web-console UI controls for entering or rotating secret values.
- No Cloudflare Secrets Store provisioning, Terraform, AWS Secrets Manager, KMS, KV cache, effective config resolution, config versioning, or feature-flag evaluation.
- No migration DDL unless code inspection proves the existing nullable `ciphertext_envelope` column is insufficient for safe encrypted storage.
- No broad policy model changes beyond existing config write actions.
- No unrelated UI polish, dependency bumps, formatting churn, or historical `ai/` cleanup.

## Constraints

1. Treat plaintext secret values as highly sensitive. They may exist only in request parsing/encryption memory and must not be returned, logged, audited, emitted, persisted in plaintext, or written into reports.
2. Do not name or print real secret values in tests, reports, logs, or snapshots. Use placeholder strings such as `[REDACTED]` in reports when discussing payloads.
3. Use authenticated encryption with a random nonce/IV per stored value. The stored envelope should include algorithm/version metadata and ciphertext/nonce material, but never plaintext.
4. The encryption key source must be configurable through environment/binding conventions. Local tests may use deterministic test keys, but production/stage/prod must not depend on committed key material.
5. Keep public IDs at API boundaries. Persist raw UUIDs internally and map public `sec_` IDs through existing helpers.
6. Preserve Task 0064 safe behavior for metadata-only list/revoke. If create/rotate request semantics change from metadata-only to write-only value-required, update contracts/tests/spec notes clearly and keep validation explicit.
7. Do not expose `ciphertext_envelope` through `PublicSecretMetadata`, list responses, audit rows, event payloads, errors, or console report output.
8. If Cloudflare-specific secret bindings cannot be fully represented safely in this PR, document the deployment prerequisite in `ai/reports/task-0065-implementer.md` and keep runtime fail-closed when the binding is absent.
