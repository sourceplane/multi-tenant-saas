## Key Shape Differences vs Task 0114 (READ CAREFULLY)

Unlike `enable`, the `disable` contract carries an **optional operator-supplied `reason`** and the response type **is re-exported** from the SDK:

1. `DisableWebhookEndpointRequest` is `{ reason?: string }` (contracts/src/webhooks.ts:65–67) — NOT empty. The CLI surfaces this through a new optional `--reason=TEXT` flag, forwarded into the request body verbatim when supplied, omitted (so the body is `{}`) when absent.
2. `DisableWebhookEndpointResponse` (`{ endpoint: PublicWebhookEndpoint }`) **IS** re-exported from `@saas/sdk` index (sdk/src/index.ts:174). The test file MUST import `DisableWebhookEndpointResponse` directly from `@saas/sdk` (do NOT reconstruct the shape locally — that local-reconstruction workaround in webhook-enable.test.ts was a consequence of the missing enable re-export; disable has no such gap).
3. After disable, the endpoint carries `status: "disabled"`, a populated `disabledReason` (the supplied reason or worker default), and a populated `disabledAt`. The human block therefore emits FOUR lines: `status`, `disabledReason`, `disabledAt`, `updatedAt` (vs enable's three). This is the only intentional human-block divergence from 0114.
