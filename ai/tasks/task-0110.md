## Objective

Add a new CLI subcommand `sourceplane webhook secrets rotate <endpointId>`
that calls `client.webhooks.rotateSecret`, prints the reveal-once
`whsec_<32hex>` plaintext exactly once (when present), and exits 0.
Mirrors the conventions of `webhook create` (`packages/cli/src/commands/
writes.ts:260`) for SDK plumbing and the 0107/0106 pattern for output
shaping and test harness.

## Architect Brief (Stripe-quality CLI)

- Product bar: **Stripe CLI rotate-key ergonomics.** The plaintext is
  the only thing a human cannot recover later — surface it loudly and
  exactly once.
- The user moment: an operator runs `sourceplane webhook secrets rotate
  wh_xxx`, sees the new `whsec_…` clearly delimited on stdout, copies
  it into their secret store, and re-pastes it nowhere else.
- Free to choose: human-output formatting (within Stripe-bar), JSON
  field ordering, exit-message wording for the legacy-no-key branch,
  flag-parsing helper extraction (inline vs `webhook-body.ts`-style
  shared util), test-fixture style.
- Must NOT change: SDK shape, contract types, idempotency-key
  semantics, the `webhook` command tree shape (subcommand path
  `[\"webhook\", \"secrets\", \"rotate\"]` is the chosen path — do not
  collapse to `[\"webhook\", \"rotate\"]` or expand to `[\"webhook\",
  \"secret\", \"rotate\"]` (singular). Three-segment plural matches
  `client.webhooks.rotateSecret` semantics and leaves room for
  future `webhook secrets list/reveal/revoke` without rename).
- Failure modes that invalidate the PR even with green CI: any
  persistence of `secret` to disk/log/env/transcript beyond the single
  stdout write; any code path that prints the secret on a non-success
  branch; any `.toString()` / template-interpolation of the secret
  into a wider object that could land in error messages.

## Surface

```
sourceplane webhook secrets rotate <endpointId>
  [--idempotency-key=KEY]
  [--output=human|json]            default human
```

`<endpointId>` is positional and required (`wh_…` shape — pass-through;
the CLI does NOT validate the prefix; the API is the source of truth).

Org is resolved via the existing `resolveOrgId(ctx, /*allowOverride*/
false)` helper in `writes.ts` — copy the import pattern, do NOT add a
`--org` override (mirrors `webhook create`).

### Output

**`--output=human` (default), with `secret` present:**

```
Webhook signing secret rotated for <endpointId> in <orgId>

  secret:           whsec_<32hex>          ← reveal-once, copy now
  secretVersion:    <n>
  previousExpires:  <ISO8601 or "(none)">
  gracePeriod:      <gracePeriodSeconds>s

⚠  This secret will not be shown again. Subscribers using the previous
   secret have until previousExpires to roll over via the
   X-Webhook-Signature-Previous header.
```

Exit 0.

**`--output=human`, with `secret` ABSENT (legacy no-encryption-key path):**

```
Webhook signing secret rotated for <endpointId> in <orgId>

  secretVersion:    <n>
  previousExpires:  <ISO8601 or "(none)">
  gracePeriod:      <gracePeriodSeconds>s

⚠  Plaintext was not returned by the server (no encryption key
   configured). Subscribers must obtain the new secret out of band.
```

Exit 0. (No fake/placeholder secret printed — mirrors the 0109 amber
affordance.)

**`--output=json`:**

```json
{"endpoint":{...full PublicWebhookEndpoint...},"secret":"whsec_…","previousSecretExpiresAt":"…","gracePeriodSeconds":86400}
```

Single line, exit 0. When `secret` is absent the field is omitted (do
NOT emit `"secret": null` — matches the contract's optional shape).

**Errors:**

- Missing `<endpointId>` → `UsageError`, exit 2.
- `--output=invalid` → `UsageError`, exit 2.
- SDK error (4xx/5xx) → propagated through the existing CLI error
  formatter (`formatCliError`); exit code per the existing pattern in
  `webhook create`. Do NOT swallow or re-wrap; do NOT log the secret
  in any error path (it cannot be present on the error path anyway,
  but defence-in-depth).

## Hard Rules

1. **Reveal-once discipline.** The `secret` field MUST be referenced
   exactly twice in the codebase: once when reading from the SDK
   response, once when writing to stdout. No interpolation into log
   strings, no inclusion in any object stashed beyond the function
   scope, no `JSON.stringify` of any wider object that contains it
   (json-mode prints the full response by design — that IS the single
   write), no debug-mode toggle.
2. **No `node:crypto` / `node:buffer` / Node-only crypto** — this
   command does not sign or verify; it just calls the SDK and prints.
3. **No `fetch(`, no manual `/v1/` paths** — go through `ctx.sdk()`.
4. **No new `eslint-disable` / `@ts-ignore` / `@ts-expect-error` /
   `as any` / `as unknown as` under `packages/cli/**`.** The
   `RotateWebhookSecretResponse` type is already exported from
   `packages/sdk/src/index.ts`.
5. **Idempotency-key passthrough only when supplied.** Use the
   existing `readIdempotencyKey(ctx)` helper from `writes.ts` (import,
   do not duplicate). Do NOT generate a default key — rotation is
   not idempotent in the same sense as create; keys are advisory.
6. **Implementer report on PR branch.** Commit
   `ai/reports/task-0110-implementer.md` to the PR branch before
   opening the PR (or in a follow-up commit before final push). Do
   NOT repeat the 0106 missing-report gap.
7. **Subcommand path is `[\"webhook\", \"secrets\", \"rotate\"]`** —
   three segments, plural \"secrets\". This is the durable shape for
   future `webhook secrets list/reveal/revoke` if/when they land.
