# Objective

Add the first narrow self-scoped account profile public API surface:

- `GET /v1/auth/profile`
- `PATCH /v1/auth/profile`

This task should let a signed-in user read and update their own safe profile field `displayName` through the public `api-edge` facade, backed by identity-owned persistence. It should establish the route contract clearly enough for a later web-console account settings UI without adding that UI now.

# Required Outcomes

- [ ] Document the public profile route contract.
  - Update the active reusable specs, minimally, to document:
    - `GET /v1/auth/profile`
    - `PATCH /v1/auth/profile`
  - Make clear these are self-scoped, identity-owned, authenticated user-session routes.
  - Make clear profile update is limited to `displayName` in this PR.
  - Make clear `email` is read-only in this route and account security settings are deferred.

- [ ] Add public contract types.
  - Extend `packages/contracts/src/auth.ts` with minimal typed shapes for:
    - profile response, likely `{ user: AuthUser }`
    - profile update request, likely `{ displayName: string | null }`
  - Preserve existing auth/session/login contract compatibility.
  - Add or update focused contract tests if the local pattern expects type-shape coverage.

- [ ] Add identity repository support for updating profile.
  - Add a repository method such as `updateUserProfile(userId, { displayName, updatedAt })`.
  - Persist only to `identity.users.display_name` and `updated_at`.
  - Return the updated `User`.
  - Use the existing identity schema; do not add a migration unless strictly needed.
  - Update `tests/identity-worker` fake repository support.

- [ ] Add identity-worker profile runtime.
  - Add handler/router support for:
    - `GET /v1/auth/profile`
    - `PATCH /v1/auth/profile`
  - Require a valid user session bearer token.
  - Do not allow API-key/service-principal actors to update user profiles.
  - Return standard success/error envelopes with `meta.requestId` and `meta.cursor`.
  - Validate `PATCH` body:
    - body must be a JSON object
    - `displayName` must be present and either `string` or `null`
    - strings must be trimmed
    - empty string should clear the display name to `null`
    - maximum stored display name length: 120 characters
    - unsupported fields such as `email`, `id`, `status`, or arbitrary keys must return `validation_failed`
  - Never expose internal UUID-only values, token hashes, bearer tokens, or raw session secrets.

- [ ] Record safe identity security event coverage for profile updates.
  - Successful `PATCH /v1/auth/profile` must record an identity-owned security event named `user.profile.updated`.
  - Event metadata must be safe: include changed field names such as `["displayName"]`, not old/new display-name values, emails, bearer tokens, or secrets.
  - Couple profile update and security-event recording transactionally where feasible using the existing `TransactionalSqlExecutor` pattern. If a transaction is not feasible without broad refactor, document the bounded reason in the report and keep failure behavior conservative.
  - Do not write org-scoped audit/event copies for this self-scoped pre-organization route.

- [ ] Extend api-edge auth facade safely.
  - Route `GET` and `PATCH /v1/auth/profile` to `IDENTITY_WORKER`.
  - Preserve `Authorization`, `content-type`, `x-request-id`, `traceparent`, and `idempotency-key` forwarding behavior where applicable.
  - Forward request body for `PATCH` as well as `POST`.
  - Return `405`/`unsupported` for unsupported methods.
  - Do not change org/project/API-key route behavior.

- [ ] Add focused tests.
  - Identity-worker tests should cover:
    - unauthenticated GET/PATCH rejection
    - GET returns signed-in user's `id`, `email`, and `displayName`
    - PATCH updates `displayName`
    - PATCH `""` stores/returns `null`
    - PATCH rejects unsupported fields and invalid types/too-long values
    - PATCH records `user.profile.updated` security event with safe metadata
    - API-key-shaped bearer is not accepted as a user-session profile updater
  - api-edge tests should cover:
    - `isAuthRoute("/v1/auth/profile")`
    - GET/PATCH forwarding to `IDENTITY_WORKER`
    - PATCH body forwarding
    - query string preservation for GET, if any
    - unsupported methods return 405
  - Contract tests should cover the new profile request/response types if consistent with existing contract-test style.
