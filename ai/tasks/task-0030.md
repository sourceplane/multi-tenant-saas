# Objective

Add a membership-owned internal authorization-context seam so future
non-membership workers, starting with `projects-worker`, can obtain policy-ready
membership facts without querying membership storage directly.

# PR Boundary

One PR adds:

- shared internal contract types for the authorization-context query;
- a private `membership-worker` internal route/handler that returns policy-ready
  membership facts for a subject and organization;
- any small helper extraction needed so existing membership-worker policy calls
  and the new internal route use the same role-assignment-to-fact mapping;
- focused tests and the implementer report.

This PR must not add `projects-worker`, public project routes, `api-edge`
forwarding, policy-engine action changes, database migrations, provider changes,
or `specs-v2/**` work.

# Required Outcomes

1. Add shared internal contract types.
   - Extend the shared policy contract surface in `@saas/contracts/policy` with
     typed request/response shapes for the new internal authorization-context
     query.
   - The request should include:
     - `subject` with `type` and `id`
     - `orgId`
   - The response should include:
     - `memberships: MembershipFact[]`
   - Keep the contract narrow. Do not add raw database row shapes, role
     assignment IDs, member IDs, or future attributes unless code reality
     clearly requires them.

2. Add a private membership-worker internal route.
   - Add `POST /v1/internal/membership/authorization-context`.
   - Do not expose it through `api-edge`.
   - Validate the JSON body and return the existing safe error envelope on bad
     input.
   - Require `SOURCEPLANE_DB`; if it is missing, fail safely with the existing
     style of `internal_error`/service-unavailable response.
   - Use the membership repository to load active role assignments for the
     requested subject within the requested organization.
   - Return policy-ready membership facts only:
     - organization scope -> `{ kind: "organization", orgId }`
     - project scope -> `{ kind: "project", orgId, projectId: scopeRef }`
   - Do not call `policy-worker` from this route. This task is only the
     membership-owned fact query seam.

3. Prevent mapping drift inside membership-worker.
   - Extract or consolidate the role-assignment -> `MembershipFact` mapping into
     a reusable helper inside `apps/membership-worker`.
   - Existing `authorizeViaPolicy(...)` callers must continue to use the same
     policy fact shape after the refactor.
   - Preserve current behavior for the existing public routes:
     - `GET /v1/organizations/{orgId}`
     - `GET /v1/organizations/{orgId}/members`
     - invitation create/list/revoke/accept
     - member role update/remove

4. Keep the seam private and bounded.
   - No new public Worker routes.
   - No new `api-edge` path matching or forwarding.
   - No new database migration.
   - No policy-engine action or evaluation change.
   - No direct `projects-worker` scaffold in this PR.
   - The response may contain raw internal `orgId` / `projectId` scope strings
     because the seam is internal and policy already consumes internal scope
     IDs, but it must not expose raw row IDs unrelated to scope resolution.

5. Add or update tests.
   - Contract tests should prove the new request/response types are structurally
     usable from `@saas/contracts/policy`.
   - Membership-worker tests must cover:
     - successful POST returning organization-scoped and project-scoped
       membership facts;
     - validation failure for malformed body;
     - unsupported method handling;
     - missing DB binding returns safe failure;
     - repository failure returns safe failure;
     - project-scoped role assignments map `scopeRef` to `scope.projectId`;
     - the response does not include raw role-assignment IDs or member IDs.
   - Existing membership-worker tests for policy-gated public routes must remain
     green after the helper extraction.

6. Preserve Orun/component behavior.
   - Local Orun validation must still pass.
   - The changed plan for this PR should include the expected contracts and
     membership-worker jobs, plus membership-worker-tests/contracts-tests as
     applicable.
   - If Orun fails to select an expected component, inspect the component path
     wiring and either fix it or report the blocker clearly.
