# Objective

Add the first deployable Membership Worker runtime slice and expose the initial
organization bootstrap/read routes through `api-edge`.

After this PR, an authenticated user should be able to call the public API edge
to:

- `POST /v1/organizations` to create an organization;
- `GET /v1/organizations` to list organizations for the current user;
- `GET /v1/organizations/{orgId}` to read an organization the current user
  belongs to.

The organization creator must become an active member with an `owner` role
assignment by using the Task 0015 `bootstrapOrganization` repository method.

## Organization Runtime Behavior

- Use `@saas/db/hyperdrive` and `@saas/db/membership` inside
  `membership-worker`; do not query identity tables or import identity Worker
  code.
- `POST /v1/organizations`:
  - require an authenticated user actor resolved by `api-edge`;
  - accept JSON with `name` and optional `slug`;
  - validate lengths and allowed slug characters conservatively;
  - normalize slug lookup with lower-case `slugLower`;
  - generate UUIDs with Web Crypto for the organization, creator member, and
    owner role-assignment storage IDs;
  - expose public organization IDs at the API boundary using a stable prefix
    such as `org_<hex>`, and persist UUIDs to Postgres;
  - store the identity public user ID from the session response as the opaque
    membership `subjectId` with `subjectType: "user"`;
  - call `bootstrapOrganization` so organization creation, creator membership,
    and owner role assignment are all-or-nothing at the repository level;
  - return a standard success envelope with the organization and the creator's
    owner membership/role summary.
- `GET /v1/organizations`:
  - list active organizations for the current actor by subject ID;
  - return standard success envelopes with public organization IDs.
- `GET /v1/organizations/{orgId}`:
  - parse the public organization ID back to the storage UUID;
  - return the organization only when the current actor has an active
    membership/role fact for that organization;
  - avoid cross-tenant leaks. Prefer `not_found` for inaccessible or unknown
    organization IDs unless you have a stronger local convention already in the
    codebase.
- Map repository outcomes to semantic API errors from
  `specs/contracts/api-guidelines.md`.
- Keep returned errors safe: no raw SQL errors, connection details, hostnames,
  usernames, token material, token hashes, stack traces, or provider internals.
- Do not implement full policy decisions. A local "current actor belongs to
  this organization" guard is acceptable for the membership context until the
  policy Worker exists; role-to-action authorization remains a future task.
