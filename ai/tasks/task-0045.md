# Objective

Expose the first public query surface for identity-owned security history via an
authenticated `GET /v1/auth/security-events` route.

This task should add the contract types, identity-worker handler/router,
api-edge forwarding, cursor pagination, and focused tests needed to let an
authenticated user list their own security events without widening the scope to
API keys, account settings, org-scoped audit copies, or web-console UI.

# PR Boundary

One PR adds the identity security-event list/query surface only:

- `@saas/contracts` types for the public security-event response shape;
- `apps/identity-worker` route/handler logic for authenticated per-user list
  queries with pagination and safe response mapping;
- `apps/api-edge` auth-facade forwarding for the new auth route;
- focused `tests/identity-worker` and `tests/api-edge` coverage.

Do not combine this with API-key/service-principal behavior, account profile or
security-settings mutations, web-console UI, notification delivery,
organization-scoped audit/event copies, or new database migrations.

# Required Outcomes

1. Add public contract types for identity security-event listing.
   - Extend `@saas/contracts` with a public list response shape for
     authenticated identity security events.
   - Keep the envelope consistent with the normative success/error contract.
   - Define a stable list item shape that is safe for public use.
   - Do not expose raw secret values, token material, or other credential
     payloads through the contract.

2. Add `GET /v1/auth/security-events` to the identity-worker route surface.
   - Require bearer-token authentication like `GET /v1/auth/session`.
   - Resolve the authenticated user through the existing auth service/runtime.
   - Query `querySecurityEventsByUser` with cursor pagination (`limit`,
     `cursor`) using the existing `(occurred_at DESC, id DESC)` semantics.
   - Return `validation_failed` for invalid `limit` or malformed `cursor`,
     matching `specs/contracts/api-guidelines.md`.

3. Map stored identity security events to a safe public response shape.
   - Keep the list user-scoped and pre-organization; do not require `orgId`.
   - Preserve useful security-history facts such as `eventType`, `outcome`,
     `occurredAt`, request/correlation context when present, and safe metadata.
   - Convert internal identity references to public-safe values where exposed.
     Avoid leaking raw UUID-only internal references unless there is already an
     established public format for that field.
   - Apply redaction paths or equivalent safe redaction behavior before sending
     metadata back to clients.

4. Add api-edge forwarding for the new route.
   - Treat the new endpoint as part of the auth route surface.
   - Preserve request ID, authorization, traceparent, and query string.
   - Keep edge behavior transport-only: no business rules, no pagination logic,
     and no auth context resolution beyond existing auth-route forwarding.

5. Add focused tests around the real behavior.
   - `tests/identity-worker` should cover successful list responses, pagination,
     invalid `limit`, invalid `cursor`, unauthenticated requests, and redaction.
   - Reuse or extend the fake identity repository’s security-event query seam
     instead of creating a second mirrored implementation.
   - `tests/api-edge` should cover route matching/forwarding for
     `GET /v1/auth/security-events`, including query-string preservation and
     safe error behavior.

# Constraints

- Preserve the rule that pre-organization identity security history stays
  identity-owned and user-scoped rather than becoming an org-less shared audit
  route.
- Follow the public API rules in `specs/contracts/api-guidelines.md`:
  `/v1` route prefix, standard success/error envelopes, cursor pagination,
  default limit `50`, max limit `100`, opaque cursor semantics.
- Keep api-edge responsible only for transport concerns and identity-worker
  responsible for validation, business rules, persistence access, and response
  shaping.
- Do not expose raw login codes, bearer tokens, token hashes, session secrets,
  API keys, provider secrets, or any other credential material.
- Avoid broad cross-app abstractions. A small local pagination helper in
  `identity-worker` is fine; do not create a new shared framework layer for one
  route.
- Do not add unrelated cleanup, README refreshes, or speculative follow-on UI
  work.

# Integration Notes

- `querySecurityEventsByUser` already exists in `@saas/db/identity`; this task
  should consume that seam rather than redesigning persistence.
- `apps/events-worker/src/handlers/list-audit.ts` and
  `apps/events-worker/src/pagination.ts` are useful reference implementations
  for pagination validation and next-cursor encoding, but do not create an app-
  to-app dependency.
- `apps/api-edge/src/auth-facade.ts` already forwards `/v1/auth/login/start`,
  `/v1/auth/login/complete`, `/v1/auth/session`, and `/v1/auth/logout`; extend
  that existing seam.
- Downstream work such as account-security UI or API-key management should build
  on this backend route after it is stable.

# Acceptance Criteria

- Authenticated clients can call `GET /v1/auth/security-events` and receive a
  standard success envelope with `meta.requestId` and `meta.cursor`.
- The route lists only the authenticated user’s identity security events.
- Pagination follows the V1 contract: `limit`/`cursor`, default `50`, max `100`,
  opaque next cursor, and `validation_failed` on invalid inputs.
- Returned metadata is redacted/safe and does not expose raw codes, bearer
  tokens, token hashes, token secrets, API keys, or provider secrets.
- api-edge forwards the route correctly and preserves query string + required
  headers.
- No API-key/account-settings UI or org-scoped audit-copy work is included.
- A GitHub PR is opened for the task.
