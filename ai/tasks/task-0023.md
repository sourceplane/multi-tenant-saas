# Objective

Add the first events/audit persistence foundation for the reusable SaaS starter:

- shared event and audit contract types in `@saas/contracts`;
- an `events` bounded-context database migration in `@saas/db`;
- Worker-safe repository adapters for appending canonical event envelopes,
  writing audit projections, and querying audit history by organization and
  target.

This is a persistence and contract foundation only. Do not add an
`events-worker` runtime or wire existing membership/identity mutations to it in
this PR.

# Required Outcomes

1. Add shared event/audit contract types.
   - Add a new contracts module, for example `packages/contracts/src/events.ts`,
     and export it from `packages/contracts/src/index.ts`.
   - Model the current event envelope schema:
     - `id`, `type`, `version`, `source`, `occurredAt`;
     - actor type/id/session/ip;
     - tenant `orgId`, optional `projectId`, optional `environmentId`;
     - subject kind/id/name;
     - trace request/correlation/causation/idempotency fields;
     - JSON payload;
     - optional audit redaction metadata.
   - Include audit-facing types for immutable audit entries and query filters.
   - Keep types transport-safe and package-safe:
     - no platform clients;
     - no database row types in the public contract module;
     - no `any` unless there is no safer JSON alternative.
   - Use stable string IDs in contracts. Do not introduce client-visible UUID
     assumptions.

2. Add the events bounded-context migration.
   - Add a new migration directory under `packages/db/src/migrations/`, expected
     shape `030_events_audit_core/up.sql` unless code reality strongly suggests
     a different next migration number.
   - Update `packages/db/src/manifest.ts` with the migration entry, context
     `events`, checksum, and clear description.
   - Create an `events` schema and at minimum:
     - a canonical immutable event-log table;
     - an audit-entry projection/index table.
   - Table design must support:
     - one canonical event row per event envelope;
     - audit entries linked to the canonical event when applicable;
     - organization-scoped audit queries;
     - target-resource queries by kind/id;
     - security/audit categorization or equivalent filtering for future
       security-event views;
     - request ID and trace fields;
     - JSON payload storage;
     - audit redaction paths.
   - Every tenant-scoped events/audit table must include `org_id` directly.
   - Project/environment columns must be present and nullable where applicable.
   - Do not add cross-context foreign keys. A foreign key from audit entries to
     the canonical event table inside the same `events` schema is allowed.
   - Use idempotent DDL (`IF NOT EXISTS`) and deterministic index names.
   - Do not require Postgres extensions unless absolutely necessary. If an
     extension is truly necessary, write a spec proposal instead of adding it.

3. Add a Worker-safe events repository adapter.
   - Add `packages/db/src/events/types.ts`, `repository.ts`, and `index.ts` or
     the local equivalent pattern.
   - Export `@saas/db/events` from `packages/db/package.json` and
     `packages/db/src/index.ts`.
   - Repository methods must include:
     - append a canonical event envelope;
     - append a canonical event envelope plus an audit projection in one
       database statement where possible;
     - query audit entries by organization with cursor pagination;
     - query audit entries by organization plus target kind/id.
   - Use the existing `SqlExecutor` seam. Do not import `pg`, Hyperdrive
     bindings, Wrangler types, Worker env types, or platform clients directly in
     events domain repository code.
   - SQL must be parameterized.
   - Duplicate event IDs must not overwrite existing events.
   - Stored event and audit payloads must be treated as immutable after insert.
   - Cursor pagination should follow the Task 0020 pattern:
     - `limit` default behavior may live at the caller layer later, but the
       repository query input must support bounded `limit` and cursor;
     - order by `(occurred_at DESC, id DESC)` or a clearly documented equivalent;
     - return a next cursor or `null`.
   - Do not log, print, or expose secret-like payload values. The repository
     stores provided JSON; callers are responsible for payload choice and
     `audit.redact` metadata.

4. Add focused tests.
   - Contract tests should prove the new types are exported and structurally
     match the event envelope shape.
   - Migration tests should prove:
     - manifest ordering and checksum include the new migration;
     - migration context is `events`;
     - `events` schema and required tables are created;
     - organization scope columns exist;
     - project/environment columns exist;
     - no cross-context foreign keys exist;
     - DDL is idempotent;
     - JSON payload and redaction storage are present;
     - useful indexes exist for organization/time and target lookups.
   - Repository tests should cover:
     - appending a canonical event;
     - appending event plus audit projection atomically in one SQL statement or
       explicitly proving the local executor pattern cannot partially apply it;
     - duplicate event ID safe conflict behavior;
     - audit query by organization;
     - audit query by target kind/id;
     - cursor pagination and next-cursor behavior;
     - JSON serialization/deserialization;
     - redaction path preservation;
     - SQL parameterization.
   - Tests must not require live Supabase, Cloudflare, AWS, Wrangler, or
     external network access.

5. Keep Orun/component behavior intact.
   - Do not add new component types or CI bypasses.
   - If migration/component path metadata needs adjustment, keep it scoped to
     `packages/db` and `tests/db`.
   - The existing db migration component must remain the path that applies
     migrations to stage/prod after merge.

# Integration Notes

- `packages/db/src/types.ts` already includes `events` as a valid bounded
  context.
- `specs/contracts/event-envelope.schema.yaml` is the durable schema reference;
  TypeScript contract types should match it closely enough for downstream code
  to build against without reading YAML at runtime.
- Existing repository adapters map database rows to domain objects and keep SQL
  behind the `SqlExecutor` interface. Follow that shape.
- Existing list pagination in membership uses `(created_at DESC, id DESC)`.
  Events/audit should use the analogous event timestamp field, normally
  `occurred_at`.
- The current migration runner applies manifest migrations in lexicographic
  order and verifies checksums. Update the manifest checksum after the SQL is
  final.
- If you find that the current event-envelope schema is insufficient for a
  production-safe audit persistence contract, write a proposal under
  `ai/proposals/` and keep this PR conservative.
