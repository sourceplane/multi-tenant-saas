# Task 0064

Agent: Implementer

## Current Repo Context

- Tasks 0055-0061 established the config bounded context: config persistence, read-only config-worker APIs, non-secret settings/feature-flag mutations, exact route-scope enforcement, and web-console mutation UI.
- Config secret metadata is currently read-only through public routes. `packages/db/src/config` has metadata-only repository methods (`createSecretMetadata`, `rotateSecretMetadata`, `revokeSecretMetadata`) that intentionally exclude `ciphertext_envelope` and never expose plaintext.
- `apps/config-worker/src/router.ts` currently returns 405 for `POST .../config/secrets`; no item-level secret metadata mutation routes exist.
- Task 0063 was verified and merged via PR #106. Post-merge main CI run `26583005845` completed successfully.

## Objective

Add the public config-worker and api-edge runtime for secret metadata create, rotate, and revoke operations at organization, project, and environment scope, without implementing encrypted secret payload storage or any plaintext reveal/read flow.

This advances the `config, secrets, feature flags, and security settings` starter module while honoring Constitution rules 6 (events/audit), 8 (secure by default), and 11 (bounded contexts).

## PR Boundary

One PR should add only the metadata mutation runtime for config secrets:

1. Add config-worker handlers and routing for:
   - `POST /v1/organizations/{orgId}/config/secrets`
   - `POST /v1/organizations/{orgId}/projects/{projectId}/config/secrets`
   - `POST /v1/organizations/{orgId}/projects/{projectId}/environments/{environmentId}/config/secrets`
   - `POST /v1/organizations/{orgId}/config/secrets/{secretId}/rotate`
   - `POST /v1/organizations/{orgId}/projects/{projectId}/config/secrets/{secretId}/rotate`
   - `POST /v1/organizations/{orgId}/projects/{projectId}/environments/{environmentId}/config/secrets/{secretId}/rotate`
   - `DELETE /v1/organizations/{orgId}/config/secrets/{secretId}`
   - `DELETE /v1/organizations/{orgId}/projects/{projectId}/config/secrets/{secretId}`
   - `DELETE /v1/organizations/{orgId}/projects/{projectId}/environments/{environmentId}/config/secrets/{secretId}`
2. Add api-edge facade forwarding for the same routes, preserving actor resolution and not forwarding raw bearer tokens.
3. Add contract types/helpers for secret metadata create/rotate/revoke request and response shapes if existing exports are insufficient.
4. Add tests for config-worker handlers/router, api-edge facade matching/forwarding, and secret-safety invariants.

## Required Outcomes

- [ ] Config-worker accepts metadata-only secret create at org/project/environment scope.
- [ ] Config-worker accepts metadata-only secret rotate and revoke by public `sec_` ID at org/project/environment scope.
- [ ] Mutation authorization uses the existing membership-worker authorization-context seam and policy-worker with the same write permissions as config setting/feature-flag writes: org scope uses `organization.config.write`; project/environment scope uses `project.config.write` with explicit `projectId`.
- [ ] Item-level secret mutations enforce exact route scope before mutation, equivalent to Task 0060's `scopeMatchesRequested()` protection for settings and feature flags.
- [ ] Mutations emit safe event/audit records atomically with the metadata mutation when running on the transaction-capable SQL executor.
- [ ] Event/audit payloads and API responses include metadata only: secret ID, scope, key/display name/status/version/rotation timestamps. They must not include plaintext, ciphertext, hashes, or secret values.
- [ ] api-edge forwards the public routes to config-worker with method, path, query, body when relevant, `x-request-id`, actor headers, and optional `Idempotency-Key`; it must not forward raw Authorization bearer tokens downstream.
- [ ] Tests cover success, validation failures, authorization denial, exact-scope mismatch, conflict/not-found mapping, event failure rollback path or throw behavior, and no secret-material exposure.
- [ ] Implementer report is written to `ai/reports/task-0064-implementer.md` and committed on the PR branch.

## Non-Goals

- No encrypted secret payload storage or `ciphertext_envelope` writes.
- No plaintext secret value in any request/response/event/audit/log.
- No secret reveal/read API.
- No web-console UI for secret creation/rotation/revoke.
- No config versioning or effective config resolution work.
- No KV cache, Cloudflare Secrets Store, Workers Secrets Store, or KMS integration.
- No Terraform, Wrangler deployment config, infrastructure, or live resource changes.
- No policy model broadening beyond reusing the existing config write actions.
- No unrelated UI polish, docs cleanup, dependency bumps, or generated file commits.

## Constraints

1. Public IDs must stay at API boundaries. Persist raw UUIDs internally and map secret metadata IDs with existing `sec_` public ID helpers.
2. Secret keys must be validated with a conservative allowlist equivalent to setting/flag keys; reject unsupported fields rather than silently ignoring them.
3. The request body for create must be metadata-only. If a caller sends `value`, `plaintext`, `secret`, `ciphertext`, `ciphertextEnvelope`, `hash`, or similar secret-material fields, return validation failure.
4. Rotate in this task means metadata rotation only: increment version/update `lastRotatedAt` using the existing repository method. It must not accept or store a replacement secret value.
5. Revoke must move active metadata to `revoked`; it must not hard-delete records.
6. Exact route-scope checks are mandatory for rotate/revoke. A `sec_` record stored at project A must not be mutable through org-only, project B, or environment B routes.
7. Config-worker must fail closed on missing/failed membership or policy service binding responses.
8. Keep all mutation event/audit payloads safe: include operation, scope, secret key, metadata ID, and version/status only; never include secret payload material.
9. Do not modify migration DDL unless a focused test proves the repository contract cannot support the task safely. If DDL changes are required, keep them task-scoped and ensure db-migrate changed-plan behavior is covered.
10. Respect existing lint/type/test conventions and avoid unrelated formatting churn.
