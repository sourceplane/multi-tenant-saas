# Objective

Add project archival through the public API:

`DELETE /v1/organizations/{orgId}/projects/{projectId}`

This endpoint must soft-archive the project, not hard-delete it, and must write
a `project.archived` event/audit entry atomically with the archive mutation.

This PR must not add project update, restore, hard delete, environment routes,
project role-assignment management, project-scoped filtered lists, policy action
changes, database migrations, Terraform, Cloudflare account-level resources, UI,

# Required Outcomes

1. Add projects-worker archival behavior.
   - Add `DELETE /v1/organizations/{orgId}/projects/{projectId}`.
   - Parse both public IDs (`org_`, `prj_`) and return 404 for malformed IDs.
   - Require actor headers exactly like existing create/get/list routes.
   - Do not accept a request body.
   - Use membership-worker authorization-context with actor subject and raw org
     UUID.
   - Authorize via policy-worker action `project.delete` with explicit resource
     `{ kind: "project", id: projectUuid, orgId, projectId: projectUuid }`.
   - Fail closed: membership-context errors, malformed envelopes, policy
     errors, malformed policy responses, or policy denial must return safe 404
     or service-unavailable style errors consistent with existing projects
     behavior.
   - Use `archiveProject(orgId, projectId, archivedAt)`; never hard-delete.
   - Return a standard success envelope with `{ project: PublicProject }`,
     using the archived row returned by the repository.
   - Archived projects must not appear in the existing list endpoint because
     `listProjectsPaged` filters active projects; do not change that behavior.

2. Write event/audit atomically.
   - Archive mutation and `project.archived` event/audit append must share the
     transaction-capable SQL executor pattern used by project creation.
   - If event/audit append fails, the archive mutation must not successfully
     commit.
   - Event/audit DB scope columns may use raw UUIDs as existing persistence
     schema requires, but public-facing event payloads, audit descriptions, API
     responses, and tests must use `org_` and `prj_` IDs and must not expose raw
     UUIDs.
   - Payload and audit text must not include database connection details,
     bearer tokens, API keys, SQL, stack traces, or other secret material.
   - Use event type `project.archived`, source `projects-worker`, category
     `projects`, and a concise audit description.

3. Update api-edge project facade.
   - Forward `DELETE /v1/organizations/{orgId}/projects/{projectId}` to
     `PROJECTS_WORKER` after identity resolution.
   - Forward actor headers, request ID, traceparent, and idempotency-key as
     appropriate.
   - Continue not forwarding bearer tokens.
   - Preserve existing POST create, GET list, and GET item behavior.
   - Do not expose project update, restore, hard delete semantics, or
     environment routes.

4. Keep policy and bounded-context seams clean.
   - Do not add or rename policy actions.
   - Use existing `project.delete` policy semantics.
   - Do not import membership repository code from projects-worker.
   - Do not query `membership.*` storage from projects-worker.
   - Do not expose policy or membership internal routes through api-edge.

5. Add tests.
   - projects-worker tests must cover:
     - archive success with authorization-context + policy allow;
     - `project.delete` policy action and explicit project resource shape;
     - malformed public org/project IDs return 404;
     - missing actor returns 401;
     - unsupported methods still return 405;
     - missing DB, membership binding, or policy binding fails safely;
     - membership-context failure/malformed envelope fails closed;
     - policy denial and policy fetch/malformed envelope fail closed;
     - repository not-found maps to 404;
     - repository internal failure maps to safe service error;
     - event/audit append failure prevents successful archive;
     - response/event payload/audit description do not expose raw UUIDs or
       secret material.
   - api-edge tests must cover:
     - DELETE project item route forwards to `PROJECTS_WORKER` after identity
       resolution;
     - actor headers are forwarded;
     - bearer token is not forwarded;
     - missing `PROJECTS_WORKER` returns safe 503;
     - existing POST create, GET list, and GET item behavior remains intact.
   - Existing policy-engine tests already cover `project.delete`; add only
     focused tests if implementation changes require it.

6. Preserve Orun/component behavior.
   - Local Orun validation must pass.
   - Changed plan should include expected projects-worker/api-edge test and
     deploy-verify jobs for the files changed.
   - It should not include db-migrate, Terraform, Supabase, AWS, S3, Secrets
     Manager, or unrelated infra apply jobs.
   - If Orun fails to select an expected component, inspect path/component
     ownership and fix it or report a blocker clearly.
