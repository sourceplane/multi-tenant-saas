## Objective

Bump the cloudflare Terraform provider in
`infra/terraform/cloudflare-domain/` from `~> 4.52` to `~> 5.x`,
rename `cloudflare_workers_domain.console` →
`cloudflare_workers_custom_domain.console`, and carry the rename
through Terraform state via a `moved {}` block so the two live
resource IDs are preserved unchanged. Post-merge `terraform apply` on
both stage and prod must be a clean no-op (or documented `moved`
no-op) with zero resource churn, and apex + rollback-hatch probes must
remain 200.

## Architect Brief

- **Product bar:** Stripe-quality infra change discipline — zero
  user-visible regression, zero resource recreate, an audit trail that
  a reviewer can follow in one pass.
- **The user moment this improves:** none directly — this is the last
  piece of Cloudflare v4 debt in the repo, and unblocks any future
  Cloudflare resource (Workers, Zero Trust, Hyperdrive v5 features)
  from being added on a current provider line.
- **Free to decide without asking:** exact upper-bound pin shape
  (`~> 5.0`, `~> 5.x.y`, etc., as long as it floats forward on patch
  releases); whether to keep the `count` + `local.has_custom_domain`
  gating pattern or refactor to `for_each` (only if the change is
  state-neutral via `moved`); how to organize the `moved` blocks
  (single block or split per-resource); whether to refresh the
  `.terraform.lock.hcl` in the PR or rely on CI; ordering of
  README/comment updates.
- **Must not decide without a spec proposal:** any change to apex
  hostnames, Worker service names, environment routing, or the
  resource's behavior beyond the rename; any change to provider
  authentication shape (`api_token` env wiring stays); any change to
  Orun composition, intent.yaml parameterDefaults, or job template;
  any silent destroy/create on the live custom-domain resource (this
  is the failure mode that would invalidate the PR even if tests pass).
- **Failure modes that invalidate the PR even with green CI:**
  - Plan diff on either env shows `1 to destroy, 1 to add` instead of
    `0 added, 0 changed, 0 destroyed` (or a clean `# … has moved …`
    no-op). Resource IDs must survive byte-identical.
  - Apex hostname stops resolving or stops serving 200 with
    `<title>Sourceplane Console</title>`.
  - Provider pin still resolves to a 4.x release after the bump.
  - v5 attribute drift (e.g. an argument renamed or moved between
    blocks in v5) silently shadowed by a default and not surfaced in
    the plan diff.
- **Recommended approach:** read the cloudflare provider v5 upgrade
  guide for `workers_custom_domain` (or the equivalent v5 changelog
  entry), translate any attribute renames into the resource block at
  the same time as the rename, and add the `moved {}` block in the
  same file. If the v5 resource exposes a different `environment`
  enum or drops the field, encode the new shape — do not paper over
  with a `lifecycle.ignore_changes`. Document any non-obvious choice
  in the implementer report with a one-line rationale.

> The implementer has full latitude on provider pin shape, `moved`
> block organization, `.terraform.lock.hcl` refresh strategy, and
> README/comment cleanup. Decisions taken under this latitude must be
> recorded with one-line rationale in the implementer report.
> Anything outside the listed dimensions requires a spec proposal
> before implementation.
