# Objective

Replace the web-console placeholder with the first usable live-test console for
the public APIs that already exist, and add the minimal `api-edge` CORS support
needed for the Cloudflare Pages UI to exercise stage and prod from a browser.

This PR should let a human sign in, choose stage or prod, and test the current
organization, membership, invitation, project, environment, and audit flows from
the deployed web console.

# Required Outcomes

1. Build an actual app surface, not a marketing page.
   - First screen must be the working console.
   - Keep environment, session, organization, and project scope visible.
   - Use a restrained operational UI suitable for repeated testing.
   - Keep the UI responsive for desktop and mobile widths.

2. Add stage/prod API target selection.
   - Provide built-in targets for the verified stage and prod API edge URLs.
   - Keep session/token state scoped per target so stage and prod do not share
     credentials accidentally.
   - A custom/local API target is acceptable for developer testing, but stage
     and prod must be first-class.

3. Implement auth/session flows against the public API edge.
   - `POST /v1/auth/login/start`
   - `POST /v1/auth/login/complete`
   - `GET /v1/auth/session`
   - `POST /v1/auth/logout`
   - Show the debug code only when the API returns `delivery.code` (stage does;
     prod does not).
   - Prod must still be testable by entering a received code or importing a
     bearer token manually, because first-party email delivery is not in scope.

4. Implement organization and membership workflows that match current public
   routes.
   - List and create organizations.
   - Select an organization and fetch its details.
   - List members.
   - Update member role.
   - Remove a member, with clear handling for last-owner or authorization
     failures.
   - Create, list, revoke, and accept invitations.

5. Implement project and environment workflows that match current public routes.
   - List, create, fetch, and archive projects.
   - List, create, fetch, and archive environments under an explicit project.
   - Do not infer project scope from an environment ID alone.

6. Implement audit inspection for current organization audit events.
   - `GET /v1/organizations/{orgId}/audit`
   - Support category filtering and cursor pagination.
   - Display request IDs and error envelopes when calls fail.

7. Add a small web-console API client layer.
   - Use the shared public contract types where available.
   - Preserve the API envelope shape and cursor semantics.
   - Send bearer tokens only through the `Authorization` header.
   - Surface semantic error codes, messages, and request IDs.
   - Avoid direct calls to internal Workers or service-binding-only routes.

8. Add browser-safe CORS support to `api-edge`.
   - Handle `OPTIONS` preflight before route-specific method checks.
   - Allow the deployed web-console origin and Cloudflare Pages preview origins
     for `sourceplane-web-console`.
   - Allow local development origins such as Vite preview/dev on localhost.
   - Allow the headers needed by the existing public API:
     `authorization`, `content-type`, `x-request-id`, `traceparent`, and
     `idempotency-key`.
   - Add CORS headers to successful and error responses, including downstream
     facade responses.
   - Use `Vary: Origin` and do not allow arbitrary production origins.

9. Update Cloudflare Pages smoke coverage.
   - Keep `apps/web-console/component.yaml` deploying through the existing
     `cloudflare-pages-turbo` composition.
   - Replace the no-op smoke command with a meaningful check that the deployed
     Pages app loads and that stage/prod API health endpoints are reachable.
   - If the composition cannot expose the deployment URL directly, use the known
     Pages project URL and document that assumption in the report.

# Integration Notes

- Stage auth returns `delivery.mode: "local_debug"` and includes `delivery.code`.
- Prod auth returns `delivery.mode: "email"` and does not include a raw code.
  Because email delivery is not implemented in this slice, the UI must not
  require debug-code behavior to test prod.
- Existing public list routes use `limit` and `cursor` query params and return
  `meta.cursor`.
- Organization IDs use `org_`, project IDs use `prj_`, environment IDs use
  `env_`, member IDs use `mem_`, and invitation IDs use `inv_` at the public
  API boundary.
- The current Pages deployment is at `https://sourceplane-web-console.pages.dev/`.
- The verified public API edge URLs are:

  - prod: `https://api-edge-prod.rahulvarghesepullely.workers.dev`
