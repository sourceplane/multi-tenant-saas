## Objective

Land a Stripe-quality per-org and per-identity rate-limiter at
`api-edge`. The limiter MUST:

- Run at the edge before any cross-binding fetch and before any
  `replayOrExecute` lookup (limit decision happens before cache work).
- Apply two independent buckets per request and reject if EITHER overflows:
  - `org`: keyed on `orgId` when an authenticated org-scoped actor is
    resolved, otherwise skipped.
  - `identity`: keyed on the `(actorType, actorId)` pair when any actor
    type is resolved (user, service-principal, api-key); for fully
    anonymous traffic the key collapses to `anon:<routeFamily>:<clientIp>`
    so abuse from a single IP cannot bypass the limit.
- Return the standard envelope on overflow: HTTP 429 with body
  `{"error":{"code":"rate_limited","message":"...","details":{"scope":"org|identity","retryAfterSeconds":N}}}`,
  `Retry-After` header set, `X-RateLimit-*` headers populated on every
  response (allowed and rejected).
- Fail open. If the rate-limit KV / Durable Object backend is unreachable,
  log via the existing observability hook and admit the request, exactly
  the same posture the replay store takes for KV failures.
- Be scoped per route family, not per literal path. Configuration is
  declarative (a single typed config map keyed on facade name + an
  optional override per route) — no per-route inline numbers scattered
  through the seven facades.

## Architect Brief

- **Product bar: Stripe API.** Rate-limit headers and envelope shape
  must match Stripe's documented contract. Specifically:
  - `X-RateLimit-Limit-<scope>` (one per active scope)
  - `X-RateLimit-Remaining-<scope>`
  - `X-RateLimit-Reset-<scope>` (UTC seconds since epoch when the bucket
    next refills)
  - `Retry-After` (integer seconds) on 429 responses
  - The `error.code` is `rate_limited` (already declared in
    `specs/contracts/api-guidelines.md`); do not invent a new code.
- **User moment improved.** A noisy retry loop from one tenant cannot
  starve other tenants on the same edge worker. A leaked API key cannot
  be brute-forced past `N` requests/minute on auth routes. Closes the
  abuse-control gap specs/components/01-edge-api.md calls out under
  "rate limiting and coarse abuse controls."
- **Free to make** (record one-line rationale in the implementer report,
  do not stop to ask):
  - Backend choice: Cloudflare KV with millisecond timestamps, or a
    Durable Object counter, or Cloudflare's native `RateLimit` binding.
    KV is the cheapest and reuses the cloudflare-kv slice; Durable
    Objects give exact counters; native RateLimit binding gives no
    custom logic. Pick one with a one-paragraph rationale weighing cost,
    accuracy, and operational surface.
  - Window strategy (fixed window, sliding window log, leaky bucket,
    token bucket). Stripe-quality default is **token bucket with refill
    per second** for smooth burst tolerance — pick this unless the
    chosen backend forces a different shape.
  - Per-route-family limits — pick defaults that don't break the
    repo's current legitimate traffic. Suggested defaults (deviate
    only with rationale):
      - `auth` family: 10 req/min per identity, 60 req/min per org.
      - `org` / `project` / `config` / `webhooks` / `metering` / `billing`:
        60 req/min per identity, 300 req/min per org.
  - Whether the limiter sees the path-prefix (e.g. `/v1/auth/`) directly
    or receives a logical `routeFamily` string from each facade. The
    facade-supplied string is more robust to future route adds and
    matches the existing `replayOrExecute` chokepoint pattern.
- **Hard rules.**
  - Do not regress any current legitimate traffic (Phase 8 stage smoke
    on the verifier MUST show every replay-store live case still
    passing post-merge).
  - Do not change error code names. The envelope must use `rate_limited`
    verbatim.
  - Do not add a new public route. The limiter is internal seam only.
  - Do not bypass `resolveActor` ordering on the under-limit path.
  - Do not touch `infra/terraform/cloudflare-domain/**`.
  - Do not touch the cloudflare-kv component's existing
    `api_edge_idempotency` namespace — if a new namespace is needed,
    add it as a sibling resource in the SAME `cloudflare-kv` slice
    (do not fork a new component).
  - Do not change `apps/api-edge/wrangler.jsonc` IDEMPOTENCY_KV IDs.
- **Failure modes that invalidate the PR even if tests pass:**
  - A non-overflow request gains observable latency above
    +20ms p95 from the limiter check (sloppy KV round-trip path).
  - A 429 leaks Hyperdrive / KV / DO internals through the wire.
  - Any facade silently bypasses the limiter (verifier scans for this).
  - Limiter backend failure 5xx's the request instead of failing open.
  - `X-RateLimit-*` headers are missing on allowed responses, breaking
    the documented Stripe-parity surface.
  - The limiter shares storage with `IDEMPOTENCY_KV` (key namespace
    collision risk — verifier will diff the keyspace).
- **Recommended pattern (deviate only with documented one-line rationale):**
  - Add a `rate-limit.ts` module in `apps/api-edge/src/` exporting
    `enforceRateLimit(request, requestId, env, ctx)` returning either
    `{kind: "allowed", headers: Record<string,string>}` or
    `{kind: "denied", response: Response}`.
  - Wire it inside the existing `replayOrExecute` chokepoint so every
    facade picks it up automatically — `replayOrExecute` calls
    `enforceRateLimit` BEFORE the KV cache lookup and short-circuits with
    the 429 response if denied. Allowed `headers` are merged into the
    final response (both replay-hit and downstream paths).
  - Each facade passes a `routeFamily` literal alongside the existing
    `replayOrExecute` arguments; do not derive the family from the URL
    inside the helper.
  - Backend: Cloudflare Workers KV is the recommended default given the
    monorepo just established the pattern. Token-bucket state encoded as
    `{tokens, refilledAt}` JSON, atomic via `get → compute → put` with
    a "last writer wins under concurrent retry" tolerance for V1.
    (Document the precision trade-off in the report.)
