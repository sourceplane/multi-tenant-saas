# Objective

Replace `apps/web-console-next/src/lib/api.ts` (297 LOC, parallel
fetch/envelope/error implementation) with `Sourceplane` from
`@saas/sdk`. After this PR the console must construct a single
`Sourceplane` client per selected target and dispatch every API call —
including the entire `/v1/auth/*` flow on `login/page.tsx` — through
`client.<resource>.*`. The duplicated wire logic in `lib/api.ts` must
be deleted; the only thing that may remain in `apps/web-console-next/**`
is consumer glue (target-selection, env lock, error-shape adapter if
you choose path B below).

# Constraints

- Stripe parity preserved: any call site that today passes an
  idempotency key (the console currently does not; verify) must
  continue to do so via `RequestOptions.idempotencyKey`. Do NOT auto-
  generate one in the console.
- The error-shape contract the console exposes (`ApiErrorBody` with
  `code`/`message`/`reason`/`details`/`requestId`) must remain
  available to the precondition-failed UX. Map the SDK's typed errors
  (`SourceplaneError` subclasses) to that shape — they carry the same
  fields under the hood. The mapping helper is allowed to live in the
  thin adapter (Path B) or in `useAsync` / a new `lib/sdkError.ts`
  (Path A); your call.
- The list-shape tolerance the current console relies on
  (`Array.isArray(data) ? data : data.organizations ?? []`) is no
  longer needed — every SDK list response uses the documented
  `{<resource>: T[], nextCursor: string | null}` shape per
  spec-13. Your migration must update consumers to read the
  documented field directly. Audit every list call to confirm.
- Per-target client lifecycle: when `setTarget` or `setToken` fires,
  rebuild the `Sourceplane` instance (or use a token-mutable wrapper).
  Don't leak the previous target's auth header into the next target.
- `useAsync` and `precondition/insight` must keep their public signature
  unless you can justify a tighter type — keep churn minimal.

# Integration Notes

- The SDK accepts a `fetch` override via `ClientOptions.fetch`. The
  console doesn't need to override it; rely on the platform global
  to keep behavior identical.
- The SDK auto-generates `x-request-id` per request when omitted —
  same behavior the console currently relies on for the
  `meta.requestId` round-trip. The error envelope-merge logic in the
  current console (preserving `meta.requestId` on the failure branch)
  is already done inside the SDK transport; verify by reading
  `packages/sdk/src/transport.ts` and `errors.ts` — no console-side
  workaround should remain.
- For `client.events.list` (audit), confirm the iterator and cursor
  semantics match the current console's `listAudit` consumer
  (Task 0102 added the audit iterator). The console likely doesn't
  use the iterator yet — keep behavior identical (single-page reads
  with the cursor passed forward).
- Login flow: `loginStart` returns `LoginStartResponse` (challengeId
  + delivery mode). `loginComplete` returns `LoginCompleteResponse`
  (token + session). On PROD, `delivery.mode === "email"` and no raw
  code is returned — preserve that branch in the page UI.
