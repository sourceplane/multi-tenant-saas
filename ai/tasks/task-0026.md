# Required Outcomes

1. Wire `invite.created` atomically.
   - In the production path for `POST /v1/organizations/{orgId}/invitations`,
     create membership and events repositories from the same transaction-bound
     executor.
   - Policy authorization must remain outside and before the transaction.
   - If policy denies, actor role lookup fails, validation fails, token
     generation fails, DB is missing, or policy-worker is missing, do not append
     an event.
   - If `createInvitation(...)` fails, do not append an event.
   - If event/audit append fails or conflicts, the invitation creation must not
     commit.
   - Keep the public response body compatible with Task 0021, including the
     existing debug-delivery behavior. Never store or emit the raw debug token
     in event/audit data.

2. Wire `invite.accepted` atomically.
   - In the production path for
     `POST /v1/organizations/{orgId}/invitations/accept`, create membership and
     events repositories from the same transaction-bound executor.
   - Acceptance remains authorized by valid token possession plus authenticated
     email match and explicit path organization match; do not add a
     policy-worker call to acceptance.
   - If validation fails, token hash lookup fails, invitation is missing,
     expired, revoked, already accepted, email mismatched, path org mismatched,
     or membership conflict occurs, do not append an event.
   - If `acceptInvitation(...)` succeeds but event/audit append fails or
     conflicts, the invitation acceptance, member creation, and role-assignment
     creation must roll back together.
   - Keep the public response body compatible with Task 0022.

3. Define safe event/audit contents.
   - Event types: `invite.created` and `invite.accepted`.
   - Event version: `1`.
   - Event source: `membership-worker`.
   - Actor: use the authenticated actor context already forwarded by api-edge.
   - Tenant org: use the public organization ID from the route (`org_...`), not
     the raw DB UUID.
   - Subject: invitation target using the public invitation ID (`inv_...`).
   - Trace: include `requestId`; include correlation/causation/idempotency only
     if available through current request context without broad api-edge changes.
   - Payload must include only audit-safe metadata, such as role, status,
     expiration timestamp, and accepted member public ID for acceptance.
   - Do not include raw invitation tokens, token hashes, bearer tokens, raw DB
     UUIDs, provider details, SQL, stack traces, or generated debug delivery
     tokens.
   - Prefer omitting invitee email from event/audit payload, subject name, and
     audit description. If email is included, add a matching redaction path and
     tests proving redaction metadata is set.
   - Audit category should be `membership`.
   - Audit descriptions should be human-readable and secret-free.

4. Keep tests focused and realistic.
   - Membership-worker tests must cover:
     - successful create appends `invite.created` event/audit;
     - create event/audit append failure returns a safe error and prevents
       commit in the production transaction path or an equivalent transaction
       fake;
     - create policy denial appends no event;
     - create validation/repository failures append no event;
     - successful accept appends `invite.accepted` event/audit;
     - accept event/audit append failure returns a safe error and prevents
       commit in the production transaction path or an equivalent transaction
       fake;
     - accept not-found/expired/revoked/already-accepted/conflict paths append
       no event;
     - event/audit values use public org/invitation/member IDs and do not expose
       raw UUIDs, tokens, token hashes, bearer tokens, SQL, or provider details;
     - public response shape remains compatible for create and accept routes.
   - Reuse or lightly extend the Task 0024 test seams. Avoid broad test rewrites.
   - Api-edge tests should only change if handler request/response or forwarding
     behavior actually changes. Prefer no api-edge changes.
   - DB repository tests should only change if event repository behavior changes.

5. Preserve deployment and Orun behavior.
   - No new infrastructure resources.
   - No new service bindings.
   - Existing membership-worker deployments to stage/prod may change because the
     Worker code changes; keep `workers_dev: false`.
   - Existing database migration state must not change in this task.
