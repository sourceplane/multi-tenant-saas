# Required Outcomes

- Create a new Terraform component, expected path:
  `infra/terraform/supabase/`.
- Add a colocated `component.yaml` using the repo's Terraform component style:
  `spec.type: terraform`, `spec.parameters`, `terraformDir: terraform`, pinned
  Terraform version, `plan-only` default profile, `apply` profile rules, labels,
  and explicit dependencies.
- Subscribe the component only to `stage` and `prod`. Do not add a `dev`
  subscription, state path, secret, or Supabase project in this task.
- Use S3 backend state with the Task 0005 backend contract.
- Add Terraform that creates two fresh Supabase projects, each with its own
  primary Postgres database:
  - `stage`: project name `multi-tenant-saas-stage`
  - `prod`: project name `multi-tenant-saas-prod`
- Use Supabase organization slug/id `dwazxcrywsdbxpuouifa` for both projects.
- Use Supabase region `ap-southeast-1` to match the historical Task 0002
  Supabase/Hyperdrive baseline unless the provider/API proves that region is
  unavailable for new projects. If it is unavailable, stop and report the exact
  provider/API response instead of choosing another region silently.
- Generate database credentials with Terraform, not by hand.
- Authenticate the Supabase provider from CI secret `SUPABASE_API_KEY`. If the
  selected provider expects `SUPABASE_ACCESS_TOKEN` or a provider
  `access_token`, map the existing GitHub secret to that provider input in an
  Orun-planned Terraform step without logging the value.
- Store generated credentials and connection details in AWS Secrets Manager
  under:

```text
sourceplane/multi-tenant-saas/supabase/stage
sourceplane/multi-tenant-saas/supabase/prod
```

- Mark sensitive Terraform variables and outputs as sensitive. Do not output
  generated passwords, service keys, or connection strings in plaintext.
- Document the secret JSON shape in the README without including real values.
  At minimum, include placeholders for project ref, project URL, database host,
  database port, database name, database user, generated password, and connection
  URI.
- Record non-secret project refs and names in Terraform outputs and the report
  after apply. Do not invent refs before Supabase assigns them.
- Decide and document whether Cloudflare Hyperdrive wiring belongs in this
  component or in a follow-up Cloudflare infra component. If it is included,
  keep it tightly scoped and verify the resulting Hyperdrive resource directly.
- Verify provider state after live apply using Supabase CLI and/or Management
  API read calls. Prefer the authenticated local Supabase CLI for read-only
  status checks where it works.
- Update relevant specs/context if implementation discovers a necessary
  contract adjustment.
- Write the implementer report.
