# Objective

Add a Terraform component that provisions the target Supabase database/project
infrastructure with S3 backend state and stores generated connection secrets in
AWS Secrets Manager.

# PR Boundary

One PR in `multi-tenant-saas`.

This task owns one Supabase infrastructure component and the directly required
Terraform composition/schema/docs changes. It may create live Supabase and AWS
Secrets Manager resources only through Orun/Terraform after plan review.

# Required Outcomes

- Create a new Terraform component, expected path:
  `infra/terraform/supabase/`.
- Add a colocated `component.yaml` using the repo's Terraform component style:
  `spec.type: terraform`, `spec.parameters`, `terraformDir: terraform`, pinned
  Terraform version, `plan-only` default profile, `apply` profile rules, labels,
  and explicit dependencies.
- Use S3 backend state with the Task 0005 backend contract.
- Add Terraform that creates the target Supabase database/project resources for
  the selected environments.
- Generate database credentials with Terraform, not by hand.
- Store generated credentials and connection details in AWS Secrets Manager
  under:

```text
sourceplane/multi-tenant-saas/supabase/<env>
```

- Mark sensitive Terraform variables and outputs as sensitive. Do not output
  generated passwords, service keys, or connection strings in plaintext.
- Document the secret JSON shape in the README without including real values.
- Decide and document whether Cloudflare Hyperdrive wiring belongs in this
  component or in a follow-up Cloudflare infra component. If it is included,
  keep it tightly scoped and verify the resulting Hyperdrive resource directly.
- Update relevant specs/context if implementation discovers a necessary
  contract adjustment.
- Write the implementer report.

# Non-Goals

- No domain schema migrations or application repository adapters.
- No identity, organization, membership, billing, audit, webhooks,
  notifications, CLI, SDK, or UI behavior.
- No broad Cloudflare Worker binding rollout unless explicitly scoped to this
  Supabase component.
- No manual Supabase dashboard or AWS console setup.
- No product-specific `specs-v2/**` work.

# Constraints

- Use official provider documentation for the selected Supabase provider and
  provider version. Do not guess resource names or schema fields.
- Do not commit secrets, Terraform state, `.terraform/**`, `.orun/**`,
  `plan.json`, generated provider caches, or local `.tfvars` files.
- Do not rotate or expose credentials for the historical Task 0002
  human-provided database.
- If Supabase org/project/account choice is unclear, pause using the human input
  protocol instead of creating resources in the wrong account.
- If Terraform plan proposes replacing or deleting an existing resource, stop
  and investigate before apply.

# Integration Notes

- The component should depend on the AWS access/backend state established by
  Task 0005.
- Secret names are non-secret and may be documented. Secret values are not.
- Apply should run only through the Orun Terraform `apply` profile after the
  plan has been reviewed and CI is green.
- Resource verification should use provider state, Terraform state, and
  read-only CLI/API checks. Do not rely on command success alone.
