# Required Outcomes

1. Add a transaction-capable SQL seam.
   - Extend the current db SQL abstraction in a backward-compatible way so
     existing repository tests and fakes typed as `SqlExecutor` do not all need
     unrelated rewrites.
   - A recommended shape is a separate exported interface such as
     `TransactionalSqlExecutor extends SqlExecutor` with:
     `transaction<T>(fn: (executor: SqlExecutor) => Promise<T>): Promise<T>`.
   - `createSqlExecutor(...)` may return a transactional executor while
     preserving existing `execute(...)` and `dispose()` behavior.
   - The transaction implementation must use the underlying `postgres` client's
     transaction support, not manual string concatenation.
   - Add tests proving:
     - normal `execute` behavior is unchanged;
     - transaction callback receives an executor;
     - successful callback commits;
     - thrown callback rolls back/rejects;
     - disposal behavior remains safe.
   - Do not add transaction behavior to the Supabase migration runner unless it
     is required for this task.

2. Wire invitation revocation to events/audit atomically.
   - In the production path for
     `DELETE /v1/organizations/{orgId}/invitations/{invitationId}`, create both
     `MembershipRepository` and `EventsRepository` over the same transaction
     executor.
   - The transaction must include:
     - the existing `revokeInvitation(...)` membership update;
     - one `appendEventWithAudit(...)` call for an `invite.revoked` event.
   - If policy authorization fails, do not start the mutation transaction and do
     not append an event.
   - If `revokeInvitation(...)` returns not found or internal error, do not append
     an event.
   - If event/audit append fails or conflicts, the invitation revocation must not
     commit.
   - Keep the public response body unchanged except for any existing request ID
     envelope behavior.
   - Continue to fail closed for missing DB, missing policy-worker, policy
     errors, malformed public IDs, and repository failures.

3. Define safe event/audit contents.
   - Event type: `invite.revoked`.
   - Event version: `1`.
   - Event source: `membership-worker`.
   - Actor: use the authenticated actor context already forwarded by api-edge.
   - Tenant: use the public organization ID from the route (`org_...`) for the
     event/audit organization identifier; do not expose raw organization UUIDs in
     event/audit rows intended for public audit queries.
   - Subject: invitation target using the public invitation ID (`inv_...`).
   - Trace: include `requestId`; include correlation/causation/idempotency only
     if available through current request context without broad api-edge changes.
   - Payload must include only audit-safe metadata. Never include raw invitation
     token, token hash, bearer token, DB UUIDs, provider details, or stack traces.
   - If invitation email is included in payload, subject name, or audit
     description, add a redaction path that matches the existing
     `redactPaths`/`audit.redact` convention. Prefer omitting email unless it is
     needed for a useful audit entry.
   - Audit category should be stable and low-cardinality, for example
     `membership`.
   - Audit description should be human-readable but must not include secrets.

4. Keep tests focused and realistic.
   - DB tests must cover the new transaction seam.
   - Membership-worker tests must cover:
     - successful revoke appends `invite.revoked` event/audit in the same
       transaction;
     - event/audit append failure causes a safe error and prevents commit;
     - policy denial appends no event;
     - invitation not found appends no event;
     - missing policy-worker and missing DB behavior remains safe;
     - response shape remains compatible with Task 0021;
     - event/audit values use public org/invitation IDs and do not expose raw
       UUIDs, tokens, token hashes, bearer tokens, SQL, or provider details;
     - transaction is not used by dependency-injected unit tests unless the test
       explicitly exercises the production path.
   - Add or adjust small test fakes rather than broad rewrites.
   - Existing invitation create/list/revoke and acceptance behavior must remain
     compatible.

5. Preserve deployment and Orun behavior.
   - No new infrastructure resources.
   - No `events-worker`.
   - No new service binding.
   - Existing membership-worker deployments to stage/prod may change because the
     Worker code changes; keep `workers_dev: false`.
   - Existing db migration state must not change in this task.
