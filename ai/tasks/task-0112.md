## Architect Brief

- **Product bar:** Vercel / Linear / Stripe Dashboard. The webhooks
  surface is a buyer-facing console artifact and must look credible
  against that bar. The existing list/detail layout already meets the
  bar — match it.
- **User moment this PR improves:** an org admin can create, rename,
  re-target, disable, re-enable, and delete a webhook endpoint **from
  the console**, instead of being told to drop into the CLI. This
  removes the last "use the API or CLI" placeholder in the webhooks
  surface.
- **Free to choose:** dialog vs. drawer for create/edit, exact form
  copy, button placement on the detail page, optimistic-vs-reload
  refetch strategy after mutations, where the destructive (delete)
  confirm lives, validation messaging.
- **Must not change without a spec proposal:** webhook endpoint
  contracts (`packages/contracts/src/webhooks.ts`), SDK signatures
  (`packages/sdk/src/webhooks.ts`), api-edge facade, webhooks-worker
  routes, the reveal-once rotate flow (Task 0109/0110 lineage —
  byte-equivalent), or any *project-scoped* endpoint creation surface
  (org-scope only this PR; project-scope is a separate follow-on).
- **Failure modes that invalidate the PR even if tests pass:** rendering
  the signing secret anywhere outside the existing reveal-once dialog;
  bypassing `wrap()` and leaking raw API error envelopes into UI strings;
  shipping a UI primitive outside `@/components/ui/*`; introducing a
  new public route on api-edge or webhooks-worker; adding inline
  duplicates of helpers that already exist in `@/components/webhooks/`.
- **Recommended stack:** `ZodForm` from `@/components/ui/zod-form` for
  create/edit (mirror `apps/web-console-next/src/app/(app)/orgs/[orgSlug]/invitations/page.tsx`
  for the dialog + zod schema + `wrap()` + `precondition_failed`
  handling shape — that's the canonical exemplar in this repo).
  Idempotency-Key on create via the SDK's `opts.idempotencyKey`.

## Objective

Wire console UX for the remaining webhook-endpoint CRUD operations
(create, edit, disable, re-enable, delete) so an org admin can manage
endpoints end-to-end from `/orgs/:orgSlug/webhooks` without ever
needing the CLI or raw API. Pure consumer of the existing SDK surface
— no contract, SDK, or backend changes.

## PR Boundary

- **In scope:**
  1. Create-endpoint dialog launched from a "New endpoint" button on
     the org-scoped list page (`webhooks/page.tsx`). Fields: `url`
     (required, http(s) URL), `name` (optional), `description`
     (optional). On success, immediately route to
     `/orgs/:orgSlug/webhooks/:endpointId` (or close + reload + toast).
     Carry an idempotency key per submission (mirror
     `invitations/page.tsx` shape).
  2. Edit-endpoint dialog (rename / re-target URL / edit description)
     on the detail page (`[endpointId]/page.tsx`). PATCH via
     `client.webhooks.updateEndpoint(orgId, endpointId, body)`.
  3. Disable / re-enable controls on the detail page. Disable goes
     through `client.webhooks.disableEndpoint(orgId, endpointId, { reason? })`
     with a reason field (optional textarea). Re-enable goes through
     `client.webhooks.updateEndpoint(orgId, endpointId, {})` with whatever
     the contract accepts to flip status back to `active` *if* the
     contract already supports it; otherwise hide the re-enable
     control behind the existing CLI/API hint and document the gap as
     a Spec Proposal in the implementer report (do not invent an SDK
     method or new contract field). Verify the actual re-enable path
     by reading the SDK + contracts before coding.
  4. Delete-endpoint destructive confirm on the detail page, behind a
     typed-confirmation dialog ("Type the endpoint URL to delete").
     On success, route back to `/orgs/:orgSlug/webhooks` and toast.
  5. Replace the list-page "use the API or CLI" empty-state copy with
     a primary "Create endpoint" CTA that opens the same dialog.
  6. Tests: vitest cases for the new dialogs (validation, success
     toast, `precondition_failed` insight rendering, idempotency-key
     propagation on create). Aim for ≥6 new test cases above existing
     floor; mirror `invitations/page.tsx` test cadence if a matching
     test file already exists in the repo, otherwise add a co-located
     `__tests__/` file.

## Constraints

1. Pure consumer of the existing SDK surface
   (`@saas/sdk` `WebhooksClient` methods listed in PR Boundary). Do
   not introduce a new method, do not add a request shape, do not
   bypass the SDK to call `fetch` directly from the console.
2. All mutations go through `wrap()` (from `@/lib/api`) and surface
   `precondition_failed` envelopes via the existing
   `<PreconditionInsight />` component, the way
   `apps/web-console-next/src/app/(app)/orgs/[orgSlug]/invitations/page.tsx`
   does.
3. Forms use `ZodForm` (`@/components/ui/zod-form`) with a co-located
   `z.object({...})` schema. Field validation messages must be
   user-readable (no raw zod error codes).
4. No UI primitive rendered outside `@/components/ui/*`. No new
   primitives without an explicit `@/components/ui/<name>.tsx` PR
   (which is out of scope here).
5. Idempotency-Key on `createEndpoint`: generate a fresh UUID per
   submission attempt and pass via SDK `opts.idempotencyKey`. If
   `crypto.randomUUID()` is unavailable in the bundle target, fall
   back to a documented helper used elsewhere in this app — do not
   inline `Math.random()`.
6. Destructive delete is gated by typed confirmation matching the
   endpoint URL exactly. Reuse the pattern from
   `rotate-secret-dialog.tsx` if it fits; otherwise mirror its visual
   density.
7. URL-driven scope: every page state derivable from the URL. Do not
   stash dialog state in `sessionStorage`. Local React state is fine.
8. No raw event-envelope payloads, signing secrets, secret material,
   secret hashes, or CRON-style internal IDs leaked into the UI. The
   existing detail-page redaction posture (no secret render outside
   the reveal-once dialog) must be preserved.
9. Tests live under `apps/web-console-next/src/**/__tests__/` (mirror
   the existing co-location pattern). At least 6 new vitest cases.
10. No changes to files outside
    `apps/web-console-next/src/app/(app)/orgs/[orgSlug]/webhooks/**`
    and `apps/web-console-next/src/components/webhooks/**` and the
    matching `__tests__/`. If you find you must touch
    `apps/web-console-next/src/components/ui/**` or
    `@/lib/**`, stop and write a Spec Proposal instead.
11. Do not touch any file under `apps/api-edge/`, `apps/webhooks-worker/`,
    `packages/sdk/`, `packages/contracts/`, `packages/db/`,
    `packages/cli/`, `packages/webhook-verifier/`, `infra/terraform/`,
    `tests/`, `tooling/`, `stack-tectonic/`, `kiox.lock`, or
    `pnpm-lock.yaml`.
12. Branch must be created or reused, committed, pushed, and a PR
