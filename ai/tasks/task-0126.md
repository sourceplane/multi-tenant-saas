## Why this task

B5 is partially shipped on main:
- Per-endpoint **delivery history (read)** — Task 0120 (list + get delivery
  attempts, SDK + CLI + Console).
- **Signing-key rotation UX** + dual-signature grace window — migration 130
  (`130_webhook_secret_rotation_grace`), `rotate-secret` route, grace-window
  dual `X-Webhook-Signature-Previous` header in `delivery.ts`.
- **`@saas/webhook-verifier`** zero-dep helper + `webhook sign` / `webhook
  verify` CLI — Task 0105.

The remaining named B5 gap is **manual delivery replay**: today the only way a
failed delivery is re-sent is the background `retryFailedDeliveries()` cron in
`apps/webhooks-worker/src/delivery.ts`, which stops at `MAX_RETRIES` (5) and
then leaves the attempt terminal (`status='failed'`, `nextRetryAt=null`). An
operator who fixed their endpoint has **no way to ask the platform to redeliver
a specific past delivery attempt on demand**. `grep -rniE "replay|redeliver"`
across `webhooks-worker/router.ts`, `contracts/webhooks.ts`, `sdk/webhooks.ts`
returns nothing — confirmed greenfield route.

## What "done" looks like

A buyer-credible "Redeliver" action: from a specific terminal delivery attempt
in the per-endpoint history, an org operator can trigger a fresh delivery of the
**same event to the same endpoint** through the existing signing/delivery seam,
and see the new attempt appear in history.

## Grounding (real code — trust these, verify before editing)

- Delivery model: `apps/webhooks-worker/src/delivery.ts`
  - `deliverAttempt(ctx, attempt, event, endpoint)` is the single delivery
    chokepoint — resolves endpoint, builds the signed payload
    (`{ id, type, occurred_at, data }`), POSTs, records status via
    `updateDeliveryAttempt`. Re-use it; do NOT fork a second delivery path.
  - `dispatchNewEvents()` creates the first attempt via
    `createDeliveryAttempt({ id, orgId, endpointId, subscriptionId, eventId,
    eventType, idempotencyKey })` then calls `deliverAttempt(...)`.
  - `retryFailedDeliveries()` re-delivers retryable attempts and passes
    `event=null` → payload `data:{}`. For a manual replay we want the FULL
    original payload, so the event should be re-fetched by id (see open
    decision below).
- Attempt persistence (`packages/db/src/webhooks/types.ts`): a
  `WebhookDeliveryAttempt` retains `endpointId`, `subscriptionId`, `eventId`,
  `eventType`, `attemptNumber`, `status` — everything needed to reconstruct a
  replay. `DeliveryAttemptStatus = "pending" | "success" | "failed" |
  "retrying"`.
- Worker routes: `apps/webhooks-worker/src/router.ts`
  - Item-level delivery routes already exist:
    `GET .../endpoints/:endpointId/delivery-attempts`,
    `GET .../webhooks/delivery-attempts/:id`.
  - Endpoint action routes follow the
    `POST .../endpoints/:id/{disable,enable,rotate-secret}` shape — mirror it
    for the replay action.
- Edge surface: `apps/api-edge/src/webhooks-facade.ts` already wraps the
  webhooks proxy in `replayOrExecute(... "webhooks" ...)` and allows
  `["GET","POST","PATCH","DELETE"]`. A new `POST` sub-path flows through
  the existing facade automatically — verify, don't rebuild auth.
- Consumer surfaces (the same trio as Task 0120/0122):
  - SDK `packages/sdk/src/webhooks.ts`
  - CLI `packages/cli/src/commands/` (mirror `webhook-deliveries.ts`)
  - Console `apps/web-console-next/src/app/(app)/orgs/[orgSlug]/webhooks/[endpointId]/page.tsx`
    + `components/webhooks/delivery-history.ts` (dependency-free helper,
    SDK-only via `wrap()`, ZERO `fetch(`).

## Scope (IN)

1. **webhooks-worker**: new `POST
   /v1/organizations/:orgId/webhooks/delivery-attempts/:id/replay` (item-route
   shape; org-scoped, deny-by-default actor auth identical to the sibling
   delivery-attempt GETs). Handler:
   - Loads the original attempt (`getDeliveryAttempt(orgId, attemptId)`); 404
     if absent / wrong org.
   - Creates a NEW delivery attempt (fresh id, `attemptNumber` reset to 1,
     `status` pending) for the SAME `(endpointId, subscriptionId, eventId,
     eventType)` with a replay-distinct `idempotencyKey` (e.g.
     `${subscriptionId}:${eventId}:replay:${newAttemptId}` — must NOT collide
     with the dispatch key `${sub.id}:${event.id}:1`).
   - Delivers it through the existing `deliverAttempt(...)` seam.
   - Returns the new `PublicWebhookDeliveryAttempt` (201 or 200 — implementer's
     call, justify).
   - Emits the normal delivery lifecycle event(s) the existing seam already
     emits (`webhook.delivery_succeeded` / `webhook.delivery_failed`); no new
     event type required, but if a `webhook.delivery_replayed` audit marker is
     warranted, the implementer may add ONE additive event type (justify in
     report; keep it best-effort/non-blocking).
2. **packages/db (webhooks)**: only if needed — a `getEventById`-style read on
   the events repo to rehydrate the full original payload for the replay
   (events repo currently exposes `queryEventsByOrg` but no single-id getter).
   If added, it is a pure additive read; NO schema/migration change.
3. **contracts**: additive `ReplayWebhookDeliveryRequest` (likely empty body) +
   `ReplayWebhookDeliveryResponse` (carrying the new `PublicWebhookDeliveryAttempt`).
   Byte-stable to all existing webhook contract types.
4. **SDK**: `replayDelivery(orgId, attemptId)` (or equivalent) on the webhooks
   client; export new types from `index.ts`. Additive — existing methods
   unchanged.
5. **CLI**: `webhook deliveries replay <attempt-id>` (mirror
   `webhook-deliveries.ts` ergonomics: `--output=json`, human confirmation,
   pure SDK consumer).
6. **Console**: a "Redeliver" action on each terminal delivery attempt row in
   the per-endpoint delivery history, SDK-only via `wrap()`, optimistic-safe
   (re-fetch history after success), designed empty/disabled states (e.g. only
   offer replay on `failed`/`success`, never on in-flight `pending`/`retrying`).
7. Tests at every layer: worker (replay creates a new attempt, reuses delivery
   seam, 404 on missing/cross-org, idempotency-key non-collision, endpoint-
   disabled path); db (event re-fetch read if added); SDK; CLI; Console helper.
8. NEW `ai/reports/task-0126-implementer.md`.

## Hard exclusions (OUT)

- NO migration / schema change / new column / manifest / checksum touch
  (replay reuses the existing `webhook_delivery_attempts` table; a new row is a
  normal insert via `createDeliveryAttempt`).
- NO change to `deliverAttempt` signing/retry/backoff semantics, the
  `MAX_RETRIES`/auto-disable thresholds, or the dispatch idempotency-key format
  for the automatic path.
- NO change to `retryFailedDeliveries()` selection or the background cron
  cadence.
- NO secret/ciphertext/decrypted-key material in any response, log, event
  payload, or report (open-risks secret rule).
- NO bypass of endpoint `status='active'` gating — replaying to a disabled
  endpoint must take the existing `endpoint_disabled` terminal path, not force
  delivery.
- NO new auth model / impersonation / admin-worker surface — this is the
  normal org-actor webhooks path on api-edge.
- NO `ai/deferred.md`, `specs/roadmap.md`, or
  `infra/terraform/cloudflare-domain/**` edits; NO cloudflare provider pin
  change; NO lockfile/package.json/component.yaml churn beyond what a pure
  additive code change legitimately requires (justify any).

## Open decision the implementer owns (justify in report)

- **Full payload vs. `data:{}` on replay.** The automatic retry path delivers
  `data:{}` (event=null). A buyer-credible manual replay should resend the
  ORIGINAL payload, which requires rehydrating the event by id. Either (a) add a
  single-id events read and pass the real `StoredEvent` to `deliverAttempt`, or
  (b) document why `data:{}` is acceptable for V1 replay. Prefer (a) unless
  there's a concrete reason not to.
- Response code (200 vs 201) and whether to add a `webhook.delivery_replayed`
  audit event.
- PR count (single combined PR — Task 0120/0121 precedent — vs.
  worker-then-consumers). Implementer's call.
