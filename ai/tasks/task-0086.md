# Objective

Stand up `apps/notifications-worker` per `specs/components/14-notifications.md`
as a deployable, policy-respecting V1 with an **adapter-backed delivery
seam** so future tasks can swap in a real email provider (Resend / Postmark
/ SES) without touching call sites. This unlocks roadmap items B1 (real
auth — magic-link email), B5 (webhook failure-budget alerts), B6 (billing
receipts), B7 (audit-log UX coverage of notification events), and the
invitation flow polish in `membership-worker`.

After this PR a domain Worker can call notifications through a stable
internal service-binding contract to:

- enqueue a transactional notification (the only V1 send shape)
- read or update notification preferences for a user or organization
- read delivery status for a specific notification

No caller wiring (no membership-worker invitation hookup, no identity-worker
magic-link hookup, no billing-worker receipt hookup) lands in this PR.
Those are deliberate follow-ups so the V1 surface can be reviewed in
isolation.
