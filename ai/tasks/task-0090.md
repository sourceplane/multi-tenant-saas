## Architect Brief

- Product bar: **Stripe-quality idempotency** on the V1 notifications
  enqueue path. A retry of the same logical event must collapse to one
  notification row and (eventually) one provider attempt â€” with a
  deterministic `idempotency_hit` outcome that the caller can ignore
  silently.
- The user moment this PR improves: a customer never receives the same
  magic-link email or invitation email twice for one logical action,
  even under Workers-runtime retry, transient 5xx, or duplicate
  `ctx.waitUntil` execution.
- The implementer has full latitude on:
    - the exact `idempotencyKey` string shape per call site, as long as
      the value is (a) present, (b) deterministic for a given logical
      action, (c) globally unique across orgs (because the dedupe is
      `(orgId, idempotencyKey)` it is sufficient to be unique per
      orgId, but keep keys stable);
    - a small helper in `packages/notifications-client` (e.g.
      `buildIdempotencyKey(scope, id)`) IF that reads cleaner than
      inlining at the call site;
    - whether to extend the `EnqueueNotificationOptions` /
      `enqueueNotification` signature to accept `idempotencyKey` as a
      first-class typed field vs. expecting callers to pass it inside
      the existing `request` object;
    - additional internal observability (e.g. logging the
      `idempotent_hit` outcome distinctly from `queued`).
- The implementer must NOT make without a spec proposal:
    - any change to `packages/contracts/src/notifications.ts` enum
      values, fields, or shapes (`idempotencyKey` is already in the
      contract â€” only its population at the caller is changing);
    - any change to the notifications-worker idempotency-hit semantics,
      the conflict-race re-fetch path, or the underlying
      `(org_id, idempotency_key)` uniqueness invariant;
    - introducing new template keys or category values;
    - any wrangler.jsonc / component.yaml / infra change.
- Failure modes that invalidate this PR even if tests pass:
    - any `idempotencyKey` derived from a value that is not stable
      across retries (e.g. `Date.now()`, `crypto.randomUUID()` minted
      at the enqueue site, or any value not already durable in the row
      being acted on);
    - any `idempotencyKey` containing a raw secret (token, magic-link
      `rawCode`, password-equivalent material) â€” keys end up in DB rows
      and logs;
    - `idempotencyKey` collisions across logically-distinct events
      sharing the same upstream id (must be scoped â€” e.g. prefix with
      template key);
    - the enqueue best-effort contract regresses (no thrown errors
      reach the caller; user-facing 2xx response unchanged on enqueue
      failure).
- Recommended pattern (deviate with one-line rationale in report):
  scope-prefixed string keys, e.g.
  `auth.magic_link:${challengeId}`,
  `invitation.created:${inv.id}`,
  `invitation.accepted:${invitationPublicId(inv.id)}:${memberPublicId(member.id)}`.

## Objective

Populate `idempotencyKey` on every notifications enqueue across the
three V1 callers so a retry of the same logical event collapses to one
notification row + one provider attempt via the existing
notifications-worker idempotent-hit path. Closes the V1 risk note
flagged in Task 0088 verifier; required hardening before any real
provider swap (`notifications-provider-swap` deferred candidate) can
ship safely.
