## Architect Brief

- Product bar: **Linear/Stripe-quality repo hygiene.** A fully clean
  `pnpm -r --no-bail lint` exit is the bar — anyone running `pnpm
  lint` should see zero errors and feel the repo is loved.
- User moment: a contributor lands on the repo, runs `pnpm install &&
  pnpm -r lint`, and gets a clean green. Today they get 39 errors
  spread across 9 workspaces and have to mentally filter "real" from
  "background" warnings.
- Implementer free choices (no spec proposal needed):
  - Per identifier, choose the smallest-diff fix: `_`-prefix rename
    (when the identifier is intentionally retained for shape/test
    harness reasons) **or** delete the binding/import outright (when
    nothing in the file references it).
  - Choose where to apply the rename in test files (parameter list,
    destructure, type-only import, value import) based on what the
    file itself communicates.
- Implementer must NOT make these decisions without a spec proposal:
  - Editing `tooling/eslint/index.js` (shared rule baseline).
  - Editing any `eslint.config.js` re-export — they must keep the
    canonical 2-line shape from Task 0092.
  - Down-grading `no-unused-vars` from error to warn anywhere.
  - Touching production source under `apps/*-worker/src/**` to change
    behaviour. The 4 production-source errors below are all in
    top-level imports / response helpers — fix by deleting or
    `_`-prefixing the unused symbol, not by changing call paths,
    handlers, or response shapes.
  - Touching `pnpm-lock.yaml`, `package.json` (any), `intent.yaml`,
    `component.yaml`, `wrangler.*`, `infra/terraform/**`, or
    `kiox.lock`.
- Failure modes that invalidate the PR even if `pnpm -r lint` exits 0:
  - Behavioural change to any prod handler (route shape, response
    body, error envelope, log line).
  - Renaming an exported symbol on a public package boundary (any
    rename inside `packages/*/src/index.ts` or `packages/contracts`
    is out of scope — the errors are all in `apps/*` and `tests/*`).
  - Suppression via inline `// eslint-disable-*` — the goal is "fix
    the underlying error," not silence it. The sole acceptable
    suppression is `_`-prefix renaming, which the shared rule
    explicitly ignores.

## Constraints

1. Shared rule baseline at `tooling/eslint/index.js` is frozen for
   this PR. The authorized fix vocabulary is exactly: delete the
   unused symbol, or rename it to a `_`-prefixed identifier (the
   shared `argsIgnorePattern: "^_"` makes that legal).
2. `// eslint-disable*` comments are NOT allowed as a fix for any of
   the 39 errors.
3. When the unused identifier is an import, prefer outright deletion
   over `_`-rename. When the unused identifier is a destructured
   parameter the test relies on as a positional shape, `_`-rename is
   correct.
4. When deleting a named import leaves the import statement empty,
   delete the entire import line.
5. Production-source files (the 4 errors in `apps/*-worker/src/**`)
   must be touched only at the import statement / response helper
   level — no body-of-handler edits.
6. Frozen pnpm lockfile must be honoured: `pnpm install
   --frozen-lockfile` exits 0 before and after.
