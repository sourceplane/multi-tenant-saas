# Required Outcomes

1. Emit safe delivery lifecycle events.
   - On successful customer webhook delivery, append a canonical event with type `webhook.delivery_succeeded`.
   - On permanent delivery failure after bounded retries are exhausted, append a canonical event with type `webhook.delivery_failed`.
   - Event payloads must be safe metadata only: public/opaque delivery attempt ID when appropriate, endpoint ID, subscription ID, source event ID/type, HTTP status if present, bounded failure reason, attempt number, and timestamps.
   - Do not include webhook signing secrets, ciphertext envelopes, raw response bodies, bearer tokens, cookies, customer request headers, full source event payloads, stack traces, SQL text, or provider credentials.
   - Lifecycle event append failure must not cause an additional customer HTTP call or duplicate delivery attempt. Record safe internal errors only.

2. Prevent delivery lifecycle event recursion.
   - The dispatcher must not enter an unbounded loop where delivering `webhook.delivery_succeeded`, `webhook.delivery_failed`, or `webhook.disabled` emits another event that is itself delivered and emits another lifecycle event forever.
   - Implement an explicit suppression rule, guard, or event-type exclusion for delivery lifecycle events. Document the chosen behavior in code comments and tests.
   - Add tests that prove a subscription to a webhook lifecycle event cannot create recursive delivery attempts across dispatcher cycles.

3. Auto-disable endpoints after repeated permanent failures.
   - Define a clear, small V1 threshold for consecutive terminal delivery failures for the same endpoint, for example `AUTO_DISABLE_FAILURE_THRESHOLD = 5`. Keep it as a named constant; do not hardcode magic numbers across the code.
   - Count only terminal failures, not transient `retrying` attempts.
   - A later successful delivery for that endpoint should break the consecutive failure streak.
   - When the threshold is reached, disable the endpoint through the repository, set a safe disabled reason such as `repeated_delivery_failures`, and emit an auditable `webhook.disabled` event.
   - Auto-disable must be idempotent: concurrent or repeated dispatcher passes must not create duplicate disable transitions or duplicate disable audit rows for an already-disabled endpoint.
   - If the endpoint is already disabled, do not call the customer URL; preserve existing `endpoint_disabled` delivery-attempt behavior.

4. Preserve public delivery-attempt and endpoint safety.
   - Public delivery-attempt responses must continue to include safe status/failure metadata only.
   - Public endpoint/subscription responses must continue to exclude `secret_ciphertext`, plaintext signing secrets, hashes, raw event payloads, and provider credentials.
   - Auto-disable audit/event payloads must include enough safe context for operators but no secrets or raw customer responses.

5. Keep event/source data immutable and delivery retries bounded.
   - Do not mutate `events.event_log` rows for source domain events.
   - Do not change Task 0069 retry limits or scheduled dispatcher architecture unless required for correctness and covered by tests.
   - Do not introduce queue bindings or fake queue resources in this task.

6. Add focused tests.
   - Webhooks-worker delivery tests must cover: success lifecycle event, final failure lifecycle event, no lifecycle recursion, threshold-based auto-disable, already-disabled endpoint behavior, and secret-safe payload/failure metadata.
   - DB tests must cover any new repository method or migration/index behavior if storage logic changes.
   - If new constants or event builders are introduced, test edge cases around retry exhaustion and threshold boundaries.
