# Objective

Add the first safe invitation acceptance flow:

- `POST /v1/organizations/{orgId}/invitations/accept`

The public route must be available through `api-edge`, require a signed-in user,
validate that the invitation token belongs to the path organization and the
authenticated user's email, and atomically create both:

- an active organization member record; and
- the organization-scoped role assignment carried by the invitation.

# Required Outcomes

1. Make `acceptInvitation` create role assignments atomically.
   - Update the repository type and implementation so acceptance creates:
     - the accepted invitation state;
     - the organization member;
     - an active role assignment using the invitation's role.
   - The role assignment must be organization-scoped:
     - `scope_kind = 'organization'`
     - `scope_ref = null`
   - The operation must validate all of the following before or inside the
     atomic write:
     - token hash matches a pending invitation;
     - invitation belongs to the supplied organization ID;
     - invitation `email_lower` matches the authenticated user's email lower;
     - invitation is not revoked;
     - invitation is not already accepted;
     - invitation has not expired at the supplied acceptance time.
   - Use one SQL statement for the acceptance write so the invitation cannot be
     marked accepted without the member and role assignment.
   - Return the accepted invitation, created member, and created role assignment.
   - If member or role-assignment uniqueness conflicts, return a safe repository
     conflict and do not leave the invitation accepted.
   - Keep raw token values out of repository inputs and SQL. The repository
     accepts only a token hash.

2. Add safe token hashing for acceptance.
   - Factor the existing invitation token hashing logic so the Worker can hash a
     submitted raw token without generating a new token.
   - Validate the accept request token as an opaque debug-delivery token string.
     The current generator returns a 64-character hex token; reject malformed,
     missing, non-string, or unreasonably large token values with
     `validation_failed`.
   - Never log, persist, return, place in cursors, or include raw tokens in
     reports.

3. Add membership-worker acceptance route.
   - Add `POST /v1/organizations/{orgId}/invitations/accept`.
   - Require actor headers from api-edge:
     - `x-actor-subject-id`
     - `x-actor-subject-type`
     - `x-actor-email`
   - Parse and validate JSON body:
     - required field: `token`
   - Parse the public `org_` ID from the path and return `not_found` for invalid
     organization IDs without parse details.
   - Hash the token, call the repository acceptance method, and map repository
     errors to safe API envelopes:
     - unknown token, wrong org, wrong email, revoked, accepted, or expired
       should not expose sensitive invitation details;
     - conflicts should return a stable client-safe error such as `conflict` or
       `precondition_failed`;
     - database failures should return `internal_error`.
   - Do not call policy-worker for acceptance. The invited actor is not yet a
     member, so the domain authorization is possession of the valid token plus
     authenticated email match.
   - Return safe public metadata only:
     - public invitation ID (`inv_...`)
     - public member ID (`mem_...`)
     - invited email, role, status, invitedBy, expiresAt, acceptedAt, createdAt
     - membership joinedAt/status/role summary
   - Never return token hash, raw DB UUIDs, raw token, bearer token, stack trace,
     policy internals, or provider details.

4. Extend api-edge routing and auth forwarding.
   - Recognize `POST /v1/organizations/{orgId}/invitations/accept` as an
     organization route before the generic invitation item route.
   - Resolve the bearer session through `IDENTITY_WORKER` before forwarding.
   - Extract `data.user.email` from the identity session response and forward it
     to membership-worker as `x-actor-email`.
   - Continue forwarding actor subject ID/type as today.
   - Continue not forwarding raw bearer tokens to membership-worker.
   - Forward JSON body, request ID, traceparent, idempotency-key, and
     content-type as appropriate.
   - Return safe `unsupported`/405 envelopes for wrong methods.
   - If the identity session response lacks a usable user ID or email, treat it
     as authentication failure.

5. Update contracts.
   - Extend `@saas/contracts/membership` with task-scoped acceptance request and
     response types.
   - Reuse existing public invitation/member types where practical.
   - Keep public IDs and response fields consistent with Task 0021 patterns.

6. Add tests.
   - DB tests must cover:
     - acceptance creates invitation accepted state, member, and role assignment;
     - wrong org does not accept;
     - wrong email does not accept;
     - expired, revoked, and already accepted invitations do not accept;
     - member/role uniqueness conflicts do not leave the invitation accepted;
     - raw token is never part of repository input or persisted data;
     - returned role assignment is organization-scoped.
   - Membership-worker tests must cover:
     - route success response shape;
     - malformed JSON and token validation failures;
     - missing actor headers, including missing actor email;
     - invalid public org ID;
     - token hashing passed to the repository, not raw token;
     - wrong org/email, revoked, expired, accepted, and conflict mappings;
     - database/repository failure safe envelopes;
     - no policy-worker call is required for acceptance.
   - Api-edge tests must cover:
     - route matching for `/v1/organizations/{orgId}/invitations/accept`;
     - POST body forwarding;
     - auth resolution before forwarding;
     - `x-actor-email` forwarding from identity session response;
     - missing email in identity response fails authentication;
     - method restrictions;
     - bearer-token redaction.
