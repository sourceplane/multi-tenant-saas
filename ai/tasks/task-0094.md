## Architect Brief

- Product bar: **Stripe API** for the header surface. Stripe's idempotency-key surface is the canonical reference: max 255 chars, ASCII printable, opaque to the server, mandatory documentation in error messages.
- User moment: an SDK or CLI caller that retries a POST after a transient network failure can send the same `Idempotency-Key` and trust that, once durable replay lands in Task 0095, the second call will not double-create. This PR plants the contract so callers can start populating the header without a behavior change.
- Free to choose: validator implementation shape, exact regex/charset (within Stripe-style constraints), error-code naming (within `packages/contracts/src/errors.ts` conventions), test layout in `tests/api-edge/**`, header spelling (canonical `Idempotency-Key`; current `idempotency-key` lower-case is fine for `Headers.get()` which is case-insensitive).
- Must NOT: introduce KV/DO/database storage; implement replay; touch rate-limit logic; modify any worker behavior; alter `tooling/eslint/index.js`; touch `infra/terraform/cloudflare-domain/**` or the `cloudflare ~> 4.52` pin; introduce `eslint-disable*` comments; touch `pnpm-lock.yaml`, `intent.yaml`, `component.yaml`, `wrangler.*`, or `kiox.lock`.
- Failure modes that invalidate even with green tests: malformed-key requests passing through to a worker (must 400 at the edge); a missing key on a POST being rejected (key MUST stay optional in this PR — required-key enforcement is part of B4 SDK rollout, not B3); the validator being defined inside `apps/api-edge` instead of `packages/contracts` (must be reusable so worker-side replay layer in 0095 can import the same parser).
- Recommended pattern: add a small `idempotency.ts` module to `packages/contracts/src/` exporting `parseIdempotencyKey(value: string | null): { ok: true; key: string } | { ok: false; reason: string }` and an `IDEMPOTENCY_KEY_HEADER` constant. Edge wraps this in a single helper that returns an `errorResponse` on `ok: false`. One reviewable seam.

The implementer has full latitude on validator shape, regex, error-code naming, and test layout inside the Constraints below. Decisions taken under this latitude must be recorded with one-line rationale in the implementer report. Anything outside the listed dimensions requires a spec proposal.

## Objective

Add a reusable idempotency-key contract to `packages/contracts` and wire validation into `apps/api-edge` so that any unsafe HTTP method (POST/PATCH/DELETE/PUT) carrying a malformed `Idempotency-Key` header is rejected with a 400 at the edge before forwarding to a downstream worker. No durable replay; no behavior change for callers that omit the header.

## PR Boundary

- **In:** `packages/contracts/src/idempotency.ts` (new) + barrel export; one new edge helper (e.g. `apps/api-edge/src/idempotency.ts`); call sites in the unsafe-POST facades (`org-facade.ts`, `project-facade.ts`, `auth-facade.ts`, `metering-facade.ts`, `config-facade.ts`, `webhooks-facade.ts`, `billing-facade.ts`) — only the unsafe-method branches; new tests in `tests/api-edge/` (or per-facade test homes) for the validator and the edge integration; one open-risks.md update reflecting partial progress.

## Constraints

1. The validator must live in `packages/contracts` so worker-side code (Task 0095) and a future SDK (B4) can import it. It must NOT live inside `apps/api-edge`.
2. The header surface in error messages must match what callers actually send: `Idempotency-Key` (canonical Stripe spelling), independent of the case-insensitive `Headers.get("idempotency-key")` lookup.
3. The error-code name must follow existing `packages/contracts/src/errors.ts` conventions and stability rules; do not invent a parallel naming style.
4. The helper must be invoked only on unsafe HTTP methods. GETs and HEADs ignore the header (do not 400 on a malformed header for a read; reads have no idempotency semantics).
5. Optional header — never reject a request *for being missing the key*. Only reject when present-and-malformed.
6. No `eslint-disable*` comments. No `// @ts-ignore`. No `_`-prefix renames unless required by an existing test harness shape.
7. Boundaries to preserve from prior tasks: do not modify `tooling/eslint/index.js`, `infra/terraform/cloudflare-domain/**`, the `cloudflare ~> 4.52` pin, or `apps/notifications-worker/**`.
