## Objective

Ship the B5 **webhook delivery history** observability surface end to end:
make the already-shipped delivery-attempts backend visible and useful to a
buyer through (1) SDK pagination plumbing, (2) a designed Console delivery

history") and the U4/U8 design bars (designed empty + skeleton states) for the
Console leg.

## Architect Brief

- Product bar: **Stripe Dashboard "Webhook attempts" / Vercel deployment-log**
  quality for the Console panel; **Stripe CLI `events`-list** ergonomics for
  the CLI command. A technical buyer should be able to answer "did my last 50
  deliveries succeed, and why did that one fail?" without leaving the surface.
- User moment improved: an operator debugging a flaky endpoint can see the
  recent attempt timeline (status, attempt #, HTTP code, safe failure reason,
  next-retry time) per endpoint, and page back through history.
- You are FREE to choose, inside the Constraints: the Console panel layout
  (table vs timeline), column set and ordering, status-badge palette, relative
  vs absolute timestamp rendering, "Load more" vs infinite-scroll pagination
  affordance, the CLI table/json column shape and flag names
  (`--limit`/`--cursor`/`--output`), and whether to also ship the
  single-attempt `getDeliveryAttempt` detail surface (CLI `webhook delivery
  <attemptId>` and/or a Console row-expand) if it stays within the panel.
- You MUST NOT, without a spec proposal: add a **replay/redeliver** action
  (no worker route exists — `apps/webhooks-worker/src/router.ts` has no
  replay; that is future spec-proposal territory), change any
  contract/DB/worker/api-edge behavior, add a status/event-type filter query
  param the worker does not parse (worker `parsePageParams` reads ONLY
  `limit`+`cursor`), or render any secret/raw-body/full-event-payload (the
  contract deliberately exposes only a safe `failureReason` summary).
- Failure modes that invalidate the PR even if tests pass: a raw
  `failureReason`/error string or HTTP code reaching the user with no designed
  treatment; an undesigned `"No deliveries yet"` + emoji placeholder instead of
  a real EmptyState; missing skeleton on the panel's loading state; the SDK
  list method silently dropping the caller's `limit`/`cursor`; any signing
  secret, raw response body, or full event payload rendered anywhere.
- Recommended pattern: thread pagination through the SDK exactly like
  `packages/sdk/src/metering.ts` does (`buildQueryRecord` + `query` on the
  request); in the Console reuse the existing `useAsync` + `wrap()` + `Card`
  + `Badge` + `Skeleton` + `EmptyState` primitives already imported by the
  detail page; in the CLI mirror the `webhook-disable.ts`/`webhook-enable.ts`
  command shape (`resolveOrgId(ctx,false)`, `assertOutputModeValid`,
  human + `--output=json`). Deviate on a one-line documented rationale.
