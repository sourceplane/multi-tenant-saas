# Objective

Wire `membership-worker` to `policy-worker` through a private service binding
and add a reusable membership authorization seam. Use that seam to policy-gate
the existing `GET /v1/organizations/{orgId}` path without expanding the public
membership API yet.

# PR Boundary

This PR must not add invitation endpoints, member administration endpoints,
audit/event persistence, email delivery, API-edge policy facade routes, policy
persistence, or policy-engine behavior changes unless a narrow defect is found
while wiring the caller.

# Required Outcomes

1. Add a private policy service binding to membership-worker.
   - Extend `apps/membership-worker/src/env.ts` with `POLICY_WORKER?: Fetcher`.
   - Add stage/prod `services` bindings in
     `apps/membership-worker/wrangler.jsonc`:
     - stage `POLICY_WORKER` -> `policy-worker-stage`
     - prod `POLICY_WORKER` -> `policy-worker-prod`
   - Preserve existing `workers_dev: false` and `SOURCEPLANE_DB` Hyperdrive
     bindings.
   - Add `dependsOn: [{ component: policy-worker }]` or the repo's accepted
     equivalent to `apps/membership-worker/component.yaml` if Orun validation
     accepts it. If validation rejects the edge, do not force it; write a spec
     proposal and record the blocker/decision in the report.

2. Add a membership-worker authorization helper/client.
   - Keep it inside `apps/membership-worker`; do not import
     `apps/policy-worker` internals or `packages/policy-engine`.
   - Depend only on shared contracts such as `@saas/contracts/policy`.
   - Convert `membership.role_assignments` rows from
     `repo.listRoleAssignments(orgUuid, actor.subjectId)` into V1
     `role_assignment` membership facts:
     - organization scope -> `{ kind: "organization", orgId }`
     - project scope -> `{ kind: "project", orgId, projectId: scopeRef }`
   - Call `POLICY_WORKER.fetch` with
     `POST /v1/internal/policy/authorize`, JSON content type, and `x-request-id`.
   - Send an `AuthorizationRequest` with:
     - `subject` from `ActorContext`
     - requested action, initially `organization.read`
     - resource `{ kind: "organization", id: orgUuid, orgId: orgUuid }`
     - context membership facts from the membership repository
   - Fail closed when the binding is missing, the policy Worker rejects the
     request, returns malformed JSON, or returns `allow: false`.
   - Do not leak policy-worker names, internal URLs, stack traces, role facts,
     or raw database IDs in public error responses.

3. Replace the existing local org-read authorization check.
   - `GET /v1/organizations/{orgId}` must authorize through policy-worker using
     action `organization.read`.
   - A policy denial for org read should keep the existing no-enumeration
     behavior: return `not_found` rather than `forbidden`.
   - Authorization infrastructure failures may return a safe `internal_error`
     or `service unavailable` style envelope, but must not return organization
     data.
   - Keep `POST /v1/organizations` as the pre-organization bootstrap path; do
     not add a policy call there.
   - Keep `GET /v1/organizations` as a membership-owned subject query unless a
     narrow code-reality reason requires otherwise.

4. Add or update tests.
   - Membership-worker tests must prove the policy helper sends the expected
     authorization request and maps role assignments into policy facts.
   - Tests must cover allow, deny, missing policy binding, policy fetch failure,
     malformed policy response, and repository role-list failure.
   - Tests must prove org-read denial still returns a non-enumerating result.
   - Add config tests that stage/prod bind `POLICY_WORKER` to the matching
     stage/prod policy Worker and never cross environments.
   - If `component.yaml` gets a `dependsOn` edge, include Orun validation in the
     report and mention the rendered plan impact.
