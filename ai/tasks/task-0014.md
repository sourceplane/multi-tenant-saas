# Objective

Add the first public `api-edge` service-binding facade for authentication.

After this PR, clients should be able to call the existing identity auth routes
through `apps/api-edge`:

- `POST /v1/auth/login/start`
- `POST /v1/auth/login/complete`
- `GET /v1/auth/session`
- `POST /v1/auth/logout`

The edge Worker must call the already-deployed Identity Worker through a
Cloudflare service binding, not through the public `workers.dev` URL and not by
importing identity-worker source or querying identity tables directly.

# Required Outcomes

## Service Binding Configuration

- Add an `IDENTITY_WORKER` service binding to `apps/api-edge/wrangler.jsonc`
  for `stage` and `prod`:
  - stage must bind to `identity-worker-stage`;
  - prod must bind to `identity-worker-prod`.
- Do not bind stage `api-edge` to prod identity, prod `api-edge` to stage
  identity, or dev/local to a live stage/prod service as a shortcut.
- If local/dev have no service binding, the code must handle that cleanly with
  a safe envelope error for auth routes.
- Preserve existing `SOURCEPLANE_DB` Hyperdrive bindings and `/health`
  database check behavior unless you can prove the facade makes a narrower
  health model necessary.
- Extend `apps/api-edge/scripts/verify-bindings.mjs` or add an equivalent
  config check so stage/prod service binding names and targets are verified
  alongside Hyperdrive IDs.
- Do not add new Cloudflare accounts, routes, custom domains, Hyperdrive
  resources, Supabase resources, AWS resources, Secrets Manager secrets, KV
  namespaces, queues, Durable Objects, or Terraform resources.

## Auth Facade Routing

- Route only the existing `/v1/auth/*` paths listed in the objective to
  `env.IDENTITY_WORKER.fetch(...)`.
- Preserve the incoming method, path, query string, request body, `Authorization`,
  `Content-Type`, `X-Request-Id`, `traceparent`, and `Idempotency-Key` headers
  when forwarding unless there is a clear security reason to filter a header.
- Do not read or log raw bearer tokens, raw login codes, token hashes, code
  hashes, connection strings, Supabase keys, Cloudflare tokens, or credentials.
- Return the Identity Worker response envelope and status code unchanged for
  successful downstream responses and expected downstream API errors.
- If `IDENTITY_WORKER` is not configured or the service binding throws, return
  a safe public error envelope using the semantic error set from
  `specs/contracts/api-guidelines.md`. Do not leak exception messages,
  stack traces, Worker names, hostnames, connection details, or provider
  internals.
- Keep `/health` available at `api-edge`. It may add a non-secret identity
  service check, but it must not leak downstream response bodies, tokens,
  codes, connection strings, or raw errors.
- Keep non-auth unknown routes returning the standard public error envelope.

## Boundary And Extraction Rules

- `api-edge` must remain a transport facade. Do not import identity Worker route
  handlers, auth services, repository adapters, or identity database code into
  `api-edge`.
- Do not make `api-edge` query identity tables or call `@saas/db/identity`.
- Do not duplicate passwordless login logic in `api-edge`; identity-worker
  remains the owner of auth behavior and persistence.
- The facade may mirror the existing HTTP identity contract because Task 0013
  shipped HTTP route handlers. Do not invent a parallel RPC contract in this
  PR unless the current Worker runtime makes HTTP service-binding forwarding
  unsafe.
- Keep the public API stable so a later migration from HTTP-style service
  binding to RPC-style service binding does not affect clients.

## Tests

- Add focused `api-edge` tests that do not require live Cloudflare, Supabase, or
  Hyperdrive access.
- Use a fake service binding/fetcher and fake database adapter seam where
  needed.
- Cover at minimum:
  - each supported auth route forwards to `IDENTITY_WORKER`;
  - method, path, query string, JSON body, bearer token, request ID,
    `traceparent`, and `Idempotency-Key` preservation;
  - downstream success envelopes are passed through with status and headers
    appropriate for JSON API responses;
  - downstream semantic error envelopes are passed through;
  - missing service binding returns a safe envelope error;
  - service-binding exceptions return a safe envelope error without raw cause;
  - unsupported auth methods and unknown routes use standard error envelopes;
  - `/health` remains safe and does not leak downstream bodies or secrets;
  - config verification catches missing or cross-environment service targets.
- If you add `tests/api-edge`, add a first-class Orun component for it. Do not
  add an unsound cross-environment `dependsOn` edge from `api-edge` to a
  dev-only test component unless you prove the rendered DAG is correct.

# Constraints

- Follow `specs/components/01-edge-api.md`: `api-edge` owns public transport,
  request context, trace/header propagation, and bounded-context forwarding; it
  does not own identity persistence or business rules.
- Follow `specs/components/02-identity.md`: identity owns sign-in, session
  issuance, session validation, and login proof facts.
- Follow `specs/contracts/api-guidelines.md`: `/v1` routes, standard success
  and error envelopes, semantic error codes, bearer auth transport, request
  IDs, trace propagation, and service-binding boundaries.
- Follow `specs/repo.md`: internal Workers should use service bindings; HTTP
  fetch between Workers is allowed only when intentionally mirroring a public
  contract.
- Keep `@saas/api-edge` Worker-safe. Do not import Node-only modules into
  Worker runtime source.
- Use parameterized/typed interfaces where applicable and avoid broad `any` or
  double-casts that hide binding-type drift.
- Preserve secret hygiene in source, tests, logs, and reports.
