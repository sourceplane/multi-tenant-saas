# Objective

Add the first production-safe Supabase Postgres migration runner and
Orun-controlled apply path for the existing `packages/db` manifest.

The task should make PR CI prove the runner and plan behavior without mutating
live databases, and make post-merge `github-push-main` jobs able to apply only
missing migrations to the live `stage` and `prod` Supabase databases.

# PR Boundary

One PR in `multi-tenant-saas` that establishes the migration apply path.

Allowed scope:

- Node-only migration runner, CLI, and helper code for `packages/db`, behind
  explicit non-default entry points;
- runner tests, fakes, fixtures, or verifier tests under `tests/db` or a
  narrowly named new test package;
- a narrowly scoped Orun component/composition/manifest change if needed to run
  migration verification on PRs and migration apply on main pushes;
- package metadata and lockfile updates required by the runner;
- README or component docs that explain migration apply behavior, secret
  sources, and non-secret verification;
- concise task/report/context updates under `ai/**`.

Non-task generated outputs such as `dist`, `node_modules`, `.orun`, Terraform
working directories, `plan.json`, and TypeScript build info files must stay
uncommitted.

# Required Outcomes

- Create or switch to a task branch before committing, using the repo branch
  convention such as `codex/task-0008-db-migration-apply`.
- Preserve `@saas/db` default public exports as Worker-safe. Do not import
  Node-only modules (`node:fs`, `node:crypto`, `pg`, AWS SDK, etc.) from the
  package's default entry point.
- Add a Node-only runner path that:
  - reads the canonical `packages/db` manifest;
  - loads migration SQL files deterministically;
  - verifies checksums before apply;
  - detects already-applied migrations;
  - fails on checksum mismatch for an already-applied migration;
  - applies missing migrations in manifest order;
  - records successful applications in `_migrations.applied`;
  - is idempotent when re-run with no pending migrations;
  - uses a transaction per migration, or documents and tests why a different
    transaction boundary is safer;
  - uses an advisory lock or equivalent single-run guard if feasible with the
    chosen Postgres client.
- Add a CLI or script entry point that can run in at least:
  - local/test mode against a fake or injected adapter;
  - `plan`/`dry-run` mode that reports pending migration IDs without mutating a
    live database;
  - `apply` mode for CI.
- Add AWS Secrets Manager loading for CI apply mode using the existing secret
  JSON shape from `infra/terraform/supabase/README.md`. Secret values,
  connection URIs, passwords, and tokens must never be printed, stored in
  reports, or copied into source files.
- Do not create a `dev` Supabase project or require a live `dev` database.
- Ensure pull-request jobs do not mutate live databases. Live apply must be
  restricted to the main-branch/post-merge Orun path for `stage` and `prod`.
- Wire the Orun/component path so the rendered plan makes the migration verify
  and apply behavior explicit. Prefer a dedicated migration component or
  composition if the existing package/Terraform compositions cannot express the
  behavior cleanly.
- If a safe Orun apply profile cannot be expressed without changing shared
  composition semantics, keep the implementation conservative and write a
  proposal instead of hiding live mutation inside an unrelated job.
- Open a GitHub PR for the task branch.
- Update `ai/reports/task-0008-implementer.md` with final checks and the real
  PR number/URL. If PR creation fails, mark the report `BLOCKED` with the exact
  command/error instead of calling the task done.
