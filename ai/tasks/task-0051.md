## Objective

Implement the V1 tenant-scoped public API-key administration runtime across `api-edge`, `identity-worker`, and the minimal policy surface needed to authorize it.

This PR should make the spec-approved public routes real:
- `POST /v1/organizations/{orgId}/api-keys`
- `GET /v1/organizations/{orgId}/api-keys`
- `DELETE /v1/organizations/{orgId}/api-keys/{apiKeyId}`

The implementation must preserve bounded-context ownership: identity owns credentials and identity security events; membership owns service-principal role bindings; policy owns authorization decisions. The runtime route is an orchestration seam across those contexts, not a redefinition of ownership.

## Required Outcomes

- [ ] Add the minimal policy action surface required for public API-key administration.
  - Introduce exactly the three public-admin actions needed by this route family:
    - `organization.api_key.create`
    - `organization.api_key.list`
    - `organization.api_key.revoke`
  - Owner/admin callers must be allowed for org-scoped API-key administration.
  - Project-admin callers must be allowed only when the request is explicitly narrowed to a project-scoped key within their project.
  - Authorization must remain deny-by-default for all other callers.

- [ ] Add the tenant-scoped public route family through `api-edge`.
  - `apps/api-edge/src/org-facade.ts` must recognize:
    - `POST /v1/organizations/{orgId}/api-keys`
    - `GET /v1/organizations/{orgId}/api-keys`
    - `DELETE /v1/organizations/{orgId}/api-keys/{apiKeyId}`
  - Continue using `resolveActor()` and explicit actor headers.
  - Continue forwarding `idempotency-key`, `traceparent`, and `x-request-id`.
  - Do not forward raw bearer tokens downstream.

- [ ] Add public API-key admin handlers in `identity-worker`.
  - Expose tenant-scoped handlers in `apps/identity-worker/src/router.ts` for create/list/revoke.
  - Add the membership/policy bindings and client helpers needed for authorization/orchestration.
  - Reuse the existing repository layer rather than writing ad hoc SQL in handlers.

- [ ] Implement safe create behavior.
  - Validate `label`, `role`, optional `projectId`, and optional `expiresAt`.
  - Generate a new dedicated backing service principal per API key in this slice. Do not introduce service-principal reuse heuristics.
  - Create the membership binding through the existing internal seam using the canonical `sp_<hex32>` subject ID.
  - Persist only safe derivatives of the API key (hash + prefix). Return the raw secret exactly once in the create response.
  - Record the identity-owned `api_key.created` security event.
  - Emit the org-scoped audit/event copy using the shared DB-backed event repository pattern already used in other workers.
  - If identity-side persistence fails after the membership binding was created, perform best-effort compensation by revoking the new binding and fail closed.

- [ ] Implement safe list behavior.
  - Use the tenant-scoped public route, not a second route family.
  - Support cursor pagination using the existing identity repository pagination surface.
  - Optional `projectId` may be used only as a narrowing filter, not as a replacement path shape.
  - List must never return raw key material.
  - Each response item must include at minimum:
    - `id`, `label`, `prefix`, `createdAt`, `expiresAt`, `lastUsedAt`, `revokedAt`
    - `servicePrincipal`: `{ id, displayName, role, projectId }`
  - Fail closed on malformed or ambiguous scope/binding state; do not guess role/project metadata.

- [ ] Implement safe revoke behavior.
  - Revoke the API key through the identity repository so the credential stops authenticating immediately.
  - Authorize the revoke decision using the stored org/project scope of the backing service principal binding.
  - Record the identity-owned `api_key.revoked` security event.
  - Emit the org-scoped audit/event copy using the shared event repository pattern.
  - Do not widen this slice into service-principal deletion or generic lifecycle management.

- [ ] Preserve explicit scope rules.
  - Org context must always come from the route path.
  - Project scope must remain an explicit narrowing attribute (`projectId`), never an implicit guess.
  - Do not allow project-only addressing or cross-org leakage.

- [ ] Add focused test coverage.
  - Policy-engine / policy-worker tests must prove allow/deny behavior for the new API-key actions.
  - Identity-worker tests must cover create/list/revoke success paths plus fail-closed auth, validation, and secret-handling behavior.
  - api-edge tests must prove org-facade route recognition, forwarding behavior, actor-header propagation, and idempotency-header forwarding for the new routes.
  - Existing session and API-key bearer-resolution behavior from Task 0048 must remain green.
