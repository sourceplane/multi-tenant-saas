# Required Outcomes

## Membership Schema Migration

- Add a membership-owned migration after `010_identity_core`, for example
  `020_membership_core`.
- The migration must declare context `membership` in `packages/db/src/manifest.ts`.
- The migration must be idempotent enough for the current Management API
  runner: use `CREATE SCHEMA IF NOT EXISTS`, `CREATE TABLE IF NOT EXISTS`,
  `CREATE INDEX IF NOT EXISTS`, and migration SQL that is safe to retry after a
  partial autocommit failure.
- Create a dedicated `membership` schema.
- Suggested minimum tables:
  - `membership.organizations`
  - `membership.organization_members`
  - `membership.organization_invitations`
  - `membership.role_assignments`
- Keep foreign keys inside the membership context only. Do not add foreign keys
  to `identity.*`, `projects.*`, `billing.*`, or future schemas.
- Store identity/user references as opaque subject IDs and subject types owned
  by the calling/service contract, not as cross-context foreign keys.
- Every membership-owned tenant table must carry `org_id` directly when it is
  organization-scoped.
- Role assignments must be representable as facts consumable by the policy
  contract from `specs/contracts/tenancy-and-rbac.md`.
- Support organization roles from the spec:
  `owner`, `admin`, `builder`, `viewer`, `billing_admin`.
- Leave room for project roles without coupling to a projects schema:
  `project_admin`, `project_builder`, `project_viewer`.
- Invitations must store only secret hashes for acceptance tokens. Do not store
  raw invitation tokens, raw bearer tokens, login codes, API keys, signing
  secrets, encryption secrets, connection strings, or provider credentials.
- Use normalized slug/email columns and indexes for lookup. Do not rely on
  case-sensitive raw slug or email uniqueness.
- Add useful indexes and table/column comments where they clarify ownership,
  scope, or secret-handling boundaries.
- Update manifest and checksum tests so migration ordering and drift detection
  still work.

## Membership Repository Adapter

- Add a Worker-safe membership repository export behind an explicit subpath,
  preferably `@saas/db/membership`.
- Repository methods should be persistence-oriented. They should not be public
  HTTP handlers, domain-service orchestration, email delivery, policy
  evaluation, token generation, slug generation, or auth/session resolution.
- Repository methods must accept caller-generated IDs, timestamps, normalized
  inputs, and hashes where appropriate.
- Include the minimum methods needed by a future membership Worker task:
  - create/get organization by ID;
  - get organization by normalized slug;
  - atomically bootstrap an organization with creator member and owner role
    assignment, using caller-provided IDs/timestamps;
  - list organizations for a subject;
  - create/get/list/update/remove organization members;
  - create/list/get/revoke invitations, including lookup by invitation token
    hash;
  - accept an invitation in a way that can create a member and mark the invite
    accepted without exposing raw token material;
  - create/list/revoke role assignments for a subject and organization.
- Use parameterized SQL for all user-controlled values. No string interpolation
  of slugs, emails, subject IDs, role names, token hashes, or timestamps.
- Map expected persistence outcomes to stable typed results or safe repository
  errors. Distinguish at least not found, conflict, expired, revoked/removed,
  already accepted, and internal failure where the repository can do so.
- Do not leak raw SQL errors, hostnames, usernames, connection strings,
  invitation token hashes, raw tokens, emails in internal exception messages,
  or credentials through returned errors.
- Keep repository code extractable: no direct imports from Worker route
  handlers, `api-edge`, `identity-worker`, future membership Worker code,
  policy Worker code, or unrelated bounded contexts.
- Keep default `@saas/db` exports Worker-safe and free of runner-only imports.
  Node-only migration runner code must remain behind `@saas/db/runner`.

# Constraints

- Follow `specs/repo.md`: repository adapters own SQL, pooling assumptions,
  transaction boundaries, and Hyperdrive-specific behavior. Domain services
  receive typed repositories or unit-of-work abstractions, not platform DB
  clients.
- Follow `specs/components/04-organizations-membership.md`: membership owns
  organizations, memberships, invitations, role assignments, and
  organization-level settings.
- Follow `specs/contracts/tenancy-and-rbac.md`: organization is the tenant
  boundary, roles are stable policy facts, and policy consumes membership facts
  without membership making authorization decisions.
- Follow `specs/domain-model.md`: every meaningful entity belongs to an
  organization, and project-scoped data later must carry `orgId + projectId`.
- Keep bounded contexts extractable: membership may store opaque identity and
  future project references, but it must not depend on identity or projects
  table ownership.
